# 智源 ANP · 8 人团队工程化完善方案（设计稿，待评审）

> 文档定位：评估当前 ANP 平台在「8 人开发团队并行使用」场景下的工程化缺口，给出分梯队、可落地的完善路线。
> 配套：[`详细设计/开发标准与规范.md`](详细设计/开发标准与规范.md)、[`开发展开计划.md`](开发展开计划.md)。
> 状态：**待评审**。评审通过后再开发。

---

## 0. 一句话结论

> **平台为别人定义了完整研发规范，自己却几乎一个都没落地。**

ANP 在《开发标准与规范.md》里写明了 CI 门禁 6 项、覆盖率 ≥80%、OpenAPI 契约先行、迁移工具化、golangci-lint、trunk-based + Conventional Commits、AI 预审……但这些标准**对平台自身代码几乎全部未生效**。当前是一个人快速迭代出来的原型（v0.2.0，约 1.5 万行），工程化基础设施近乎为零。

**对 8 人团队而言，最大风险不是"功能不够"，而是"工程化撑不住 8 人并行"**。不先把平台自己定义的标准落地到自己身上（dogfooding），8 人一上就会陷入"无 CI + 无测试 + 无契约 + 迁移冲突"的混乱，反而比单人更慢、更易出故障。

**建议节奏**：先用 **2–3 周**补齐 P0 工程化地基，再开 8 人并行功能开发。

## 3. 完善方案（三梯队）

### 梯队 P0 —— 不做，8 人无法并行（第 1–2 周，冲刺工程化地基）

> 目标：让仓库具备"任何分支 push 都能自动验证质量"的能力。这是平台《开发标准与规范.md》第 11 节"质量门禁"的自我落地。

#### P0-1 CI/CD 流水线（堵 R1）

- 选型：GitHub Actions（仓库已在 git）或 GitLab CI（开发展开计划推荐，含评审一体）。
- 每个 PR 必跑 6 项（对齐规范 §11）：
  1. lint / format / type-check（Go: golangci-lint + goimports；前端: eslint+prettier+tsc；Python: ruff+black+pyright）
  2. 单元测试 + 覆盖率门禁（起步门槛设 **30%**，逐周爬坡到 80%）
  3. 安全扫描（gitleaks 扫密钥；govulncheck / npm audit 依赖）
  4. 规则合规校验（调平台自己的 rule 引擎——**dogfooding 起点**）
  5. API 契约一致性检查（P1 引入 OpenAPI 后接）
  6. 构建镜像 + 冒烟（docker build 三服务 + healthz 探活）
- 全绿方可合入；失败硬阻断。
- 落地：`.github/workflows/ci.yml`（或 `.gitlab-ci.yml`），按矩阵跑 Go/Node/Python 三 job。

#### P0-2 代码规范工具链落地（堵 R3）

- Go：加 `.golangci.yml`（启用 govet / staticcheck / errcheck / gosec / revive / goimports），CI 跑 `golangci-lint run`。
- 前端：引入 `prettier`，与现有 eslint 联动；`tsconfig` 确认 `strict: true`；加 `lint-staged` + `husky` pre-commit 本地兜底。
- Python：加 `ruff` + `black` + `pyright` 配置 + `requirements.txt` 固定版本（当前无锁文件）。
- 统一：根目录加 `Makefile` 或扩展 `package.json` scripts，提供 `make lint / fmt / test / check`（Go 目前未纳入根脚本，见下 P0-6）。

#### P0-3 数据库迁移工具化（堵 R4）

- 现状：`internal/db/migrate.go` 659 行，schema + seed 全在一个 `const`，无版本、无顺序。
- 方案：引入 **golang-migrate**（或 goose），把现有 schema 拆成 `infra/migrations/000001_init.up.sql` 等版本文件；seed 独立成可重入的 `seed.go`。
- 收益：多人改表各加独立编号文件，CI 检查无编号冲突；可 up/down 回滚。
- 注：迁移工具化与 P1-3 切 PostgreSQL 可合并进行（PG 是迁移工具的最佳载体）。

#### P0-4 测试基线（堵 R2）

- 不追求一步到 80%。先建"**最小门禁 + 核心模块必测**"：
  - 后端：优先补 `change / release / rule / auth(guard) / qa / codews` 的单测（当前零测试的核心写路径）。
  - 前端：把 `lib/devstep.ts` 已有的测试模式推广到 `lib/api.ts` 关键函数。
  - Python：给 `gateway.py`（GLM 网关）加 mock 测试。
- CI 覆盖率门禁从 30% 起步，每两周上调，写入 P0-1 流水线。

#### P0-5 分支与提交规范 enforce（堵 R1/R3 配套）

- trunk-based：主干 `main` 可发布，短分支 + PR 合入（规范 §3.1 已定，缺工具）。
- 加 `commitlint` + Conventional Commits（规范 §3.2 已定，当前 commit 风格已基本遵循，补校验）。
- 加 PR 模板（`.github/pull_request_template.md`）：含评审 checklist（规范 §4.2）、关联需求号、变更类型。
- 加 `CONTRIBUTING.md`：本地起服务、提交流程、分支命名。

#### P0-6 统一工程命令（基建体验）

- 现状根 `package.json` 只有 `dev/build/lint/test/frontend`，**Go 完全不在统一脚本里**，前后端割裂。
- 加 `Makefile`（或 `Taskfile.yml`）：`make dev / test / lint / fmt / build / migrate / seed`，三语言统一入口。
- `scripts/dev.sh` 升级为依赖该统一命令。

### 梯队 P1 —— 不做，协作低效 / 有风险（第 3–6 周）

#### P1-1 API 契约 OpenAPI 化（堵 R5）

- 后端用注解或 schema 生成 OpenAPI（Go: `swaggo/swag` 或 `ogen`）；导出 `docs/openapi.yaml`。
- 前端用 `openapi-typescript` 或 `openapi-fetch` 生成类型化客户端，**替换手写 `lib/api.ts`**。
- 收益：前后端字段契约由工具保证，联调零口头；CI 加"契约一致性"检查（P0-1 第 5 项）。

#### P1-2 真实鉴权（堵 R6）

- 撤掉 M1 的 `X-User` + localStorage 模拟回退（`lib/api.ts` 当前并存）。
- 接企业 SSO：LDAP / 钉钉 / 企业微信 OIDC；后端签发 JWT，前端 httpOnly cookie 或安全存储 token。
- RBAC 已有中间件（`auth.AuthUser` + 集中式鉴权），需对接真实身份源。

#### P1-3 切 PostgreSQL（堵 R7）

- 规范自述"切 PG 后改 golang-migrate"。SQLite 写锁不适合 8 人并发 + 生产。
- 用 `pgvector` 顺带承接向量检索（开发展开计划推荐）。
- 与 P0-3 合并落地：迁移工具直接按 PG 写。

#### P1-4 密钥与配置安全（堵 R8）

- `.env` 明文 → **SOPS + age** 或 git-crypt；生产密钥不入库。
- CI 接 gitleaks（P0-1 第 3 项）扫历史与增量。
- `system_config` 表已有"配置入库"机制，扩展为密钥引用而非明文存储。

#### P1-5 可观测性闭环（堵 R11）

- `trace_id` 已埋（`server.Trace()`）但未导出。接 **OpenTelemetry** → OTLP 导出（Jaeger / Tempo）。
- 指标：Prometheus（/metrics）覆盖请求数/延迟/错误率/模型 token 消耗。
- 日志：结构化 zap 日志聚合（Loki/ELK）。
- ops 模块从"看被监控对象"扩展到"平台自身也被监控"。

#### P1-6 前端共享层（堵 R10）

- 抽 `lib/hooks/useApi.ts`（统一 fetch + 鉴权拦截 + 错误处理），替代 19 个 page 各自 fetch。
- 引入 **React Query / SWR** 做数据层（缓存、重试、loading 态）。
- 统一 shadcn/ui 组件（规范 §2.4 已约定，未统一）；加全局 ErrorBoundary。
- 错误模型统一：对接后端 `{code, message, trace_id}`（规范 §6）。

#### P1-7 main.go 装配解耦（堵 R9）

- 现状：`main.go` 手动 new 22 个 handler + `Register(v1)`，是合并冲突集中点。
- 方案：每个模块暴露 `Register(deps, v1 *RouterGroup)` 自包含装配；`main.go` 只组装依赖（db/config/logger）并调用各模块 Register。可引入轻量 DI（`wire`）或约定式注册。

#### P1-8 文档落地（堵规范 §7）

- 建 `docs/adr/` 目录，关键决策补 ADR（背景/决策/理由/后果）。
- 建 `CHANGELOG.md`（规范 §7 要求，当前靠 git log）。
- 每个后端模块补 `README.md`（职责/接口/依赖/负责人）——8 人分工后明确边界。

### 梯队 P2 —— 规模化与体验（第 7 周+）

- **P2-1 本地一键起全栈**：`docker-compose.dev.yml`（含 PG + 三服务 + 热重载 + seed），新人 10 分钟 onboard。
- **P2-2 E2E 回归**：Playwright 覆盖主线闭环（需求→编码→审批→发布），堵 R12。
- **P2-3 性能压测基线**：编码任务并发、模型网关吞吐。
- **P2-4 文档站**：mkdocs / vitepress 把 `docs/` 组织成可浏览站点。
- **P2-5 dogfooding 闭环**：平台自己的 PR 跑平台自己的规则引擎 + 审批流 + AI 预审（呼应《AI 协作规范》§10）。
- **P2-6 告警接入实际通知**：钉钉/飞书 webhook（ops 模块已有预案）。

## 6. 风险与取舍

- **"先补工程化会拖功能进度"**：但 8 人无 CI 上功能，返工+回归成本更高；2 周地基换后续持续提速，ROI 明确。
- **覆盖率门禁起步设 30% 而非 80%**：避免"为达标写无意义测试"；逐周爬坡，核心模块优先。
- **PG 切换有数据迁移成本**：当前 SQLite 数据量小（demo+试点），迁移窗口可控；越早切成本越低。
- **OpenAPI 引入有改造成本**：可分模块渐进（先新模块契约化，老模块逐步补注解）。

## 7. 与现有规范的关系

本方案**不新增标准**，只把《开发标准与规范.md》已有的标准**落到平台自身代码**。每一项 P0/P1 都能在规范里找到对应条款：

| 本方案           | 规范条款                    |
| ---------------- | --------------------------- |
| P0-1 CI 门禁     | §11 质量门禁                |
| P0-2 lint/format | §2 编码规范                 |
| P0-3 迁移工具化  | §2.4 SQL/数据               |
| P0-5 分支提交    | §3 分支与提交               |
| P1-1 OpenAPI     | §6 API 契约                 |
| P1-2 鉴权        | §8 安全 + §10 AI 协作守权限 |
| P1-5 可观测      | §9 发布 + ops 板块          |
| P1-8 文档        | §7 文档规范                 |

> **核心叙事**：ANP 的 dogfooding（《开发展开计划》§0.3「第一天就 dogfooding」）还没真正开始——平台先用自己定义的标准把自己武装起来，才有资格要求平台上跑的项目遵守。
