# 智源 ANP · 工程化落地执行计划（P0）

> 基于 [`2026-07-18-8人团队工程化完善方案.md`](2026-07-18-8人团队工程化完善方案.md) 的 P0 梯队，拆为可执行、可验证的任务卡。
> 目标：**2 周内让仓库具备"任何分支 push 都能自动验证质量"的能力**（平台自我 dogfooding 第一步）。
> 状态：执行中。每个任务卡含【改动】【验证】【验收】。

---

## 0. 落地顺序与依赖

```
P0-6 Makefile（基础入口，其余皆依赖）
   ├─> P0-2 Go lint / 前端 format / Python lint（可并行，各自验证）
   ├─> P0-1 CI（引用 make 命令，需 lint 先就绪）
   ├─> P0-5 提交规范（独立）
   ├─> P0-3 迁移工具化（高风险，独立分支评估）
   └─> P0-4 测试基线（持续，门禁随 CI 上线后逐步爬坡）
```

**执行原则**：

- 每个 P0 子项独立 commit（Conventional Commits：`chore(ci): ...` / `test: ...`），可单独 revert。
- 每步**本地验证通过**再进下一步（`make lint` / `make test` 全绿）。
- P0-3 迁移工具化风险最高（动运行中库），单列分支、回滚预案，最后处理。

## 1. 任务卡

### P0-6 统一 Makefile（基础入口）

- **改动**：根目录新增 `Makefile`，封装三语言命令：
  - `make dev` / `make build` / `make test` / `make lint` / `make fmt` / `make check`(lint+typecheck+vet)
  - `make be-test` / `make fe-test` / `make py-test`（分语言）
  - `make migrate` / `make seed`（后端）
- **验证**：`make lint` 在空仓库状态下可跑（即使报问题，命令本身可用）。
- **验收**：`make` 列出所有目标；`make lint` 不报 "command not found"。

### P0-2 Go lint 工具链

- **改动**：`platform/backend/.golangci.yml`，启用 govet / staticcheck / errcheck / gosec / revive / goimports；调整现有代码至全绿（或按严重度分级豁免并留 TODO）。
- **验证**：`cd platform/backend && golangci-lint run`（无 golangci-lint 则 `go vet ./...` 兜底）。
- **验收**：lint 通过或仅剩已记录的低优项。

### P0-2 前端 prettier + eslint

- **改动**：`platform/frontend` 加 `prettier` + `.prettierrc`；与现有 `eslint.config.mjs` 联动；`lint-staged` + `husky` pre-commit；`tsconfig` 确认 `strict`。
- **验证**：`pnpm --filter frontend lint`、`pnpm exec prettier --check .`。
- **验收**：lint/format 通过；`pnpm build`（tsc）通过。

### P0-2 Python ruff + black + pyright

- **改动**：`platform/agent-runtime` 加 `ruff` / `black` / `pyright` 配置（`pyproject.toml`）；`requirements.txt` 补 dev 依赖并锁定版本。
- **验证**：`ruff check .`、`black --check .`、`pyright`。
- **验收**：三工具通过。

### P0-1 CI 流水线（GitHub Actions）

- **改动**：`.github/workflows/ci.yml`，矩阵跑 Go / Node / Python：
  - Go job：`make be-lint && make be-test`
  - Node job：`make fe-lint && make fe-build`
  - Python job：`make py-lint`
  - 安全：gitleaks（扫密钥）+ govulncheck + npm audit
  - 构建：docker build 三服务 + healthz 探活
  - 覆盖率门禁起步 30%（go test -coverprofile）
- **验证**：本地用 `act` 或推到特性分支触发。
- **验收**：PR 上 CI 全绿方可合入（分支保护规则后续配置）。

### P0-5 分支/提交规范 enforce

- **改动**：
  - `commitlint` + `@commitlint/config-conventional`（husky `commit-msg` 钩子）。
  - `.github/pull_request_template.md`（含评审 checklist、关联需求号）。
  - `CONTRIBUTING.md`（本地起服务、提交流程、分支命名）。
- **验证**：构造非规范 commit 信息 → 被拒。
- **验收**：合规 commit 通过；CONTRIBUTING 可指引新人 onboard。

### P0-3 数据库迁移工具化（⚠️ 高风险）

- **改动**：引入 `golang-migrate`，将 `internal/db/migrate.go` 的 659 行 `sqliteSchema` 常量拆为 `infra/migrations/000001_init.up.sql` / `.down.sql`；seed 逻辑独立为可重入函数。
- **风险**：动当前运行中的 SQLite；schema 拆分遗漏会导致启动失败。
- **缓解**：①独立分支；②保留 `Migrate()` 签名不变（内部改实现），上层无感；③拆分后用全新空库 `make migrate && make seed` 验证 schema 一致（与原 `sqliteSchema` 对齐）；④对现有库做 `migrate force` 兜底。
- **验证**：空库迁移后表结构与原一致；现有 dev 库可平滑接入。
- **验收**：迁移可 up/down；多人改表各加独立编号文件。
- **决策点**：本轮先评估 + 出方案，确认无破坏后再执行（可能拆到下一 sprint）。

### P0-4 测试基线补齐（持续）

- **改动**：补核心写路径单测——`change` / `release` / `rule` / `qa` / `codews`（当前零测试）。
- **验证**：`make be-test` 通过；覆盖率报表生成。
- **验收**：CI 覆盖率门禁 30% 通过；核心模块关键路径有异常/边界用例。

## 2. 风险与回滚

| 风险                            | 缓解                                                                                          |
| ------------------------------- | --------------------------------------------------------------------------------------------- |
| lint 引入大量历史告警，阻塞开发 | 严重度分级：error 级必修，warn/info 级先 `//nolint` + TODO，分批清                            |
| CI 太严反而拖慢                 | 门禁分阶段：先 lint+build，再逐步加 test/覆盖率/安全                                          |
| 迁移工具化破坏现有库            | 独立分支 + 空库验证 + 保留旧 `Migrate` 签名 + force 兜底                                      |
| 工具版本不一致                  | 所有工具配置入仓；Go 用 go.mod、Node 用 package.json、Python 用 requirements/pyproject 锁版本 |
