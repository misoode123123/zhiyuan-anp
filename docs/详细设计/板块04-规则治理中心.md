# 智源 ANP 平台 · 规则治理中心详细设计

> **文档定位**：规则治理中心（Rule Governance Center）是平台的**心脏与护城河**。将企业制度、流程、标准从"墙上文件"结构化为可执行规则（Rule-as-Code），驱动并约束所有 AI 行为。本文定义其完整设计：定位价值、角色、功能模块、规则三层模型、规则 DSL、数据模型、规则引擎执行流程、结构化引擎、与各板块校验接口、人机闸门、KPI 与落地优先级。
>
> **技术基线**：Go（平台业务后端，Gin + sqlx + validator + viper + zap）+ Python（AI/Agent 运行时，LangGraph + LiteLLM）混合栈；规则引擎在 **Go 侧**（`rule` 模块），规则结构化的 AI 辅助在 **Python 侧**，Go 经 gRPC 调用；数据 PostgreSQL 多 schema + pgvector + Redis + MinIO。
>
> **配套文档**：软件开发架构设计、开发标准与规范、总体方案 V1.0。
>
> **关键决策**：规则内嵌为一等公民（ADR-003）；`project_space_id` 是贯穿的**多租户路由键**；公共规则全空间共享，项目私有规则只在本空间生效，**项目空间可叠加更严规则，不得放宽全局红线**。

## 1. 定位与价值

### 1.1 从"墙上文件"到"AI 硬约束"

传统企业治理的痛点：制度/流程/标准以 Word/PDF 形式存在于文件服务器、OA 公告、知识库，**条文无法被执行，违规只能事后审计**。AI 原生研发平台引入了更强的需求——AI Agent 自主产生代码、测试、变更、运维操作，如果没有**机器可读、可执行、可阻断**的规则约束，AI 的规模化自治将带来灾难性风险。

规则治理中心把治理范式从**事后追责**升级为**事前阻断 + 事中约束 + 事后追溯**：

| 传统治理 | 规则治理中心（RaC） |
|---|---|
| 制度是 Word/PDF，人读人执行 | 规则是结构化代码（YAML/JSON DSL），机器读机器执行 |
| 违规事后审计，已造成损失 | 违规事前阻断，不产生违规产出 |
| 依据模糊（"我觉得符合规范"） | 依据精确（违反 R-SEC-0001 v3 第 3.2 条） |
| 制度更新与执行脱节 | 规则版本化发布，**按当时版本可追溯审计** |
| 规则散落各部门，冲突无人发现 | 规则冲突自动检测，优先级明确 |

### 1.2 护城河与数据资产

⭐ **规则库（RaC）是平台最深的护城河，也是越用越值钱的数据资产**：

- **护城河**：企业花数月把制度结构化入库，沉淀的是**本企业独有的治理知识**，无法被通用 AI 平台复制。换模型、换编码工具都不影响规则资产；反之，带走规则资产才能快速重建平台能力。
- **数据资产**：规则库随业务演进不断丰富——新踩坑入规则、新合规要求入规则、审计发现的漏洞入规则。规则覆盖率从 20% → 95% 的过程，就是企业治理能力数字化的过程。
- **驱动 AI 行为**：规则不是"约束 AI"那么简单，更是**驱动 AI**——Agent 在执行任何任务前，先拉取生效规则集作为"操作依据"，规则定义了"必须做什么"（Process 应遵循）和"可以参考什么"（Knowledge），而不仅仅是"不能做什么"（Rule 强制）。

### 1.3 核心价值主张

| # | 价值 | 量化锚点 |
|---|---|---|
| 1 | 治理前置：违规产出不落地 | 违规阻断率 ≥ 99%（违规即阻断） |
| 2 | 合规可追溯：任意 AI 产出可查依据 | 审计查询响应 ≤ 2s，覆盖率 100% |
| 3 | 制度数字化：制度从文件变资产 | 制度数字化率阶段一 70%、阶段三 95% |
| 4 | 跨场景一致：同一规则约束所有 Agent | 规则一处发布，全 Agent 生效 |
| 5 | 冲突可见：规则矛盾自动检测 | 冲突检测召回率 ≥ 95% |

## 2. 用户与角色

### 2.1 角色矩阵

| 角色 | 职责 | 来源部门 | RaC 权限 | 典型场景 |
|---|---|---|---|---|
| **制度 Owner** | 提出本部门制度/流程/标准，录入原文 | 业务/IT/安全/法务/财务 | 录入、起草规则草案、本部门规则查看 | 法务录入《数据出境合规要求》；安全录入《编码安全规范》 |
| **规则架构师** | 把自然语言制度翻译为可执行规则（RaC），评审规则质量 | 平台架构组/资深架构师 | 规则全生命周期操作（起草/评审/发布）、DSL 编写 | 把"禁止硬编码密钥"翻译为 `when: secret_pattern_match → then: block` |
| **合规审计员** | 审计 AI 产出的规则合规性，追溯依据，核查规则有效性 | 安全/合规/内审 | 审计查询、合规报告、规则有效性分析 | 季度合规审计：抽查 100 个 PR 是否都过了安全规则校验 |
| **平台管理员** | 规则引擎配置、公共规则管理、冲突仲裁 | 平台运营 | 全局配置、公共规则、优先级仲裁 | 裁决安全规则与效率规则的冲突 |
| **研发工程师** | 消费规则（被约束），可提规则反馈 | 各研发团队 | 查看生效规则、提反馈/规则改进建议 | 编码 Agent 提示"违反 R-CODE-0012"后查看规则详情与原因 |

### 2.2 RACI 矩阵（规则全生命周期）

| 活动 | 制度 Owner | 规则架构师 | 合规审计员 | 平台管理员 | 系统自动 |
|---|---|---|---|---|---|
| 制度录入 | **A/R** | C | I | I | — |
| 规则结构化（NL→RaC） | C | **A/R** | I | I | R（AI 辅助） |
| 规则评审 | C | **A** | R | C | — |
| 规则发布 | I | **A** | C | I | R（版本化） |
| 高风险规则多角色确认 | C | R | **A** | R | — |
| 规则冲突仲裁 | I | R | C | **A** | R（检测告警） |
| 合规审计追溯 | I | C | **A/R** | I | R（查询） |
| 规则下线/废弃 | **A** | R | C | I | R（版本管理） |

> A=Accountable（最终负责） R=Responsible（执行） C=Consulted（咨询） I=Informed（知会）

## 3. 功能模块

规则治理中心共 **8 大功能模块**，覆盖从制度录入到规则执行再到审计追溯的完整闭环。

### 3.1 模块总览

```
│   制度→草稿    │   NL→可执行    │  Rule/Process  │  草稿→评审→发布  │
│               │               │   /Knowledge   │  →废弃(版本化)  │
├───────────────┼───────────────┼───────────────┼─────────────────┤
│ ⑤ 生效范围映射 │ ⑥ 规则校验与   │ ⑦ 合规审计追溯 │ ⑧ 规则冲突检测   │
│  环节×Agent×域 │    阻断引擎    │  按当时版本审计 │  新旧/交叉冲突   │
```

### 3.2 模块①制度/流程/标准录入工作台

**定位**：制度 Owner 把本部门的制度、流程、标准录入系统的入口。支持三种录入形态，降低门槛。

| 录入形态 | 适用场景 | 流程 |
|---|---|---|
| **文档上传** | 已有 Word/PDF/Markdown 制度文件 | 上传 → 文档解析（Python 侧）→ 抽取条款候选 → 人工确认 → 进入结构化队列 |
| **结构化录入** | 新建制度或逐条录入 | 表单填写（制度名/条款编号/条款正文/分类/标签）→ 保存为条款草案 |
| **对话式录入** | 不确定如何拆条的业务用户 | 与"制度录入助手"Agent 对话 → AI 引导拆分条款 → 确认后入库 |

**文档上传解析流程**（Python 侧负责，Go 经 gRPC 调用）：

```
制度Owner ──上传Word/PDF──▶ Go后端(MinIO存储)
                                │
                          gRPC: ParseDocument
                                ▼
                         Python AI服务
                         ├─ OCR/文本抽取
                         ├─ 章节结构识别
                         ├─ 条款切分（按"第X条"/编号）
                         ├─ 条款分类（强制/应遵循/参考）
                         └─ 候选规则抽取
                                │
                          返回条款候选列表
                                ▼
                        制度Owner确认/修正
                                │
                          保存为条款(draft)
                                ▼
                      进入"规则结构化队列"
```

**关键字段**（录入时的原始条款）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `document_id` | string | 制度文档唯一 ID（如 `DOC-SEC-2024-001`） |
| `document_name` | string | 制度名称（如《信息安全编码规范》） |
| `document_type` | enum | `policy` / `standard` / `process` / `regulation` |
| `department` | string | 归属部门（`security` / `legal` / `finance` / `it` / `business`） |
| `clause_no` | string | 条款编号（如 `3.2`） |
| `clause_text` | text | 条款原文 |
| `clause_category` | enum | AI 建议的分类（`mandatory` / `should` / `informative`），人审确认 |
| `status` | enum | `draft` / `structuring` / `structured` / `rejected` |

### 3.3 模块②规则结构化引擎

**定位**：把自然语言条款翻译为可执行规则（RaC）。这是规则治理中心的核心智力环节——**AI 辅助起草，规则架构师人审确认**。

详细设计见 [第 5 章](#5-规则结构化引擎设计)。

### 3.4 模块③规则三层模型

**定位**：并非所有规则都是硬阻断。按约束强度分三层，兼顾**安全可控**与**灵活高效**。

| 层级 | 类型 | 约束强度 | 违规后果 | 典型示例 |
|---|---|---|---|---|
| 🟥 **第一层** | **Rule（强制）** | 硬约束，不可违反 | **阻断**（产出不落地）+ 提示违反哪条 | 禁止硬编码密钥；禁止绕过 G5 发布生产；SQL 必须参数化 |
| 🟧 **第二层** | **Process（应遵循）** | 强期望，偏离需说明 | **告警**（产出落地但标记偏离）+ 要求记录原因 | 代码应附 ADR；PR 应绑定需求号；API 应有版本前缀 |
| 🟩 **第三层** | **Knowledge（参考）** | 软引导，推荐遵循 | **仅提示**（Agent 产出时附带推荐） | 推荐使用 shadcn/ui 组件；推荐函数复杂度 ≤ 10 |

**三层模型的执行逻辑**：

```
Agent 产出 ──▶ 规则引擎校验
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 🟥 Rule      🟧 Process   🟩 Knowledge
 (强制)       (应遵循)      (参考)
    │           │           │
    ▼           ▼           ▼
 命中→阻断   命中→告警    附带推荐
 (不落地)   (落地+标记)   (不阻断)
    │           │           │
    └─────┬─────┘           │
          ▼                 ▼
     触达人工闸门       产出携带推荐信息
     (G3/G4/G5/G6)     （Agent 可选择性采纳）
```

> ⭐ **设计要点**：Rule 层不可滥用——若所有规则都是硬阻断，AI 将寸步难行。Rule 层只放**不可妥协的红线**（安全/合规/资金/数据安全）；Process 放**研发规范**（偏离可解释即可）；Knowledge 放**最佳实践推荐**。三者比例建议 2:5:3。

### 3.5 模块④规则版本与生命周期

**定位**：规则全程版本化，**历史 AI 产出的审计按当时的规则版本**——这是合规审计可追溯的基础。

**生命周期状态机**：

```
        草稿                评审中             已发布               已废弃
       (draft)    ──提交──▶ (reviewing) ──通过──▶ (published) ──下线──▶ (deprecated)
          │                     │                     │
          │编辑                  │驳回                 │修订(产生新版本)
          ▼                     ▼                     ▼
       (draft)              (draft)            新版本 draft
                                                 旧版本自动
                                                 进入 superseded
```

| 状态 | 含义 | 谁能操作 | 规则引擎是否加载 |
|---|---|---|---|
| `draft` | 草稿，可反复编辑 | 规则架构师 | ❌ |
| `reviewing` | 评审中，锁定不可改 | 评审人 | ❌ |
| `published` | 已发布，生效中 | —（自动管理） | ✅ |
| `deprecated` | 已废弃，不再生效 | 制度 Owner/规则架构师 | ❌ |
| `superseded` | 被新版本取代 | —（自动管理） | ❌（审计可查） |

**版本规则**：

- 每条规则有 `rule_id`（永久不变）和 `version`（每次发布递增，如 `v1`, `v2`, `v3`）。
- 同一 `rule_id` 的已发布版本**同时只有一个**（新版本发布，旧版本自动 `superseded`）。
- 审计追溯时，根据 `audit_time` 查询当时处于 `published` 状态的版本——**版本不变，审计结论不变**。
- 版本变更记录完整留存：谁改了什么、何时、为什么（关联变更原因/Jira）。

### 3.6 模块⑤生效范围映射

**定位**：一条规则不是全局生效，而是绑定到**具体的环节、Agent、业务域、项目空间**。这是"精准约束"的基础——避免规则误伤无关场景。

**生效范围四维模型**：

```
生效范围 (RuleScope) = 环节(stage) × Agent(agent) × 业务域(business_domain) × 项目空间(project_space)

示例：
  规则 R-SEC-0001 的生效范围：
    stage:          ["coding", "code_review"]
    agent:          ["coding_agent", "review_agent"]
    business_domain: ["*"]              ← 所有业务域
    project_space:  ["*"]               ← 所有项目空间（公共规则）
    priority:       100                 ← 优先级（冲突时裁决）
```

| 维度 | 取值示例 | 说明 |
|---|---|---|
| `stage` | `requirement` / `architecture` / `coding` / `review` / `testing` / `security` / `deploy` / `ops` | 研发流水线环节（对应 G1-G6 闸门所在环节） |
| `agent` | `req_agent` / `arch_agent` / `coding_agent` / `review_agent` / `test_agent` / `deploy_agent` / `ops_agent` | 受约束的 Agent 角色 |
| `business_domain` | `finance` / `hr` / `customer_service` / `*` | 业务域，`*` 表示所有 |
| `project_space` | UUID / `*` | 项目空间 ID，`*` 表示公共规则全空间生效 |
| `priority` | 0-1000 | 优先级，冲突时高优先级规则覆盖低优先级 |

> ⭐ **公共 vs 私有**：`project_space = "*"` 是公共规则（全公司红线），由平台管理员管理；`project_space = 具体ID` 是项目私有规则。**项目私有规则可以比公共规则更严，不可以更松**（引擎校验，见 [8.4](#84-规则加载与缓存)）。

### 3.7 模块⑥规则校验与阻断引擎

**定位**：Agent 产出经规则引擎校验，命中 Rule 层即阻断并告知违反哪条。这是"AI 硬约束"的执行核心。

详细设计见 [第 8 章](#8-规则引擎执行流程go-侧)。

**阻断时的用户/Agent 体验**：

```
编码 Agent 产出代码 ──▶ 规则引擎校验 ──▶ 命中 R-SEC-0001
                                            │
                                            ▼
                                      阻断（代码不落地）
                                            │
                            返回违规详情给 Agent：
                            ┌─────────────────────────────┐
                            │ ❌ 违反规则: R-SEC-0001 v3    │
                            │ 名称: 禁止硬编码密钥          │
                            │ 来源: 《信息安全编码规范》3.2  │
                            │ 原因: 检测到 api_key 硬编码   │
                            │ 建议: 使用环境变量或密钥服务   │
                            │ 动作: block（需修正后重试）   │
                            └─────────────────────────────┘
                                            │
                              Agent 自动修正后重新提交
```

### 3.8 模块⑦合规审计与追溯

**定位**：任意 AI 产出（代码/测试/变更/运维操作）可查"**依据了哪些规则、当时是什么版本、是否合规**"。这是给合规审计员、监管、内审的能力。

**审计追溯流程**：

```
合规审计员 ──查询某个产出(如PR#123)──▶ 规则引擎
                                         │
                                         ▼
                              查 ComplianceAudit 记录
                              （产出时已留痕）
                                         │
                    ┌────────────────────┼────────────────────┐
                    ▼                    ▼                    ▼
              生效规则快照          校验结果(通过/违反)     留痕元数据
              (当时版本)            (命中了哪些/未命中)     (Agent/模型/trace)
                                         │
                                         ▼
                              生成合规审计报告：
                              "PR#123 于 2025-03-15 14:30 校验，
                               依据 R-SEC-0001 v3、R-CODE-0012 v1...，
                               共 8 条规则，全部通过，
                               Agent: coding_agent v2.1，trace: abc123"
```

**审计快照设计**（关键）：每次 Agent 产出校验时，不仅记录结果，还**快照当时的规则集版本**。这样即使规则后续更新，历史审计结论不受影响。

| 审计字段 | 说明 |
|---|---|
| `target_type` | 审计对象类型（`pr` / `requirement` / `deploy` / `ops_action`） |
| `target_id` | 对象 ID（如 PR 编号、变更单号） |
| `target_snapshot` | 对象内容快照（MinIO 路径，避免大对象入库） |
| `rules_evaluated` | 本次校验评估的规则列表（含版本号） |
| `rules_hit` | 命中的规则（违反/告警） |
| `decision` | 最终决策（`pass` / `block` / `warn` / `escalate`） |
| `evaluated_at` | 校验时间 |
| `rule_engine_version` | 规则引擎版本（引擎自身也版本化） |

### 3.9 模块⑧规则冲突检测

**定位**：多条规则可能存在矛盾（一个要求阻断、另一个要求放行）或冗余（重复约束）。发布前和运行时双重检测。

| 冲突类型 | 示例 | 检测时机 |
|---|---|---|
| **动作冲突** | R-A 说"阻断"，R-B 说"允许"，同一条件 | 发布前 + 运行时 |
| **条件矛盾** | R-A 要求 `complexity ≤ 10`，R-B 要求 `complexity ≤ 5` | 发布前 |
| **范围重叠** | 两条规则约束同一 stage+agent+domain，行为不同 | 发布前 |
| **冗余** | R-A 完全包含 R-B 的条件和范围 | 发布前（提示） |
| **公共-私有矛盾** | 公共规则要求阻断，项目私有规则试图放行 | 发布前（🟥 禁止） |

**冲突检测算法**（Go 侧，发布前执行）：

```
新规则 R_new 提交发布
  │
  ▼
遍历同 scope（stage∩agent∩domain∩project_space 非空交集）的已发布规则
  │
  ├─ 条件交集非空？──▶ 动作不同？──▶ 标记"动作冲突"（阻断发布）
  │                └─ 动作相同？──▶ 标记"冗余"（提示，可发布）
  ├─ 条件互斥？─────▶ 标记"条件矛盾"（提示，需人审）
  └─ 公共-私有─────▶ 私有规则比公共规则更松？──▶ 标记"违规"（🟥 阻断发布）
```

## 5. 规则结构化引擎设计

### 5.1 定位

规则结构化引擎把**自然语言制度条款**翻译为**可执行规则（RaC）**。这是整个治理中心智力密度最高的环节——AI 辅助起草，规则架构师人审确认。

### 5.2 架构

```
制度条款(draft)
    │
    │ Go 后端调 gRPC
    ▼
┌────────────────────────────────────────────┐
│       Python AI 服务（规则结构化引擎）        │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌─────────┐│
│  │ 条款理解  │─▶│ 规则抽取  │─▶│ DSL生成 ││
│  │ (LLM)    │  │ (LLM)    │  │ (LLM)  ││
│  └──────────┘  └──────────┘  └─────────┘│
│       │              │             │     │
│       ▼              ▼             ▼     │
│  ┌──────────────────────────────────┐   │
│  │  上下文增强：                     │   │
│  │  · 已有规则库(查重，避免重复)      │   │
│  │  · 规则DSL模板库(提高一致性)      │   │
│  │  · 企业术语表(消歧)              │   │
│  │  · 历史结构化案例(few-shot)      │   │
│  └──────────────────────────────────┘   │
│       │                                  │
│       ▼                                  │
│  输出：规则草案（YAML DSL + 置信度 + 解释）│
└────────────────────┬───────────────────────┘
                     │
                     ▼
              Go 后端（保存草案）
                     │
                     ▼
           规则架构师人审（可视化对比 NL ↔ DSL）
                     │
                ┌────┴────┐
                ▼         ▼
            确认通过   修改/驳回
                │
                ▼
            进入评审→发布
```

### 5.3 结构化步骤详解

| 步骤 | 输入 | AI 动作 | 输出 | 人审点 |
|---|---|---|---|---|
| **1. 条款理解** | 条款原文 | 识别条款意图（禁止/要求/推荐）、约束对象、适用场景 | 结构化意图描述 | 意图是否正确？ |
| **2. 分类判定** | 意图描述 | 判定层级：Rule(强制) / Process(应遵循) / Knowledge(参考) | 建议层级 + 置信度 | 层级是否恰当？（不能把推荐升为强制） |
| **3. 条件抽取** | 意图描述 | 识别"在什么条件下"触发（when），提取事实因子 | when 表达式（草案） | 条件是否完整？有无漏判？ |
| **4. 动作抽取** | 意图描述 | 识别"触发后做什么"（then: block/warn/allow/require_approval） | then 表达式 | 动作力度是否恰当？ |
| **5. 范围推断** | 条款上下文 | 推断生效的 stage/agent/business_domain | scope（草案） | 范围是否准确？有无误伤？ |
| **6. DSL 生成** | 上述全部 | 生成 YAML DSL 草案 | 完整规则 YAML | DSL 是否准确表达原意？ |
| **7. 查重** | DSL 草案 | 检索已有规则库，判断是否重复/冲突 | 相似规则列表 + 冲突提示 | 是否重复？需合并吗？ |

### 5.4 结构化示例

**输入（自然语言条款）**：

#### 6.2.3 RuleCondition（when 条件）

```go
// RuleCondition 是规则的条件表达式（when），支持 AND/OR/NOT 逻辑组合。
// 设计为树形结构，可递归求值。
type RuleCondition struct {
    // 逻辑组合（与叶子节点二选一）
    All []RuleCondition `json:"all,omitempty"`  // AND：所有子条件满足
    Any []RuleCondition `json:"any,omitempty"`  // OR：任一子条件满足
    Not *RuleCondition  `json:"not,omitempty"`  // NOT：取反
    
    // 叶子条件（与逻辑组合二选一）
    Fact     FactRef     `json:"fact,omitempty"`      // 事实因子引用
    Operator Operator    `json:"operator,omitempty"`  // 操作符
    Value    interface{} `json:"value,omitempty"`     // 比较值（可为 string/number/[]string/regex）
}

// FactRef 引用一个事实因子（由调用方在 Validate 时提供）
type FactRef string

// 示例 FactRef：
//   "code_diff.contains_secret_pattern"  —— 代码差异中是否包含密钥模式
//   "pr.target_branch"                   —— PR 目标分支
//   "pr.author_role"                     —— PR 作者角色
//   "deploy.environment"                 —— 部署环境
//   "ops.action_type"                    —— 运维操作类型
//   "requirement.estimated_budget"       —— 需求预估预算

type Operator string

const (
    OpEquals            Operator = "eq"
    OpNotEquals         Operator = "ne"
    OpGreaterThan       Operator = "gt"
    OpGreaterEq         Operator = "gte"
    OpLessThan          Operator = "lt"
    OpLessEq            Operator = "lte"
    OpIn                Operator = "in"
    OpNotIn             Operator = "not_in"
    OpContains          Operator = "contains"
    OpNotContains       Operator = "not_contains"
    OpMatches           Operator = "matches"      // 正则匹配
    OpMatchesAny        Operator = "matches_any"  // 匹配任一正则
    OpExists            Operator = "exists"
    OpBefore            Operator = "before"       // 时间比较
    OpAfter             Operator = "after"
)
```

## 7. 规则 DSL 设计

### 7.1 DSL 设计原则

| 原则 | 说明 |
|---|---|
| **声明式** | 规则描述"什么条件做什么"，不描述"怎么做"（执行由引擎负责） |
| **机器可执行** | Go 引擎可直接解析执行，无需 LLM 参与（运行时零 AI 延迟） |
| **人可读** | 规则架构师能读懂 YAML，便于人审 |
| **可组合** | 支持条件嵌套（AND/OR/NOT），表达复杂逻辑 |
| **可溯源** | 每条规则必须标明来源（制度文档+条款） |

### 7.2 YAML DSL 完整示例

#### 示例 1：安全强制规则（Rule 层）

```yaml

  version: v3
  name: 禁止硬编码敏感凭证
  description: 源代码中不得硬编码 API 密钥、密码、Token 等敏感信息
  type: Rule                    # 🟥 强制层
  category: security
  severity: critical
  priority: 100                 # 最高优先级
  
  scope:
    stages: [coding, code_review]
```

### 8.1 规则引擎架构

### 8.2 核心接口（Go）

```go
// RuleEngine 是规则引擎的核心接口。
// 被所有板块的 Agent（经统一校验 API）调用。
type RuleEngine interface {
    // Validate 校验一个产出是否合规。
    // facts 是由调用方提供的事实因子（如 code_diff、pr 信息等）。
    // 返回 ValidationResult + 命中的规则列表。
    Validate(ctx context.Context, req ValidationRequest) (*ValidationResult, error)
    
    // GetActiveRules 获取指定 scope 下当前生效的规则集。
    // Agent 在产出前可调用此接口，将规则注入 prompt（约束自身行为）。
    GetActiveRules(ctx context.Context, scope RuleScope) ([]Rule, error)
    
    // BatchValidate 批量校验（一次产出校验多条规则，优化性能）。
    BatchValidate(ctx context.Context, reqs []ValidationRequest) (*BatchValidationResult, error)
}

// ValidationRequest 是校验请求
type ValidationRequest struct {
    ProjectSpaceID string                 `json:"project_space_id"`
    Stage          string                 `json:"stage"`           // coding/review/testing/...
    AgentID        string                 `json:"agent_id"`
    
    TargetType     string                 `json:"target_type"`     // pr/requirement/deploy/ops_action
    TargetID       string                 `json:"target_id"`
    
    Facts          map[string]interface{} `json:"facts"`           // 事实因子（由调用方填充）
    TraceID        string                 `json:"trace_id"`
}

// ValidationResult 是校验结果
type ValidationResult struct {
    Decision   ValidationDecision `json:"decision"`     // pass/block/warn/escalate
    Hits       []RuleHit           `json:"hits"`         // 命中的规则
    Evaluated  int                 `json:"evaluated"`    // 评估的规则数
    AuditID    string              `json:"audit_id"`     // 审计记录ID
    EngineVer  string              `json:"engine_version"`
}

// RuleHit 是命中的单条规则
type RuleHit struct {
    RuleID     string     `json:"rule_id"`
    Version    string     `json:"version"`
    Name       string     `json:"name"`
    Type       RuleType   `json:"type"`
    Action     ActionType `json:"action"`
    Message    string     `json:"message"`
    Suggestion string     `json:"suggestion"`
    Severity   Severity   `json:"severity"`
    Source     *RuleSource `json:"source"`   // 溯源（违反了哪份制度哪条）
}
```

### 8.3 条件求值器（Evaluator）

条件求值器递归求值 `RuleCondition` 树：

```go
// Evaluator 递归求值条件表达式树。
type Evaluator struct{}

func (e *Evaluator) Evaluate(cond *RuleCondition, facts map[string]interface{}) (bool, error) {
    // 逻辑组合节点
    if len(cond.All) > 0 {
        for _, sub := range cond.All {
            matched, err := e.Evaluate(&sub, facts)
            if err != nil {
                return false, err
            }
            if !matched {
                return false, nil // AND：任一不满足即 false
            }
        }
        return true, nil
    }
    
    if len(cond.Any) > 0 {
        for _, sub := range cond.Any {
            matched, err := e.Evaluate(&sub, facts)
            if err != nil {
                continue // OR：忽略求值错误，继续尝试
            }
            if matched {
                return true, nil
            }
        }
        return false, nil
    }
    
    if cond.Not != nil {
        matched, err := e.Evaluate(cond.Not, facts)
        return !matched, err
    }
    
    // 叶子节点：从 facts 取值，按 operator 比较
    return e.evaluateLeaf(cond, facts)
}

func (e *Evaluator) evaluateLeaf(cond *RuleCondition, facts map[string]interface{}) (bool, error) {
    factValue, exists := facts[string(cond.Fact)]
    if !exists {
        // 事实因子不存在时：默认 false（安全侧 fail-closed）
        return false, nil
    }
    
    switch cond.Operator {
    case OpEquals:
        return reflect.DeepEqual(factValue, cond.Value), nil
    case OpNotEquals:
        return !reflect.DeepEqual(factValue, cond.Value), nil
    case OpIn:
        // factValue in cond.Value([]interface{})
        return contains(cond.Value, factValue), nil
    case OpMatches:
        regex := regexp.MustCompile(cond.Value.(string))
        return regex.MatchString(fmt.Sprintf("%v", factValue)), nil
    case OpMatchesAny:
        patterns := cond.Value.([]interface{})
        str := fmt.Sprintf("%v", factValue)
        for _, p := range patterns {
            regex := regexp.MustCompile(p.(string))
            if regex.MatchString(str) {
                return true, nil
            }
        }
        return false, nil
    case OpGreaterThan, OpLessThan, OpGreaterEq, OpLessEq:
        return e.compareNumeric(factValue, cond.Value, cond.Operator)
    // ... 其他 operator
    }
    return false, fmt.Errorf("unknown operator: %s", cond.Operator)
}
```

> ⭐ **fail-closed 设计**：当事实因子不存在或求值出错时，**默认返回 false（不命中）**。这避免"因数据缺失而误阻断"，但也意味着调用方必须提供完整的事实因子——这是 FactProvider 接口的职责。

### 8.4 规则加载与缓存

生效规则的加载采用**两级缓存**（Redis + 内存），保证毫秒级校验。

```
Validate 请求进来
    │
    ▼
[内存缓存 L1] ──命中──▶ 直接使用规则集
    │ 未命中
    ▼
[Redis 缓存 L2] ──命中──▶ 加载到 L1 ──▶ 使用
    │ 未命中（或版本过期）
    ▼
[PostgreSQL] 查询 status=published 的规则
    │ 按 scope 匹配（stage ∩ agents ∩ domains ∩ project_space）
    │ 优先级排序
    ▼
写入 Redis（key: rule:active:{project_space}:{stage}:{agent}）
    │ TTL 5 分钟 + 规则发布时主动失效
    ▼
写入内存 L1（TTL 60s）
```

**规则发布时的缓存失效**：

```go
// 规则发布时，主动推送缓存失效事件
func (s *RuleService) PublishRule(ctx context.Context, ruleID, version string) error {
    // 1. 更新规则版本状态
    // 2. 旧版本 → superseded，新版本 → published
    // 3. 发布 RulePublishedEvent 领域事件
    s.eventBus.Publish(ctx, Event{
        Type:    "RulePublished",
        Payload: RulePublishedPayload{RuleID: ruleID, Version: version},
    })
    
    // 4. 失效相关缓存（Redis 删除 + 通知各实例清 L1）
    s.cacheInvalidator.Invalidate(ctx, scope)
    return nil
}
```

**公共-私有规则叠加校验**：

```go
// 加载生效规则时，同时加载公共规则和项目私有规则。
// 关键：项目私有规则只能更严，不能更松（引擎校验）。
func (s *RuleService) LoadActiveRules(ctx context.Context, scope RuleScope) ([]Rule, error) {
    // 查公共规则（project_space = '*'）
    publicRules := s.repo.FindByScope(ctx, scope.WithProjectSpace("*"))
    
    // 查项目私有规则（project_space = 具体 ID）
    privateRules := s.repo.FindByScope(ctx, scope)
    
    // 合并 + 冲突检测：
    // 若私有规则与公共规则作用域重叠，私有规则的 priority 必须 ≥ 公共规则
    // 且私有规则的动作不得比公共规则更松（block → warn 不允许）
    merged := s.mergeRules(publicRules, privateRules)
    
    return merged, nil
}
```

### 8.6 事实因子（Fact）体系

规则引擎不直接访问代码/PR/部署数据，而是由**调用方提供事实因子（Facts）**。这保证引擎的通用性与边界清晰。

| Fact 命名空间 | 提供方 | 示例 |
|---|---|---|
| `code_diff.*` | 研发工作台（板块2） | `code_diff.contains_pattern`, `code_diff.file_extension`, `code_diff.lines_changed` |
| `pr.*` | 研发工作台（板块2） | `pr.target_branch`, `pr.author_role`, `pr.linked_requirement_id` |
| `test.*` | 测试中心（板块3） | `test.coverage_pct`, `test.failed_count`, `test.has_boundary_cases` |
| `deploy.*` | 发布中心（板块6） | `deploy.environment`, `deploy.timestamp`, `deploy.cab_approved` |
| `ops.*` | 运维中心（板块7） | `ops.action_type`, `ops.target_system`, `ops.data_classification` |
| `requirement.*` | 需求工作台（板块1） | `requirement.estimated_budget`, `requirement.priority` |
| `context.*` | 调用方通用 | `context.project_space_id`, `context.user_role`, `context.timestamp` |

> **Fact 注册表**：所有可用的事实因子在 `rule` 模块维护一个注册表（Fact Registry），供规则架构师在编写 DSL 时参考。新板块接入时注册新的事实因子。

```protobuf
  // 获取生效规则集（Agent 产出前调用，注入 prompt）
  rpc GetActiveRules(GetActiveRulesRequest) returns (GetActiveRulesResponse);
  
  // 批量校验（一次校验多个产出，优化 gRPC 往返）
  rpc BatchValidate(BatchValidateRequest) returns (BatchValidateResponse);
  
  // 查询合规审计（合规审计员调用）
  rpc QueryComplianceAudit(QueryAuditRequest) returns (QueryAuditResponse);
}

message ValidateRequest {
  string project_space_id = 1;
  string stage = 2;
  string agent_id = 3;
  string target_type = 4;
  string target_id = 5;
  map<string, string> facts = 6;     // 事实因子（复杂值用 JSON 序列化）
  string trace_id = 7;
}

message ValidateResponse {
  string decision = 1;               // pass/block/warn/escalate
  repeated RuleHit hits = 2;
  int32 evaluated_count = 3;
  string audit_id = 4;
  string engine_version = 5;
}

message RuleHit {
  string rule_id = 1;
  string version = 2;
  string name = 3;
  string type = 4;
  string action = 5;
  string message = 6;
  string suggestion = 7;
  string severity = 8;
  RuleSource source = 9;
}

message GetActiveRulesRequest {
  string project_space_id = 1;
  string stage = 2;
  string agent_id = 3;
}

  repeated ActiveRule rules = 1;
  string cache_version = 2;
```

### 9.3 Agent 调用校验的两种模式

| 模式 | 时机 | 说明 |
|---|---|---|
| **预防式（Pre-check）** | Agent 产出**前** | Agent 调 `GetActiveRules`，将规则摘要注入 prompt，**约束自身行为**，避免违规 |
| **校验式（Post-check）** | Agent 产出**后** | Agent 调 `Validate`，引擎评估产出，命中 Rule 则阻断 |

⭐ **两种模式必须同时使用**：预防式降低违规概率，校验式保证违规不落地。这是"AI 硬约束"的双保险。

## 10. 人机闸门

### 10.1 规则生命周期中的闸门

规则本身作为约束所有 AI 行为的"法律"，其**发布和变更**也必须经过严格的闸门管控。

```
规则草案(draft)
    │
    │ 提交评审
    ▼
🚪 G-R1 规则评审（规则架构师 + 合规审计员）
    │ 评审内容：
    │   · 规则是否准确表达制度原意？
    │   · DSL 条件是否完整，有无漏判？
    │   · 动作力度是否恰当（block/warn/allow）？
    │   · 生效范围是否准确，有无误伤？
    │   · 是否与现有规则冲突？
    │
    ├─ 通过 ──▶ 🚪 G-R2 发布审批
    └─ 驳回 ──▶ 返回 draft
                │
                ▼
🚪 G-R2 发布审批（制度 Owner 确认 + 规则架构师批准）
    │
    ├─ 通过 ──▶ 发布（版本化，旧版本 superseded）
    └─ 高风险 ──▶ 升级 🚪 G-R3
                    │
                    ▼
            🚪 G-R3 高风险规则多角色确认（L3 双确认）
```

### 10.2 高风险规则识别与升级

以下规则变更属于**高风险**，必须升级 L3 多角色确认：

| 高风险情形 | 示例 | 需确认角色 |
|---|---|---|
| 🟥 **放宽安全策略** | 将硬编码密钥规则从 block 降级为 warn | 安全负责人 + 规则架构师 + 合规审计员 |
| 🟥 **放宽合规要求** | 放宽数据出境校验、减少审计字段 | 合规审计员 + 法务 + 规则架构师 |
| 🟥 **调整闸门等级** | 将某类发布从 G5 降为 G1 | 闸门负责人 + 平台管理员 |
| 🟥 **修改资金/数据安全规则** | 放宽生产数据操作约束 | 安全负责人 + 业务 Owner + 平台管理员 |
| 🟥 **下线/废弃公共红线规则** | 废弃全局安全红线 | 平台管理员 + 原制度 Owner + 合规审计员 |

**高风险检测规则**（引擎自动判定）：

```go
func (s *RuleService) assessRisk(rule *Rule, oldVersion *RuleVersion) RiskLevel {
    // 1. 安全/合规类规则，动作从 block 降为 warn/allow → 高风险
    if rule.Category == "security" || rule.Category == "compliance" {
        if oldVersion.Action.ActionType == ActionBlock &&
           rule.Action.ActionType != ActionBlock {
            return RiskHigh
        }
    }
    
    // 2. 公共规则（project_space="*"）被废弃 → 高风险
    if rule.Status == RuleStatusDeprecated && rule.ProjectSpaceID == "*" {
        return RiskHigh
    }
    
    // 3. 降低闸门等级 → 高风险
    if oldVersion.Action.EscalateGate > rule.Action.EscalateGate {
        return RiskHigh
    }
    
    // 4. 生效范围扩大 → 中风险
    if s.scopeExpanded(oldVersion.Scope, rule.Scope) {
        return RiskMedium
    }
    
    return RiskLow
}
```

### 10.3 规则发布审计

规则发布本身也是审计对象（`rule_audit_entries` 表），记录：

- 谁在何时发布/修改/废弃了什么规则
- 变更 diff（新旧版本对比）
- 评审人和审批人链
- 变更原因

> **规则即法律，法律的制定比法律的执行更需要透明**——规则变更记录对合规审计员完全开放。

## 11. KPI 与度量

### 11.1 核心 KPI

| KPI | 定义 | 计算方式 | 阶段一目标 | 阶段三目标 |

| **规则误判率** | 规则阻断后被 waive（人工豁免）的比例 | `waived violations / total violations` | ≤ 10% | ≤ 3% |
| **规则冲突数** | 活跃冲突数量 | 冲突检测未解决的条数 | 趋降 | 0 |

## 12. 落地优先级（P0）

### 12.1 为什么是 P0

> **先建护栏，再放车跑。**
