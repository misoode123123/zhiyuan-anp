# 智源 ANP 平台 · 功能增强 · 编码规范（详细设计）

> **文档定位**：研发工作台 + 规则治理中心的联合增强。为 AI 编码注入**全局编码规范**与**项目级编码方法/规范**，让产出按企业/项目的约定生成。属「生成指导」，与板块4「规则治理中心」的「硬约束」互补。
> **技术基线（当前实现层）**：业务后端 Go（Gin + sqlx + SQLite）；编码引擎 opencode + 智谱 GLM；前端 Next.js + TS。本文档对应**当前 M1 实现**，非愿景 gRPC/LangGraph 架构。
> **多租户**：所有数据带 `project_space_id` 作为路由键；全局规范以 `project_space_id IS NULL` 表达。

---

## 1. 定位与价值

### 1.1 定位

「编码规范」是**注入式生成指导**：在 AI 编码前，把企业全局规范与项目级编码方法以自然语言拼进编码 prompt，指导 AI「怎么写」。它不拦截、不阻断，只提升产出与约定的一致性。

### 1.2 价值

| 价值点 | 说明 |
|---|---|
| 风格统一 | 全局规范保证全平台产出遵循同一套编码标准（命名、结构、安全基线） |
| 项目适配 | 项目级规范让同一平台为不同技术栈（FastAPI / Gin / 前端）给出针对性写法 |
| 规则前置 | 把「期望」前置成 prompt 指导，减少事后规则 block 的返工 |

---

## 2. 与「规则治理中心(RaC)」的分工

二者**并存互补**，职责不重叠：

| 维度 | 编码规范（本设计） | RaC 规则（板块4，已实现） |
|---|---|---|
| 性质 | 生成指导（软） | 硬约束 |
| 形式 | 自然语言条文 | 正则条件 + `condition_field` |
| 作用 | 注入 prompt，告诉 AI「该怎么写」 | 校验 prompt/output，命中 `block` 拒绝 |
| 层级 | 全局 + 项目级 | 全局（按 `scope` 区分域） |
| 时机 | 编码前注入 | 编码前 `checkRules` 校验 |
| 落地 | 拼 prompt | `rule.Engine.Check` + `HasBlock` |

**执行流（编码派发）**：取生效规范拼进 prompt → `checkRules`(block 校验) → opencode 编码 → 登记 change → G3 审批。规范在前段指导生成，规则在校验段拦截违规。

## 3. 数据模型

单表 + 层级（方案 A）。`project_space_id` 为 `NULL` 表全局，非空表项目级。

```sql
CREATE TABLE IF NOT EXISTS coding_standard (
  id               TEXT PRIMARY KEY,
  project_space_id TEXT NULL,          -- NULL=全局；非空=该项目空间专属
  name             TEXT NOT NULL,      -- 规范名，如「Python 后端编码规范」
  category         TEXT NOT NULL DEFAULT 'general',  -- general/language/framework/security/testing
  content          TEXT NOT NULL,      -- 规范正文，自然语言/Markdown
  priority         INTEGER NOT NULL DEFAULT 100,     -- 小者先注入、先展示
  enabled          INTEGER NOT NULL DEFAULT 1,
  created_at       TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at       TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (project_space_id) REFERENCES project_space(id)
);
CREATE INDEX IF NOT EXISTS idx_coding_standard_ps ON coding_standard(project_space_id);
```

**字段说明**：
- `category` 取值固定五类：`general`（通用）、`language`（语言特定）、`framework`（框架）、`security`（安全）、`testing`（测试）。仅用于前端分组展示与筛选，注入时不区分类别。
- `priority` 升序；同 `priority` 按 `created_at` 升序。
- `enabled=0` 的规范不注入、不在 effective 列表出现。

## 4. 注入逻辑与执行流

### 4.1 新增 `internal/standard` 包

- `model.go`：`Standard` 结构（对应上表字段，`ProjectSpaceID` 用 `*string` 兼容 NULL）。
- `store.go`：
  - `Create / Update / Delete / Get`
  - `ListGlobal(ctx)` — `WHERE project_space_id IS NULL`
  - `ListByProjectSpace(ctx, psID)` — `WHERE project_space_id=?`
  - `ListEffective(ctx, psID)` — `WHERE enabled=1 AND (project_space_id IS NULL OR project_space_id=?) ORDER BY priority, created_at`
- `service.go`（可选薄封装）：`BuildPromptSection(ctx, psID) string` — 取 effective 列表，拼成规范段落。

### 4.2 注入点：收敛到 `dev.CodingAgent.Submit`

**单一注入点**。`Submit` 已被需求派发（`requirement.Dispatch` → `buildCodePrompt`）和手动派发（`POST /code`）共同调用，在此注入可全覆盖，无需改动 `buildCodePrompt`。

`CodingAgent` 新增字段 `standards *standard.Store`（构造函数注入）。`Submit` 内部流程调整为：

```
1. checkRules(prompt)              — 既有，block 则拒绝
2. standards := a.standards.ListEffective(ctx, psID)
3. finalPrompt := prompt + "\n\n" + buildStandardSection(standards)
4. opencodeRun(repoDir, finalPrompt, model)   — 用 finalPrompt
5. MarkCompleted → 登记 change      — 既有
```

`buildStandardSection` 拼装格式（追加在原 prompt 之后）：

```
【编码规范·必须遵循】
[全局][general] 产出五约束：可校验/可追溯/可回滚/守边界/守权限
[全局][security] 密钥与敏感信息不得硬编码
[项目][language] Python 用 FastAPI + Pydantic v2，命名 snake_case
[项目][testing] 每个公开函数配 pytest 单测
```

每行前缀 `[全局]/[项目]` + `[category]`，便于 AI 区分来源与类型。

> **注**：`opencodeRun` 签名不变（仍接收单个 prompt 字符串），只是传入的 prompt 已含规范段落。`codetask.Task.Prompt` 字段记录**原始 prompt**（不含规范），规范注入属于运行期行为，不持久化到任务记录，避免冗余。

### 4.3 不注入的场景

明确不注入：需求规格生成（`generateSpec`，调 GLM 出需求 JSON）、测试用例生成、发布。规范只作用于「编码」。

## 5. 后端 API

路由前缀 `/api/v1`，复用 `AuthUser` / `ProjectSpaceInjector` 中间件。

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/standards` | 列全局规范 |
| POST | `/standards` | 建全局规范（`project_space_id` 留空） |
| GET | `/standards/:id` | 取单条 |
| PUT | `/standards/:id` | 改（含启用/禁用） |
| DELETE | `/standards/:id` | 删 |
| GET | `/project-spaces/:id/standards` | 列该项目空间的项目级规范 |
| POST | `/project-spaces/:id/standards` | 建项目级规范 |
| GET | `/project-spaces/:id/standards/effective` | 预览该空间**实际生效**（全局+项目级叠加）列表 + 拼出的 prompt 片段 |

- 全局规范接口的多租户：全局规范不属于任何空间，`project_space_id` 强制为 NULL（接口内忽略请求体里的该字段，与 workspace 模块"租户键只取路径/不信任请求体"原则一致）。
- 入参校验：`name`、`content` 非空；`category` 限定五类；`priority` ≥ 0。
- 权限：M1 不强制 `Require`（与现有各域一致，避免锁死匿名体验）；框架就绪后可给写操作加 `rule_architect`/`admin`。

## 6. 前端：治理中心「编码规范」Tab

在 `app/governance/page.tsx` 现有规则管理之上增加 Tab 切换：「规则(RaC)」/「编码规范」。

**编码规范 Tab 结构**：
- 顶部：项目空间切换（复用全局 `ProjectSpace` 选择器）
- 分区一「全局规范」：`GET /standards`
- 分区二「当前项目空间规范」：`GET /project-spaces/:id/standards`
- 每条卡片：`name` + `category` 彩色标签 + `[全局]/[项目]` 标 + `enabled` 开关（PUT 切换） + 编辑/删除
- 新建/编辑表单：`name`、`category`（下拉五类）、`content`（textarea，支持多行）、`priority`（数字）
- 「预览生效规范」按钮：调 `/standards/effective`，展示叠加列表 + 拼出的 prompt 片段（只读），用于确认 AI 实际会看到什么

**视觉**：沿用现有 governance 页 `rounded-md border bg-white p-3` 风格；`category` 用不同色标区分（general 灰 / language 蓝 / framework 紫 / security 红 / testing 绿）。

  2. `name=安全基线`, `category=security`, `content=密钥与敏感信息不得硬编码；外部输入必须校验`

## 8. 验收标准

1. 建一条全局规范、一条项目级规范，在「预览生效规范」中能看到两者叠加，且 prompt 片段含 `[全局]`/`[项目]` 前缀。
2. 派发一次编码（需求派发或手动），`opencode` 实际收到的 prompt 包含规范段落（可通过临时打印或 completed 任务的产出回显验证）。
3. `enabled=0` 的规范不出现在 effective 列表、不注入。
4. 全局规范对**所有**项目空间生效；项目级规范仅对该空间生效、不影响其他空间。
5. 规范注入与 RaC `block` 规则互不干扰：命中 block 规则仍拒绝编码（规范注入发生在 checkRules 之后）。
