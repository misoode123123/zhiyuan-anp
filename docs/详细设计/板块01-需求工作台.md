# 智源 ANP 平台 · 板块 1 · 需求工作台（详细设计）

> **文档定位**：业务部门唯一主入口——非技术人用自然语言对话提需求，AI 把"业务问题"变成"可执行规格"，并全程跟踪到上线。是 9 大板块链路的**第一环**，定义研发主链路的输入契约。
> **技术基线**：业务后端 Go（Gin + sqlx + zap）；需求澄清/规格生成/查重/预估由 **Python 侧 LangGraph Agent** 承载，Go 经 **gRPC** 调用。持久化 PostgreSQL（schema: `requirement`）+ pgvector（需求向量）+ MinIO（原型/附件）。
> **配套**：软件开发架构设计、开发标准与规范、总体方案 V1.0。
> **多租户**：所有数据带 `project_space_id` 作为路由键。

---

## 1. 定位与价值

### 1.1 定位

需求工作台是平台的**业务侧总入口**。业务人员全程只在这里活动：提需求、看进度、给反馈、做确认。它的核心使命是把"业务语言"无损地翻译成"工程可执行规格"，并让非技术人能看懂全程进度。

```
业务人员                 需求工作台                     研发流水线
  │  说话提需求             │  AI澄清+生成规格           │
  │ ─────────────────────▶ │ ────────────────────────▶ │
  │  看进度卡片/人话        │  全程跟踪+闸门G1           │
  │ ◀───────────────────── │ ◀──────────────────────── │
```

### 1.2 价值（可量化目标）

| 价值点 | 指标 | 18 个月目标 |
|---|---|---|
| 需求传递零损耗 | 澄清后规格一次通过率 | ≥ 70% |
| 业务自助 | 业务侧自主推进的需求占比 | 40%→70% |
| 防重复造轮子 | 查重命中率 | ≥ 60%（命中即提示复用） |
| 全程透明 | 需求→上线周期 | ≤ 3 周（标准需求） |
| 业务满意 | 需求交付 NPS | ≥ 40 |

## 2. 用户与角色

| 角色 | 身份 | 在本板块的活动 | 权限边界 |
|---|---|---|---|
| 业务需求方 | 业务部门（非技术） | 对话提需求、看进度、确认范围/优先级/预算 | 仅本空间；可创建/编辑自己的需求 |
| 产品经理 | 技术侧桥梁 | 评审规格、补充非功能需求、协调排期 | 可编辑本空间所有需求 |
| 规则架构师 | 治理人员 | 评审合规红线命中、需求模板符合度 | 只读需求；可阻断违规需求 |
| 研发代表 | 下游消费方 | 接收已确认需求规格、反馈可行性 | 只读；写可行性结论 |
| 平台管理员 | 运营 | 看板、配额、模板维护 | 跨空间只读 |

> **L0-L3 自治等级映射**：需求采集/规格生成/查重/预估属 L2（AI 担主体，人审结果）；G1 需求确认属 L0（纯人工决策）。

## 3. 功能模块

### 3.1 对话式需求采集（核心）

- 业务人员打开需求工作台，选项目空间后进入**对话窗口**，自然语言描述业务问题。
- AI 多轮澄清：主动追问**歧义点、边界、权限、数据来源、异常路径**，而非一次性生成。
- 对话全程留痕（`ClarificationLog`），每一轮可回溯。
- 支持：文字、语音转文字、贴图（业务流程截图/表格）、上传文档（自动抽取要点）。

### 3.2 需求规格自动生成

AI 根据澄清结果，套用**需求模板标准**（来自规则治理中心）生成结构化规格：

| 规格组成部分 | 内容 | 形态 |
|---|---|---|
| 用户故事 | 作为 X，我想 Y，以便 Z | 文本（结构化） |
| 验收标准 | Given/When/Then 场景 | 可测试条目 |
| 业务流程图 | 主流程 + 异常分支 | Mermaid/ASCII |
| 线框原型 | 关键界面草图 | 渲染图（MinIO 存图） |
| 非功能需求 | 性能/安全/合规/可用性 | 结构化清单 |
| 数据对象 | 涉及实体、字段、来源 | 草案实体清单 |

### 3.3 需求查重与关联

- 对新需求做**向量相似检索 + 语义匹配**，检索历史需求库与已上线功能。
- 命中后给出：复用建议（已实现/部分实现/重复提案）、关联关系（依赖/冲突/合并候选）。
- 业务侧确认"确实重复"则合并；确认"看似重复实则不同"则标记差异点。

### 3.4 工期与算力预估

- 预估 Agent 回归历史需求工时 + 模型/算力消耗，给出**区间估算**（非单点，避免假精确）。
- 输出：预估人天区间、预估算力（token/GPU 时）、置信度、参考的相似历史需求。

### 3.5 需求池与优先级看板

- 业务自助拖拽排序优先级；支持 MoSCoW（Must/Should/Could/Won't）或加权评分。
- 看板视图：待澄清 / 待确认(G1) / 已确认待排期 / 研发中 / 已上线。
- 优先级变更留痕，可回溯。

### 3.6 全程进度透视（业务侧）

- 业务看到的不是技术 ticket，而是**进度卡片 + 人话**：
  - "你的需求正在测试，预计 2 天后上线"
  - "测试发现 3 个问题，正在修复，预计延后 1 天"
- 进度卡片由研发工作台/测试中心/发布中心的领域事件驱动，需求工作台只做**翻译展示**，不持有研发状态细节。

## 4. 核心业务流程

### 4.1 主流程：从对话到入池

```
┌──────────┐
│ 业务对话  │ 业务人员描述业务问题
└────┬─────┘
     ▼
┌──────────────────┐    追问/补全
│ 需求澄清 Agent    │ ◀────────── 业务回答
│ (Python/LangGraph)│ ──────────▶ (多轮, 最多 N 轮)
└────┬─────────────┘
     │ 澄清记录 + 结构化要点
     ▼
┌──────────────────┐
│ 需求查重 Agent    │ ── 向量检索历史需求库(pgvector)
└────┬─────────────┘
     │ 命中报告(复用/关联/冲突)
     ▼
┌──────────────────┐
│ 需求规格 Agent    │ ── 套需求模板标准(读规则中心)
│ (生成用户故事/AC/ │ ── 业务术语表/领域本体对齐
│  流程图/原型)     │ ── 合规红线预检(规则引擎)
└────┬─────────────┘
     │ 草案规格
     ▼
┌──────────────────┐
│ 预估 Agent        │ ── 回归历史工时/算力
└────┬─────────────┘
     │ 工期+算力区间+置信度
     ▼
┌──────────────────────────────────────────┐
│ 🚪 G1 需求确认 (人工闸门, L0)              │
│   AI 起草范围/优先级/预算草案              │
│   业务拍板: 范围 ✗/✓  优先级  预算         │
│   规则架构师: 合规 ✗/✓                    │
└────┬─────────────────────────────────────┘
     │ 确认通过
     ▼
┌──────────────────┐
│ 排期入池           │ ── 需求池 + 优先级看板
└────┬─────────────┘
     │ 发事件 requirement.confirmed
     ▼
┌──────────────────┐
│ 进入研发工作台    │ 板块2 消费需求规格
└──────────────────┘
```

### 4.2 变更与补充流程

任意时刻业务可对话变更/补充需求 → 触发**影响分析**（影响哪些已开发模块/已排期任务）→ 变更评审（轻量 G1 复核）→ 同步研发工作台。

### 4.3 进度回传流程

```
研发/测试/发布 领域事件 ──▶ 需求工作台(订阅)
                               │
                               ▼
                      进度翻译器(规则+模板)
                               │
                               ▼
                      进度卡片(人话) ──▶ 业务看板
```

## 5. AI Agent 设计

> 所有 Agent 在 **Python 侧**（agent-runtime 服务，LangGraph 编排），Go 后端经 gRPC 调用。Agent 产出遵守五约束（可校验/可追溯/可回滚/守边界/守权限）。

### 5.1 Agent 清单

| Agent | 语言/运行时 | 职责 | 输入 | 输出 | Go 调用入口(gRPC) |
|---|---|---|---|---|---|
| 需求澄清 Agent | Python/LangGraph | 多轮追问歧义、补全边界 | 对话历史 + 术语表 + 本体 | 结构化澄清要点 + 追问问题 | `ClarifyRequirement` |
| 需求规格 Agent | Python/LangGraph | 生成用户故事/AC/流程图/原型 | 澄清要点 + 需求模板 | 结构化规格 JSON + 原型图 | `GenerateSpec` |
| 需求查重 Agent | Python/LangGraph + pgvector | 向量+语义检索历史需求 | 新需求向量 + 元数据 | 命中列表 + 复用建议 | `DedupRequirement` |
| 预估 Agent | Python/LangGraph | 回归历史工时/算力 | 规格摘要 + 历史样本 | 工期/算力区间 + 置信度 | `EstimateEffort` |

### 5.2 澄清 Agent 的 LangGraph 状态机（要点）

```
[receive] ─▶ [extract_entities] ─▶ [check_ambiguity]
                                        │
                            ┌───────────┴────────────┐
                            ▼                        ▼
                    [ambiguity?] ─ yes ─▶ [ask_question] ─▶ [wait_user] ─▶ 回到 receive
                            │ no
                            ▼
                    [summarize_points] ─▶ [emit_clarification_result]
```

- **终止条件**：无新增歧义点 或 达到最大轮次（防无限循环，默认 ≤ 8 轮）。
- **不确定性声明**：低置信度歧义点主动标记，不"假装确定"。
- 出站模型调用经 **模型网关**（LiteLLM），不直连供应商。

### 5.3 gRPC IDL 概要（requirement_agent.proto）

```protobuf
service RequirementAgent {
  rpc ClarifyRequirement(ClarifyReq) returns (ClarifyResp);
  rpc GenerateSpec(SpecReq) returns (SpecResp);
  rpc DedupRequirement(DedupReq) returns (DedupResp);
  rpc EstimateEffort(EstimateReq) returns (EstimateResp);
}
// 每个请求必带 project_space_id + trace_id；流式接口用于多轮对话 (Clarify 用 server-streaming)
```

### 5.4 透明度要求

每次 Agent 产出必须随附：依据了哪些规则 ID、参考了哪些历史需求 ID、用了哪个模型版本（`model_version`）、置信度——全部落库可追溯。

## 6. 数据模型（Go struct / PostgreSQL）

> schema: `requirement`。所有表带 `project_space_id`、审计字段。Go 侧用 sqlx，struct tag 做 JSON 序列化与校验（`go-playground/validator`）。

### 6.1 Requirement（需求主表）

```go
// Requirement 需求主记录
type Requirement struct {
    ID              string    `db:"id" json:"id"`                         // ULID
    ProjectSpaceID  string    `db:"project_space_id" json:"project_space_id" validate:"required"`
    Title           string    `db:"title" json:"title" validate:"required,min=2,max=120"`
    BusinessGoal    string    `db:"business_goal" json:"business_goal"`   // 业务目标(人话)
    Status          string    `db:"status" json:"status" validate:"oneof=draft clarifying deduping spec_ready g1_pending confirmed in_dev done closed"`
    Priority        string    `db:"priority" json:"priority" validate:"oneof=must should could wont"`
    PriorityScore   float64   `db:"priority_score" json:"priority_score"` // 加权评分(看板排序)
    RequesterID     string    `db:"requester_id" json:"requester_id" validate:"required"`
    OwnerID         string    `db:"owner_id" json:"owner_id"`             // 产品经理/责任人
    DuplicateOfID   *string   `db:"duplicate_of_id" json:"duplicate_of_id,omitempty"` // 查重合并指向
    EmbeddingID     string    `db:"embedding_id" json:"embedding_id"`     // 向量索引引用(pgvector)
    Tags            []string  `db:"tags" json:"tags"`                      // pq.StringArray
    CreatedAt       time.Time `db:"created_at" json:"created_at"`
    UpdatedAt       time.Time `db:"updated_at" json:"updated_at"`
    CreatedBy       string    `db:"created_by" json:"created_by"`
    UpdatedBy       string    `db:"updated_by" json:"updated_by"`
}
```

### 6.3 RequirementSpec（需求规格）

```go
// RequirementSpec AI生成的结构化需求规格
type RequirementSpec struct {

    ID      string `json:"id"`

type AcceptCriterion struct {
    ID     string `json:"id"`

    When   string `json:"when"`

}
```

## 7. API 设计（Gin REST 端点）

> 路由前缀 `/api/v1`。所有端点经 API 网关鉴权 + 项目空间校验。错误统一 `{code, message, trace_id}`。

### 7.1 需求采集与会话

| 方法 | 路径 | 说明 | 角色 |
|---|---|---|---|
| POST | `/projects/{psid}/requirements` | 创建需求(进入 draft) | 业务需求方 |
| POST | `/projects/{psid}/requirements/{rid}/clarify` | 发起/继续一轮澄清(流式 SSE) | 业务需求方 |
| GET  | `/projects/{psid}/requirements/{rid}/clarifications` | 查看澄清历史 | 业务/产品 |
| PATCH| `/projects/{psid}/requirements/{rid}` | 编辑标题/目标/优先级 | 业务/产品 |

### 7.2 规格生成与查重

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/projects/{psid}/requirements/{rid}/spec:generate` | 触发规格 Agent 生成(异步任务) |
| GET  | `/projects/{psid}/requirements/{rid}/spec` | 获取当前规格(含版本) |
| GET  | `/projects/{psid}/requirements/{rid}/spec/versions` | 规格版本历史 |
| POST | `/projects/{psid}/requirements/{rid}/dedup` | 触发查重 Agent |
| GET  | `/projects/{psid}/requirements/{rid}/dedup-results` | 查重命中列表 |
| POST | `/projects/{psid}/requirements/{rid}/dedup-results/{did}:decide` | 业务确认复用/拒绝 |

### 7.3 预估与看板

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/projects/{psid}/requirements/{rid}/estimate` | 触发预估 Agent |

## 8. 规则与约束

> 本板块**读**规则治理中心（板块4）的规则资产，不自己维护规则。

### 8.1 知识资产消费清单

| 资产类型 | 内容 | 用途 | 强制度 |
|---|---|---|---|
| Rule | 需求模板标准 | 规格生成必套模板 | 🟥 强制 |
| Rule | 业务术语表 | 对话术语对齐，防歧义 | 🟥 强制 |
| Rule | 领域本体 | 实体/关系定义，规格数据对象校验 | 🟧 应遵循 |
| Rule | 合规红线（数据分级/隐私） | 规格预检，命中即提示/阻断 | 🟥 强制 |
| Knowledge | 历史需求库 | 查重向量检索 | 🟧 应遵循 |
| Process | 需求变更流程 | 变更走影响分析+轻量G1 | 🟥 强制 |

### 8.2 规则校验点

- 规格生成后调**规则引擎**（Go 侧 `rule` 模块）做：模板符合度、术语一致性、合规红线命中、必填字段。
- 命中阻断型规则 → 禁止提交 G1，返回违规项与建议。
- 命中提示型规则 → 标注但允许继续。

## 9. 人机闸门

### 9.1 🚪 G1 需求确认（L0 纯人工决策）

| 项 | 内容 |
|---|---|
| 触发 | 规格生成 + 查重 + 预估完成，业务提交 G1 |
| 决策者 | 业务需求方（拍板范围/优先级/预算）+ 规则架构师（合规） |
| AI 干的 | 起草 G1 审批单：范围摘要、优先级建议、预算估算、风险提示 |
| 人干的 | 确认或调整范围、定优先级、批预算、过合规 |
| 产出 | `confirmed` 状态需求 + 确认记录进审计 + 发 `requirement.confirmed` 事件 |
| 拒绝路径 | 驳回回澄清 / 直接关闭 / 标记暂缓 |
| 留痕 | 决策人、时间、依据、模型版本全入 audit |

> G1 是研发主链路第一道闸门。**未过 G1 的需求不得进入研发工作台**——这是硬约束，由 dev 模块订阅 `requirement.confirmed` 事件实现。

## 12. 落地优先级

| 优先级 | 范围 | 阶段 | 说明 |
|---|---|---|---|
| 🟥 P0 | 对话式采集 + 规格生成 + G1 闸门 + 需求池 | 阶段一 | 试点首选，最小可用闭环 |
| 🟥 P0 | 进度透视（接研发事件） | 阶段一 | 业务侧必须能看进度 |
| 🟧 P1 | 查重与关联（pgvector） | 阶段一末 | 历史数据少时效果有限，先埋点 |
| 🟧 P1 | 工期与算力预估 | 阶段一末 | 依赖历史样本积累 |
| 🟩 P2 | 语音/文档抽取/线框原型增强 | 阶段二 | 体验增强 |
| 🟩 P2 | 需求变更影响分析 | 阶段二 | 与研发工作台深度联动 |

### 关键里程碑

1. **MVP**：对话提需求 → AI 生成规格 → G1 确认 → 进入研发工作台（端到端跑通一个试点项目）。
2. **闭环**：研发/测试/上线事件回传，业务看到全程进度卡片。
3. **增强**：查重 + 预估上线，沉淀澄清经验为知识资产。

> 与研发工作台（板块2）的衔接是关键：本板块产出"已确认的需求规格"，板块2 消费它启动架构设计与任务拆解。规格的版本号是两者的握手契约。
