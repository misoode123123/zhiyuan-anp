# 板块06 · 发布中心（详细设计）

> **文档定位**：定义平台发布中心的完整设计——从"人工熬夜变更"转向"AI 准备 + 人审批 + 自动灰度 + 可回滚"，变更安全第一。覆盖变更评审、环境与配置管理、灰度/金丝雀发布、一键回滚、发布报告归档。
>
> **技术基线**：业务后端 **Go**（Gin + sqlx + zap）；变更评估 / 发布执行 / 守护 Agent 在 **Python** 侧（LangGraph + 工具链），Go 经 **gRPC** 调用。
>
> **配套**：软件开发架构设计（§6 部署架构、§7 集成架构）、开发标准与规范（§9 发布与变更规范）、板块02-研发、板块03-测试与质量中心、板块04-规则治理中心、板块05-安全与合规中心、板块07-运维中心。
>
> **核心原则**：🟥 所有生产发布必过 🚪G5 上线审批（CAB）；对外/资金操作升级 🚪L3 双确认；每次发布有变更单+回滚预案+检查清单。

---

## 1. 定位与价值

### 1.1 问题

传统发布：人工熬夜变更、变更单靠手填、影响面靠猜、灰度靠运气、回滚靠祈祷、出事找不到根因。结果：发布成功率低、变更事故频发、MTTR 长。

### 1.2 定位

发布中心将上线从**高风险人工操作**转变为**AI 准备 + 人审批 + 自动灰度 + 可回滚**的标准化流程：

| 转变 | 从 | 到 |
|---|---|---|
| 变更评审 | 人填变更单 | AI 生成（影响面/回滚/检查清单）+ 人确认 |
| 灰度策略 | 全量或手动 | 自动分流 + 指标守护 + 自动暂停 |
| 回滚 | 手动操作靠记忆 | 一键回滚 + 回滚预案预生成 |
| 审批 | 口头/邮件 | CAB 闸门 + L3 双确认 + 决策留痕 |
| 发布报告 | 事后补写 | 自动生成 + 归档可审计 |

### 1.3 价值

- **安全第一**：AI 预评估风险 + 人审批 + 自动守护，三重保障。
- **降低 MTTR**：指标异常自动暂停/回滚，不用等人发现。
- **标准化**：每次发布都有变更单+回滚预案+检查清单，无例外。
- **可审计**：发布全链路留痕，合规审计有据可查。

## 5. AI Agent 设计

> 所有 Agent 在 **Python 侧**（LangGraph 编排），Go 后端经 **gRPC** 调用。Agent 须遵守五项约束。

### 5.1 变更评估 Agent（ChangeAssessmentAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `ChangeAssessmentService.Assess(req) → resp` |
| **输入** | PR列表/commits、变更文件、版本号、前次发布记录、服务拓扑 |
| **输出** | `ChangeOrder`（impact_analysis / rollback_plan / checklist / risk_level / risk_reasons） |
| **工具** | 代码diff分析器、服务拓扑图查询、依赖分析器、历史发布库（RAG）、模型推理 |
| **编排图** | `StateGraph`: 收集变更 → 影响面分析（文件→模块→服务→依赖链） → 回滚预案生成 → 检查清单 → 风险评级 → 输出 |
| **约束** | 变更单必须含回滚预案（无回滚方案不得发布）；每项检查标注来源规则 |
| **自治等级** | L1：变更单为草案，人确认后方可提交审批 |

### 5.2 发布执行 Agent（ReleaseExecutionAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `ReleaseExecutionService.Execute(req) → stream resp` |
| **输入** | ChangeOrder（已审批）、环境信息、灰度策略、镜像/制品引用 |
| **输出** | 每批次执行状态（deployed / paused / rolled_back + 指标快照） |
| **工具** | K8s/Compose 部署控制器、流量路由器（Istio/Nginx）、配置注入器、健康检查器 |
| **编排图** | `StateGraph`: 初始化 → Batch循环(部署→路由→等待→指标检查→决策) → 完成/回滚 |
| **自治等级** | L2：审批后自动执行灰度批次，异常自动暂停 |

### 5.3 守护 Agent（GuardianAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `GuardianService.Watch(req) → stream events` |
| **输入** | 发布ID、目标指标基线、阈值规则、Prometheus 查询 |
| **输出** | `GuardianEvent`（normal / warning / critical / auto_pause / auto_rollback） |
| **工具** | Prometheus 查询器、日志分析器（Loki）、异常检测模型、告警路由 |
| **编排图** | `StateGraph`: 采集指标 → 基线对比 → 异常检测 → 分级告警 → 自动决策(暂停/回滚) |
| **核心能力** | 实时指标对比（error_rate/latency/资源）；多维度异常检测（突增/趋势/对比基线）；自动暂停+回滚决策 |
| **自治等级** | L3：自动暂停+自动回滚（人监督，事后确认） |

### 5.4 gRPC 接口定义（IDL 摘要）

```protobuf
// agent_runtime/proto/release.proto

service ChangeAssessmentService {
  rpc Assess(ChangeAssessmentRequest) returns (ChangeAssessmentResponse);
}

service ReleaseExecutionService {
  rpc Execute(ReleaseExecutionRequest) returns (stream ReleaseEvent);
}

service GuardianService {
  rpc Watch(GuardianWatchRequest) returns (stream GuardianEvent);
}

message ChangeAssessmentRequest {
  string project_space_id = 1;
  repeated string pr_ids      = 2;
  repeated string commit_shas = 3;
  string target_version       = 4;
  string target_environment   = 5;
}

message ChangeAssessmentResponse {
  string  agent_run_id    = 1;
  string  change_order_id = 2;
  string  risk_level      = 3;   // low / medium / high / critical
  ImpactAnalysis impact   = 4;
  RollbackPlan rollback   = 5;
  repeated CheckItem checklist = 6;
  repeated string risk_reasons = 7;
  repeated string rules_applied = 8;
}

message ImpactAnalysis {
  repeated string affected_services  = 1;
  repeated string affected_modules   = 2;
  repeated string changed_apis       = 3;
  repeated string downstream_deps    = 4;  // 下游依赖服务
  bool   has_db_migration            = 5;
  bool   has_breaking_change         = 6;
  bool   affects_external_users      = 7;
  bool   affects_fund_operations     = 8;
}

message RollbackPlan {
  repeated RollbackStep steps = 1;
  repeated string verification_checks = 2;  // 回滚后验证点
  bool   data_rollback_required = 3;
  string data_rollback_strategy = 4;        // migration_down / snapshot_restore / compensating_tx
  int32  estimated_rollback_seconds = 5;
}

message RollbackStep {
  int32  order       = 1;
  string action      = 2;   // deploy_previous / route_traffic / run_migration_down / flush_cache
  string target      = 3;   // 服务/组件
```

```go
func RegisterRoutes(rg *gin.RouterGroup, h *Handler) {
    // 变更单
```

## 8. 规则与约束

> 规则来源：板块04-规则治理中心。本中心是规则**消费者**。

### 8.1 规则消费清单

| 规则ID | 类型 | 内容 | 强制度 |
|---|---|---|---|
| RULE-REL-001 | Rule（强制） | 所有生产发布必过 🚪G5 CAB 审批 | 🟥 强制 |
| RULE-REL-002 | Rule（强制） | 对外/资金操作发布升级 🚪L3 双确认 | 🟥 强制 |
| RULE-REL-003 | Rule（强制） | 每次发布必须有回滚预案（无回滚不发布） | 🟥 强制 |
| RULE-REL-004 | Rule（强制） | 质量签章（板块03）+ 安全签章（板块05）必过 | 🟥 强制 |
| RULE-REL-005 | Rule（强制） | 生产发布必须在变更窗口内（规则治理中心下发） | 🟥 强制 |
| RULE-REL-006 | Rule（强制） | 金丝雀每批次必须有指标守护+自动暂停 | 🟥 强制 |
| RULE-REL-007 | Rule（强制） | 数据库 schema 变更必须先做 down migration | 🟥 强制 |
| RULE-REL-008 | Process（应遵循） | 变更单由 AI 生成但必须经人确认 | 🟧 应遵循 |
| RULE-REL-009 | Process（应遵循） | 发布期间锁定环境（防并发变更） | 🟧 应遵循 |
| RULE-REL-010 | Knowledge（参考） | 变更风险评级标准（范围×影响×数据×对外） | 📘 参考 |

### 8.2 发布门判定逻辑

```
  发布门(ReleaseGate) = AND(
      质量签章通过 (板块03 QualityReport.gate_passed),
      安全签章通过 (板块05 SecurityGate.gate_passed),
      配置检查通过 (新增配置项均已赋值),
      依赖检查通过 (下游依赖兼容),
      在变更窗口内 (板块04规则),
      环境未锁定/已为本次发布锁定,
      回滚预案就绪 (RollbackPlan.status=prepared),
      G5 CAB 审批通过,
      (高风险 → L3 双确认)
  )

  任一条件不满足 → 🟥 gate_passed = false → 阻断发布执行
```

### 8.3 风险评级标准

```
  风险分 = 变更范围(行数×文件数) × 影响面(服务×API) × 数据变更 × 对外影响

  ┌──────────┬────────────────────────────┬────────┐
  │ 风险等级 │ 条件                        │ 审批要求│
  ├──────────┼────────────────────────────┼────────┤
  │ 低       │ 内部服务+无DB变更+无对外    │ G5单人 │
  │ 中       │ 多服务+无DB变更+无对外      │ G5单人 │
  │ 高       │ 有DB变更 OR 多服务+对外影响 │ CAB    │
  │ 极高     │ 对外+资金+DB变更+破坏性     │ CAB+L3 │
  └──────────┴────────────────────────────┴────────┘
```

## 9. 人机闸门

### 9.1 本中心涉及的闸门

| 闸门 | 触发点 | 人/AI 分工 | 阻断方式 |
|---|---|---|---|
| 🚪G5 上线审批 | 所有生产发布执行前 | AI 生成变更单+前置检查→CAB 审批 | 发布门 gate_passed=false 阻断 |
| 🚪G5 CAB 审批 | 高风险发布 | CAB 成员审批 | 审批不通过=不可发布 |
| 🚪L3 双确认 | 对外/资金操作发布 | 双签确认 | 需两个独立审批人 |
| 守护闸门 | 灰度执行中 | AI 守护自动暂停/回滚 | 指标异常自动阻断 |

### 9.2 闸门决策矩阵

| 场景 | 决策者 | AI 角色 | 升级条件 |
|---|---|---|---|
| 变更单确认 | 人（发布经理） | AI 生成草案 | 高风险→CAB |
| G5 审批 | 人（CAB） | AI 预检查+风险评级 | 极高→L3双确认 |
| L3 双确认 | 人（双签） | AI 标注高风险原因 | 任一拒绝=不可发布 |
| 灰度执行 | AI 自动执行 | 部署+守护 | 人审批后自动执行 |
| 自动暂停 | 🟥 AI 自动 | 守护 Agent | 人介入判定 |
| 自动回滚 | 🟥 AI 自动（L3） | 守护 Agent | 事后人确认+复盘 |
| 手动回滚 | 人（SRE） | AI 提供预案 | 回滚后验证 |
| 发布完成 | AI 自动归档 | 生成报告 | →板块07 运维 |

### 9.3 审批链示例（极高变更）

```
  变更单确认(发布经理)
        │
        ▼
  前置检查(AI 自动)
        │
        ▼
  G5 CAB 审批(CAB成员 #1 审批)
        │
        ▼
  L3 双确认
   ├── 审批人 #1 (发布经理) ── approve
   └── 审批人 #2 (技术总监) ── approve
        │
        ▼
  发布执行(AI 自动灰度)
```

## 11. KPI

| KPI | 定义 | 目标 | 说明 |
|---|---|---|---|
| 发布成功率 | 成功发布/总发布 | ≥98% | 不含自动回滚 |
| 自动回滚率 | 自动回滚/总发布 | ≤5% | 太高=质量/评估有问题 |
| MTTR（变更故障） | 从异常到恢复 P50 | ≤5min | 自动回滚快速恢复 |
| 发布平均时长 | 从执行到完成 P50 | ≤30min | 含灰度观察时间 |
| 变更单采纳率 | AI变更单无需大改/总变更单 | ≥70% | AI 评估质量 |
| 回滚预案准确率 | 回滚预案可直接执行/实际回滚 | ≥95% | 预案有效性 |
| CAB 审批及时率 | 审批≤2h / 总审批 | ≥90% | 不阻塞发布节奏 |
| 灰度检测率 | 灰度阶段发现异常/总异常 | ≥90% | 在小流量阶段拦截 |
| 发布报告完成率 | 有归档报告 / 总发布 | 100% | 全量审计 |
| 变更事故率 | 引发事故的变更/总变更 | ≤2% | 越低越好 |

## 12. 落地优先级

### 12.1 阶段规划

| 阶段 | 范围 | 优先级 | 依赖 |
|---|---|---|---|
| **P1-阶段一（MVP）** | 变更单CRUD + G5审批 + 多环境管理 + 基础灰度（手动批次）+ 一键回滚 + 发布报告 | 🟥 必须 | 板块02/03/05、GitLab CI、K8s/Compose |
| **P1-阶段二** | AI 变更评估Agent + 自动金丝雀（指标守护+自动暂停）+ 守护Agent + L3双确认 | ⭐ 核心 | 板块04 规则、Python agent-runtime、板块07 指标 |
| **P2-阶段三** | 自动回滚（L3 守护自决）+ 配置漂移检测 + 蓝绿部署 + 数据回滚自动化 | 🟧 增强 | 板块07 运维 |
| **P3-阶段四** | 智能发布时间推荐 + 风险预测模型 + 自适应灰度（按指标自动调速） | 📘 演进 | 板块08 算力 |

### 12.2 MVP 必须项（阶段一验收标准）

- [ ] ChangeOrder CRUD + 人确认流程
- [ ] G5 CAB 审批闸门（🟥所有 prod 发布必过）
- [ ] Environment CRUD + 环境锁定
- [ ] 金丝雀手动分批执行（1%→10%→50%→100%）
- [ ] 一键回滚（版本切换 + 回滚验证）
- [ ] 前置检查：质量签章（板块03）+ 安全签章（板块05）接入
- [ ] 发布报告生成 + MinIO 归档
- [ ] Go struct 全部实现 + sqlx 持久化

### 12.3 技术债红线

- **G5 审批阶段一必须就位**（RULE-REL-001 🟥强制），无审批不得发布。
- **回滚预案阶段一必须生成**（RULE-REL-003 🟥强制），无回滚预案不得发布。
- 自动回滚可在阶段二上线，但**回滚命令框架**阶段一必须预留（手动回滚必须可用）。
- L3 双确认**阶段一必须有框架**（至少双人审批），自动判定 `affects_funds` 可在阶段二完善。
- 守护 Agent 的指标查询能力**阶段一必须有基础版**（至少查 error_rate），自动暂停可在阶段二。
