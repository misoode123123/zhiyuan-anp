# 智源 ANP 平台 · 功能增强 · 对话式需求梳理（详细设计）

> **文档定位**：需求工作台增强——把「一次性生成规格」升级为「AI agent 对话式梳理需求」。混合对话模式（先自由描述、agent 识别缺失要素针对性追问），要素齐全后 agent 提议、用户确认生成规格，复用现有 `requirement` 表接编码闭环。完整记录对话过程。支持多模态（图片），架构预留语音。
> **技术基线（当前实现层）**：业务后端 Go（Gin + sqlx + SQLite）；AI 运行时 Python FastAPI（`/v1/chat`，已支持多轮 messages + 多模态 content）；前端 Next.js + TS。
> **多租户**：所有数据带 `project_space_id`。

---

## 1. 定位与价值

把需求采集从「填一段描述→出规格」变成「和 AI 聊天→AI 引导补全→确认出规格」。降低写需求门槛，提升规格完整度；对话过程留痕，可追溯需求是怎么得出的。

| 价值点 | 说明 |
|---|---|
| 引导补全 | agent 识别缺失要素（角色/目标/场景/约束/验收）并追问，规格更完整 |
| 门槛低 | 用户自然语言聊天，不必会写结构化需求 |
| 可追溯 | 对话历史落库，需求来源可回溯 |

---

## 2. 与现有需求工作台的关系

- **现有**：`POST /project-spaces/:id/requirements` 一次性生成规格（`requirement.generateSpec`）→ 入 `requirement` 表。
- **本设计**：新增「对话」入口，对话确认后**仍生成 `requirement` 行**（复用现有表与后续编码闭环）。对话本身存新表 `conversation`/`message`。
- 两种入口并存：快速场景仍可用原一次性生成；复杂/不清晰场景走对话梳理。

---

## 3. 数据模型（新两表）

```sql
CREATE TABLE IF NOT EXISTS conversation (
  id               TEXT PRIMARY KEY,
  project_space_id TEXT NOT NULL REFERENCES project_space(id),
  status           TEXT NOT NULL DEFAULT 'active',  -- active/submitted/archived
  title            TEXT,                            -- 提交时填（取规格标题）
  requirement_id   TEXT,                            -- 提交后回填
  created_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_conversation_ps ON conversation(project_space_id);

CREATE TABLE IF NOT EXISTS message (
  id             TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL REFERENCES conversation(id),
  role           TEXT NOT NULL,                      -- user/assistant
  content        TEXT NOT NULL,                      -- JSON: {"text":...,"images":[...]}
  media_kind     TEXT NOT NULL DEFAULT 'text',       -- text/image/audio(预留)
  created_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_message_conv ON message(conversation_id);
```

**字段说明**：
- `message.content` 统一存 JSON 字符串：`{"text":"...","images":["data:..."]}`。`images` 可空。多模态时 Go 拼 `/v1/chat` 的 content 数组（复用 `requirement.generateSpec` 既有做法）。
- `media_kind`：本轮 `text`/`image`；`audio` 预留（语音落地时用）。
- `conversation.status`：`active`（对话中）→ `submitted`（已生成规格入库）→ 可 `archived`。
- `conversation.requirement_id`：提交后回填，关联生成的 `requirement.id`。

## 4. 对话流程与编排（Go 后端编排，复用 `/v1/chat`）

`/v1/chat` 已是无状态多轮（每次传完整 messages）。Go 侧管会话持久化与历史拼接。

```
用户发消息
  → Go 存 user message
  → 取该 conversation 全部历史 messages
  → 构造 messages = [system(需求梳理员), ...历史, 新 user(多模态)]
  → POST agent-runtime /v1/chat
  → 存 assistant message
  → 返回 assistant 回复（本轮一次性返回）

要素齐全（agent 自然语言提议 / 用户主动点「生成规格」）
  → POST /conversations/:cid/generate-spec
  → Go 取全历史 → 构造总结指令 → /v1/chat → 解析 {title,user_story,acceptance_criteria}
  → 返回草稿（不入库）

用户确认
  → POST /conversations/:cid/commit
  → requirement.Repository.Create 写 requirement 行
  → conversation.status=submitted, requirement_id 回填, title 填
  → 接现有编码闭环（dispatch-code → 审批 → 发布）
```

### agent system prompt（混合模式需求梳理员）

```
你是资深需求分析师，与用户对话梳理需求。先听用户自由描述，
识别缺失要素（用户角色 / 目标价值 / 使用场景 / 约束 / 验收标准）并针对性追问。
要素齐全时，主动询问「是否要我生成结构化需求规格？」。
回复简洁，一次只问 1-2 个关键问题。
```

要素齐全的判断完全由 LLM 在对话中自然语言完成；前端「生成规格」按钮始终可用（用户也可主动触发），**不依赖 agent 输出结构化信号**，实现简单稳健。

## 5. 多模态与语音预留

- **图片（本轮实现）**：user message 可带 images（data URL）。Go 拼多模态 content 数组 `[{type:text},{type:image_url}]` 走 GLM-4V（复用 `requirement.generateSpec` 的 content 构造与 model 选择）。
- **语音（本轮仅预留，不实现 ASR）**：
  - `message.media_kind` 已含 `audio` 取值；`content` JSON 预留 `audio_url` 字段。
  - 未来加 `POST /conversations/:cid/voice`（上传音频 → ASR 转写 → 作为 text message 入对话 → 正常走 LLM）。
  - 本轮不引入 ASR 依赖，架构不阻塞后续。

## 7. 后端 API

路由前缀 `/api/v1`。

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/project-spaces/:id/conversations` | 建会话（返回 cid） |
| GET | `/project-spaces/:id/conversations` | 会话列表 |
| GET | `/conversations/:cid` | 会话详情 + 全部 messages 历史 |
| POST | `/conversations/:cid/messages` | 发消息 `{text, images?}` → 返回 assistant 回复 |
| POST | `/conversations/:cid/generate-spec` | 生成规格草稿 `{title,user_story,acceptance_criteria}`（不入库） |
| POST | `/conversations/:cid/commit` | 确认入库 → 写 `requirement` + 会话标记 submitted |

- 发消息与生成草稿均经 Go 调 agent-runtime `/v1/chat`（Go 持 `agentRuntimeURL`，复用 `requirement` 既有 HTTP 调用模式）。
- `commit` 复用 `requirement.Repository.Create`（直接传已生成 spec 字段，不走 `generateSpec`）。
- 多租户：会话属项目空间（路径 `:id`）；消息操作经 `conversation.project_space_id` 校验归属（不信任请求体租户键）。

- **聊天界面**：消息流（user/assistant 气泡）+ 输入框 + 图片上传（复用现有 data URL 做法）+ 「生成规格」按钮。

## 10. 验收标准

1. 建会话 → 发多条消息 → agent 正常回复（混合模式：能追问缺失要素）。
2. 发消息带图片 → agent 能识别图片内容回复（多模态 GLM-4V）。
3. 点「生成规格」→ 返回结构化草稿（title/user_story/AC）。
4. 「确认入库」→ `requirement` 表新增一行 + `conversation.status=submitted` + `requirement_id` 回填；需求列表出现新需求，可派发编码。
5. `GET /conversations/:cid` 返回完整消息历史（user+assistant 全记录）。
6. 语音字段预留：`message.media_kind` 支持 audio 取值（本轮无 ASR 调用）。
