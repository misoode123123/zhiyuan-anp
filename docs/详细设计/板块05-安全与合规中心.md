# 板块05 · 安全与合规中心（详细设计）

> **文档定位**：定义平台安全与合规中心的完整设计——守住"不泄密 / 不违规 / 不带病上线"，安全检查嵌进流水线而非事后审计。覆盖代码安全扫描、数据安全管控、合规策略下发、AI 行为审计、模型与提示安全。
>
> **技术基线**：业务后端 **Go**（Gin + sqlx + zap）；安全扫描 Agent / 脱敏 Agent / 合规审查 Agent 在 **Python** 侧（LangGraph + 工具链），Go 经 **gRPC** 调用。
>
> **配套**：软件开发架构设计（§8 安全架构）、开发标准与规范（§8 安全编码）、板块02-研发、板块04-规则治理中心、板块06-发布中心。
>
> **核心原则**：🟥 安全扫描**嵌入流水线**（PR 阶段即扫描），高危发现**自动阻断**；合规策略由法务转规则入治理中心，平台消费执行。

---

## 1. 定位与价值

### 1.1 问题

传统安全：开发完→安全团队事后扫→发现高危→已上线→紧急修复。合规：人工对照检查清单→遗漏多→审计被动。数据安全：测试拷生产、AI对话泄密、权限过宽。

### 1.2 定位

安全与合规中心将安全从**事后检查**前移为**流水线内嵌的持续扫描**：

| 转变 | 从 | 到 |
|---|---|---|
| 安全介入时机 | 上线前 | PR 阶段（编码即扫描） |
| 扫描方式 | 手动触发 | CI 自动触发 + Agent 交叉验证 |
| 合规检查 | 人工对照清单 | 法规→规则→自动校验 |
| 数据保护 | 依赖人工自觉 | 数据分级自动识别+脱敏+最小权限 |
| AI 安全 | 无管控 | 防注入/防越狱/输出过滤/行为审计 |
| 审计 | 事后补记录 | 全链路自动留痕 |

### 1.3 价值

- **左移**：在 PR 阶段阻断高危代码，修复成本降低 10x+。
- **合规内嵌**：法规变更→规则治理中心→自动入流水线，不再依赖人记忆。
- **数据不泄密**：测试数据自动脱敏 + AI 对话出站过滤。
- **可审计**：每次扫描/每次AI产出都有留痕，合规审计秒级出证。

## 5. AI Agent 设计

> 所有 Agent 在 **Python 侧**（LangGraph 编排），Go 后端经 **gRPC** 调用。Agent 须遵守五项约束。

### 5.1 安全扫描 Agent（SecurityScanAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `SecurityScanService.Scan(req) → resp` |
| **输入** | 仓库路径、commit_sha、PR diff、项目空间配置 |
| **输出** | `[]ScanFinding`（severity / category / file / line / rule_id / remediation / confidence） |
| **工具链** | SAST(Semgrep/CodeQL)、SCA(Trivy/Grype)、密钥检测(Gitleaks/TruffleHog)、License(ScanCode) |
| **编排图** | `StateGraph`: 并行调用多工具 → 聚合结果 → 去重(同根因) → 交叉验证(多工具命中=高置信) → 误报研判 → 输出 |
| **核心能力** | 多工具交叉验证降低误报率；规则引擎校验（与板块04规则匹配）；AI研判边界case |
| **约束** | 高危发现不得降级；每条 Finding 标注 rule_id（可追溯） |
| **自治等级** | L2：CI 自动扫描执行；阻断决策由 Go 后端闸门执行 |


### 5.2 脱敏 Agent（DataMaskingAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `DataMaskingService.Mask(req) → resp` |
| **输入** | 原始数据（或数据描述/schema）、请求方角色、用途 |
| **输出** | 脱敏后数据 + 脱敏审计记录 |
| **工具** | 敏感数据识别器（正则+NLP模型）、脱敏策略库、合成数据生成器 |
| **编排图** | `StateGraph`: 识别敏感字段 → 判定分级 → 匹配脱敏策略 → 执行脱敏 → 审计记录 |
| **自治等级** | L2：按规则自动脱敏，restricted 级需人授权 |

### 5.3 合规审查 Agent（ComplianceReviewAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `ComplianceReviewService.Review(req) → resp` |
| **输入** | PR diff、合规规则集（从板块04获取）、项目空间合规上下文 |
| **输出** | `ComplianceResult`（passed / violations[] / risk_level / recommendation） |
| **工具** | 规则匹配引擎、法规知识RAG（GDPR/等保/数据安全法/行业规范）、模型推理 |
| **编排图** | `StateGraph`: 加载合规规则 → 逐条匹配 → 法规知识辅助研判 → 风险评级 → 输出 |
| **自治等级** | L1：审查结果为建议，合规违规阻断由 Go 后端闸门执行 |

```protobuf
service SecurityScanService {
  rpc Scan(SecurityScanRequest) returns (SecurityScanResponse);

message SecurityScanRequest {
  string project_space_id = 1;

  int32   critical_count   = 4;

  repeated string rules_applied = 7;

  string  category     = 2;   // sast / sca / secret / license / iac

  string  description  = 6;
  string  file_path    = 7;

  string  code_snippet = 9;
```

## 8. 规则与约束

> 规则来源：板块04-规则治理中心。本中心同时是规则**消费者**（扫描规则）和**生产者**（合规规则来源）。

| 规则ID | 类型 | 内容 | 强制度 |

| RULE-SEC-002 | Rule（强制） | 高危依赖（CVSS≥7.0）阻断合入 | 🟥 强制（CI 阻断） |
| RULE-SEC-003 | Rule（强制） | SQL 必须参数化（禁拼接） | 🟥 强制（SAST 阻断） |
| RULE-SEC-004 | Rule（强制） | GPL/AGPL 传染性许可证阻断 | 🟥 强制（CI 阻断） |
| RULE-SEC-005 | Rule（强制） | 敏感数据脱敏后方可用于测试/AI | 🟥 强制 |
| RULE-SEC-006 | Rule（强制） | AI 对话出站须过滤敏感内容 | 🟥 强制 |
| RULE-SEC-007 | Process（应遵循） | Dockerfile 基线（非root/最小镜像） | 🟧 应遵循 |
| RULE-SEC-008 | Process（应遵循） | 最小权限审计每季度执行 | 🟧 应遵循 |

| RULE-SEC-010 | Rule（强制） | 提示注入/越狱检测必须启用 | 🟥 强制 |

### 8.2 合规规则下发流程

```
  法务/合规要求（法规/行业标准/内规）
         │
         ▼
  合规专员 + 安全工程师
  结构化为 RaC 规则
         │
         ▼
  入板块04 规则治理中心
  (草稿→评审→发布)
         │
         ▼
  安全中心订阅合规规则
  自动并入扫描管线
         │
         ▼
  每个 PR / 每次发布自动校验
  违规→阻断+报告
```

### 8.3 安全门判定逻辑

```
  安全门(SecurityGate) = AND(
      无 critical 级漏洞未修复,
      无 high 级漏洞未修复 (或已获 accept-risk 审批),
      无密钥泄露未修复,
      无许可证不兼容,
      合规检查通过,
      提示安全检测已启用
  )

  任一条件不满足 → 🟥 gate_passed = false → 阻断PR/发布
```

### 9.1 本中心涉及的闸门

| 闸门 | 触发点 | 人/AI 分工 | 阻断方式 |

### 9.2 闸门决策矩阵

| 场景 | 决策者 | AI 角色 | 升级条件 |
|---|---|---|---|
| 密钥泄露 | 🟥 AI 自动阻断 | 扫描+判定 | 不可升级（硬阻断） |
| 高危依赖 | 🟥 AI 自动阻断 | SCA 检测 | 修复或获 accept-risk 审批 |
| 合规违规 | 人（合规专员+安全负责人） | AI 审查建议 | 高风险→升级法务 |
| 误报研判 | 人（安全工程师） | AI 标注置信度 | 低置信→安全负责人复核 |
| 数据分级 | 人（DPO 确认） | AI 识别建议 | restricted→DPO 强制审批 |
| 安全签章放行 | 人（安全负责人） | AI 汇总风险 | 有 critical→禁止放行 |
| AI 对话泄密 | 🟥 AI 自动拦截 | 出站过滤 | 记录审计+告警安全团队 |

| 高危漏洞修复时长 | critical 从发现到 fixed P50 | ≤4h | 高危快速闭环 |
| 密钥泄露拦截率 | 拦截次数/(拦截+漏到线上) | 100% | 零容忍 |
| 扫描误报率 | false_positive / 总发现 | ≤15% | 持续优化工具链+研判 |
| 合规检查覆盖率 | 有合规校验的PR / 总PR | 100% | 全覆盖 |
| 敏感数据识别覆盖率 | 已分级字段 / 实际敏感字段 | ≥90% | 持续发现新字段 |
| AI 对话泄密拦截率 | 拦截/(拦截+漏出) | 100% | 零容忍 |
| 安全签章及时率 | 签章生成时间 / 发布等待时间 | ≤5min | 不阻塞发布节奏 |
| 高危漏洞检出率 | 扫描发现 / (发现+线上漏出) | ≥95% | 尽可能在测试阶段发现 |
| License 合规率 | 兼容许可证依赖 / 总依赖 | 100% | 零违规 |

## 12. 落地优先级

### 12.1 阶段规划

| 阶段 | 范围 | 优先级 | 依赖 |
|---|---|---|---|
| **P1-阶段一（MVP）** | SAST + SCA + 密钥检测 + 安全门（🟥CI阻断）+ 基础数据分级 | 🟥 必须 | 板块02 研发、GitLab CI、板块04 规则 |
| **P1-阶段二** | 合规审查Agent + AI 对话防泄密 + 安全签章→板块06 + License扫描 | ⭐ 核心 | 板块04 规则、板块06 发布 |
| **P2-阶段三** | 脱敏Agent（自动识别+合成数据）+ 提示注入/越狱检测 + 最小权限审计 | 🟧 增强 | 板块03 测试、板块09 AI能力市场 |
| **P3-阶段四** | 红队自动化 + 安全态势感知 + 零信任服务网格 | 📘 演进 | 板块07 运维 |

### 12.2 MVP 必须项（阶段一验收标准）

- [ ] SAST(Semgrep) + SCA(Trivy/Grype) + 密钥检测(Gitleaks) 三工具链接入
- [ ] CI Webhook → 自动触发安全扫描
- [ ] 安全门判定（critical/high = 🟥阻断）
- [ ] 漏洞 CRUD + PR 评论
- [ ] 数据分级基础框架（public/internal/confidential/restricted）
- [ ] 基础合规规则集（等保基线 + 内规）
- [ ] Go struct 全部实现 + sqlx 持久化

### 12.3 技术债红线

- **基础安全策略阶段一必须就位**（RULE-SEC-001~003 🟥强制），不因 MVP 放松安全红线。
- 合规规则**最小集**（等保2.0基线）阶段一必须入板块04，后续逐步扩展（GDPR/PIPL/行业）。
- 密钥泄露检测**必须第一天就开**——一次泄露 = 一次事故。
- AI 对话防泄密可在阶段二上线，但**出站过滤框架**阶段一必须预留。
