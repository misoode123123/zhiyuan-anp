# 智源 ANP 平台 · 板块 7 · 运维中心（详细设计）

> **文档定位**：从"被动救火"到"AI 巡检 + 智能研判 + 自愈 + 人在高风险节点把关"的自治运维体系，目标自治等级 L2-L3。是 9 大板块运维能力的集中承载，为全平台提供 7×24 健康保障、智能告警研判与受控自愈。线上缺陷经此反哺需求/研发，形成闭环。
> **技术基线**：业务后端 Go（Gin + sqlx + zap）管理告警/故障/SOP/自愈执行记录与看板；巡检/根因分析/自愈 Agent 由 **Python 侧 LangGraph** 承载，Go 经 **gRPC** 调用。监控数据来自 Prometheus + Loki + Tempo；持久化 PostgreSQL（schema: `ops`）+ Redis（告警缓存/实时看板）。
> **配套**：软件开发架构设计、开发标准与规范、总体方案 V1.0。
> **多租户**：所有数据带 `project_space_id` 作为路由键。

## 1. 定位与价值

### 1.1 定位

运维中心是平台**稳定性的守门人**。传统运维靠人盯屏幕、手动排查、事后补救；运维中心把运维经验沉淀为 SOP，让 AI Agent 7×24 巡检、研判、执行自愈，人只在高风险节点把关。运维知识来自规则治理中心（板块4），规则即代码（RaC）。

```
监控源(Prom/Loki/K8s)        运维中心                      人工
    │ 告警/指标                │                             │
    │ ──────────────────────▶ │ AI巡检+研判+自愈            │
    │                         │ ──── 低风险自动执行 ──────▶ │ (知会)
    │                         │ ──── 中风险审批 ──────────▶ │ (审批)
    │                         │ ──── 🚪G6高风险双确认 ────▶ │ (双确认)
    │                         │                             │
    │                         │ ◀── 缺陷反哺 ───────────────│ 板块1/2/3
```

### 1.2 价值（可量化目标）

| 价值点 | 指标 | 18 个月目标 |
|---|---|---|
| 告警降噪 | 有效告警占比（去噪后） | ≥ 90% |
| 快速恢复 | 自愈覆盖故障 MTTR | ≤ 5 分钟 |
| 主动预防 | 巡检提前发现占比 | ≥ 40% |
| 根因定位 | 根因分析准确率 | ≥ 75% |
| 成本可控 | 资源利用率 | 提升 30% |
| 知识沉淀 | 故障产出 SOP 比例 | ≥ 60% |

### 1.3 自治等级映射

| 场景 | 自治等级 | 说明 |
|---|---|---|
| 告警去噪/关联分析 | L0-L1 | Agent 自动处理，知会即可 |
| 根因分析报告 | L1 | Agent 产出报告，知会运维 |
| 低风险自愈（重启 Pod/清缓存） | L1 | Agent 自动执行，知会通知 |
| 中风险自愈（扩容/切流/限流） | L2 | 审批后执行 |
| 🚪G6 高风险（删数据/资金/对外变更） | L3 | 强制双确认 |

## 3. 功能模块

### 3.1 AI 巡检引擎

- **定期健康检查**：按策略（5min/1h/每日）自动巡检 Go 后端、Python Agent 服务、PG、Redis、MinIO、K8s 集群，产出健康报告。
- **容量预测**：基于历史指标时序（CPU/内存/GPU/Token 消耗）预测趋势，提前预警瓶颈。
- **成本异常发现**：检测算力/模型消耗突增或异常趋势，联动板块8算力中心。
- **巡检报告**：自动生成结构化报告，按严重度分级推送（healthy/warning/critical）。

### 3.2 智能告警研判

- **告警去噪**：同类告警按 fingerprint 聚合去重，抑制告警风暴。
- **关联分析**：时间窗口内多告警关联（同一根因的多表象合并为一个 Incident）。
- **根因定位**：基于服务拓扑图 + 日志（Loki）+ 指标（Prometheus）+ 变更历史（CI/CD），AI 推断根因。
- **影响评估**：自动评估受影响的服务/用户/业务流程范围。

### 3.3 自愈预案库

- **预案管理**：常见故障（Pod 重启、缓存清理、扩容、切流、限流降级）的自动化 SOP。
- **风险分级**：低（重启/缓存清理）/ 中（扩容/切流/限流）/ 高（删数据/资金/对外变更）。
- **受控执行**：低风险 Agent 自动执行→知会；中风险→审批后执行；高风险→🚪G6 强制双确认。
- **可回滚**：每个自愈动作记录变更快照，支持一键回滚（rollback_cmd）。
- **预案来源**：来自规则治理中心（板块4）的 SOP，规则即代码。

### 3.4 容量与成本运营

- **容量看板**：实时资源利用率 + 趋势预测曲线。
- **成本看板**：按项目空间/部门/服务拆解（联动板块8算力中心数据）。
- **优化建议**：AI 识别闲置资源/过度配额/可降配实例。
- **预算预警**：接近阈值自动告警。

### 3.5 运维知识管理

- **SOP 沉淀**：Incident 闭环后可将处置过程沉淀为新 SOP。
- **复盘管理**：Incident 关闭自动生成复盘报告模板。
- **知识类型**：Rule（强制遵循）/ Process（应遵循）/ Knowledge（参考）。

## 4. 核心业务流程

### 4.1 主流程：告警研判与自愈

```
┌──────────────┐
│ 监控源触发告警 │ Prom/Loki/K8s → Webhook
└──────┬───────┘
       ▼
┌──────────────────┐
│ Go 后端接收告警   │ 创建 Alert 记录, 计算 fingerprint
└──────┬───────────┘
       │ gRPC 推送告警
       ▼
┌──────────────────────────────────────────┐
│ 根因分析 Agent (Python/LangGraph)         │
│   1. 去噪: fingerprint 聚合, 抑制重复       │
│   2. 关联: 时间窗口内多告警合并             │
│   3. 根因: 拓扑+日志+指标+变更历史推断       │
│   4. 影响评估 + 匹配 SOP 预案              │
└──────┬───────────────────────────────────┘
       │ 研判结果(根因+影响+建议+matched_sop_id)
       ▼
┌──────────────────┐
│ 创建 Incident     │ 关联告警, 填充根因/影响/建议
└──────┬───────────┘
       │ 匹配到 SOP?
       ├── 否 ──▶ 推送通知 → 人工介入
       │
       ├── 是 ──▶ 风险分级路由
       │            ┌─────────────────────────┐
       │            │ 低风险(L1)               │
       │            │  自愈Agent自动执行        │
       │            │  知会通知                 │
       │            ├─────────────────────────┤
       │            │ 中风险(L2)               │
       │            │  推送审批 → 运维负责人     │
       │            │  审批通过 → 执行          │
       │            ├─────────────────────────┤
       │            │ 🚪G6 高风险(L3)          │
       │            │  双确认(操作人+负责人)    │
       │            │  确认通过 → 执行          │
       │            └────────────┬────────────┘
       ▼                         │
┌──────────────────┐             │
│ 自愈 Agent 执行   │ ◀───────────┘
│  逐步执行SOP      │
│  每步记录+可回滚  │
└──────┬───────────┘
       │
       ├── 恢复 ✓ ──▶ 关闭 Incident → 生成复盘 → 沉淀 SOP
       │
       └── 未恢复 ✗ ──▶ 回滚 → 升级人工介入 → 分析缺陷
                                    │
                                    ▼
                         事件 → 板块2研发(缺陷修复)
```

## 5. AI Agent 设计

> 所有运维 Agent 运行在 **Python 侧**（agent-runtime 服务，LangGraph 编排），Go 后端经 gRPC 调用。Agent 产出遵守五约束（可校验/可追溯/可回滚/守边界/守权限）。

### 5.1 Agent 清单

| Agent | 职责 | 输入 | 输出 | 自治级 |
|---|---|---|---|---|
| 巡检 Agent (PatrolAgent) | 定期/按需健康检查、容量预测、成本扫描 | 巡检策略（范围/频率/检查项） | 巡检报告（健康状态+异常项+建议） | L1 |
| 根因分析 Agent (RootCauseAgent) | 告警去噪、关联分析、根因定位、影响评估 | 告警事件+拓扑+日志+指标 | Incident（根因+影响+建议+匹配SOP） | L1 |
| 自愈 Agent (HealingAgent) | 按 SOP 预案执行自愈动作 | Incident+匹配SOP+审批状态 | HealingRun（步骤+结果+回滚信息） | L1/L2/L3 |

### 5.2 根因分析 Agent 的 LangGraph 状态机（要点）

```
[receive_alerts] ─▶ [dedup_by_fingerprint] ─▶ [correlate_window]
                                                    │
                                          ┌─────────┴──────────┐
                                          ▼                    ▼
                                  [build_topology]      [query_logs_metrics]
                                          │                    │
                                          └────────┬───────────┘
                                                   ▼
                                          [infer_root_cause]
                                                   │
                                          ┌────────┴───────────┐
                                          ▼                    ▼
                                  [assess_impact]      [match_sop]
                                          │                    │
                                          └────────┬───────────┘
                                                   ▼
                                          [emit_incident]
```

- 终止条件：产出根因结论或达到最大推理轮次（防死循环）。
- 出站模型调用经 **模型网关**（LiteLLM），不直连供应商；根因分析用强推理模型。

### 5.3 gRPC IDL 概要（ops_agent.proto）

```protobuf
service OpsAgentService {
  // 巡检
  rpc RunInspection(InspectionRequest) returns (InspectionResponse);
  // 告警研判（去噪+关联+根因）
  rpc AnalyzeAlerts(AnalyzeAlertsRequest) returns (AnalyzeAlertsResponse);
  // 执行自愈预案（流式返回执行进度）
  rpc ExecuteHealing(HealingRequest) returns (stream HealingProgress);
}

message AnalyzeAlertsRequest {
  string project_space_id = 1;
  string trace_id = 2;
  repeated Alert alerts = 3;
  TopologyGraph topology = 4;   // 服务拓扑图
}

message AnalyzeAlertsResponse {
  string incident_id = 1;
  string root_cause = 2;
  string impact = 3;
  string recommendation = 4;
  string matched_sop_id = 5;
  repeated string correlated_alert_ids = 6;
  float confidence = 7;
  string model_version = 8;
}
```

### 5.4 透明度要求

每次 Agent 产出随附：依据了哪些 SOP/规则 ID、参考了哪些历史 Incident、用了哪个模型版本（`model_version`）、置信度——全部落库可追溯。

## 6. 数据模型（Go struct / PostgreSQL）

> schema: `ops`。所有表带 `project_space_id`、审计字段。Go 侧用 sqlx，struct tag 做 JSON 序列化与校验（`go-playground/validator`）。

### 6.1 Alert（告警事件）

### 6.4 SOP（自愈预案/运维标准操作流程）

```go
// SOP 自愈预案 / 运维标准操作流程
type SOP struct {

    RiskLevel        string            `db:"risk_level" json:"risk_level" validate:"oneof=low medium high"`
    RequiresApproval bool              `db:"requires_approval" json:"requires_approval"`
    AutoLevel        string            `db:"auto_level" json:"auto_level" validate:"oneof=L0 L1 L2 L3"`
    Version          int               `db:"version" json:"version"`
    Status           string            `db:"status" json:"status" validate:"oneof=draft active deprecated"`
    SourceRuleID     *string           `db:"source_rule_id" json:"source_rule_id,omitempty"` // 来自板块4
    CreatedBy        string            `db:"created_by" json:"created_by"`
    CreatedAt        time.Time         `db:"created_at" json:"created_at"`
    UpdatedAt        time.Time         `db:"updated_at" json:"updated_at"`
}

type TriggerCondition struct {
    AlertLabels map[string]string `json:"alert_labels,omitempty"`
    SeverityIn  []string          `json:"severity_in,omitempty"`
    Pattern     string            `json:"pattern,omitempty"`     // 告警标题匹配模式

type SOPStep struct {
    Order       int                    `json:"order"`
    Action      string                 `json:"action"`
    Target      string                 `json:"target"`
    Params      map[string]interface{} `json:"params"`
    RollbackCmd string                 `json:"rollback_cmd,omitempty"` // 回滚指令
    VerifyCmd   string                 `json:"verify_cmd,omitempty"`   // 验证指令(可校验)
}
```

```go
// internal/ops/api/router.go

func RegisterOpsRoutes(r *gin.Engine, h *OpsHandler, auth middleware.AuthMiddleware) {
    v1 := r.Group("/api/v1/projects/:psid/ops")
    v1.Use(auth.Require(), auth.RequireTenant())
    {
        // 告警
        v1.POST("/alerts", h.ReceiveAlert)               // Webhook 独立鉴权
        v1.GET("/alerts", h.ListAlerts)
        v1.GET("/alerts/:aid", h.GetAlert)
        v1.POST("/alerts/:aid:suppress", h.SuppressAlert)

        // 故障
        v1.GET("/incidents", h.ListIncidents)
        v1.GET("/incidents/:iid", h.GetIncident)
        v1.PUT("/incidents/:iid:assign", h.AssignIncident)
        v1.PUT("/incidents/:iid:close", h.CloseIncident)

        // 自愈
        v1.POST("/healing/runs", h.TriggerHealing)
        v1.GET("/healing/runs/:hid", h.GetHealingRun)
        v1.POST("/healing/runs/:hid:approve", h.ApproveHealing)
        v1.POST("/healing/runs/:hid:rollback", h.RollbackHealing)

        // SOP
        v1.GET("/sops", h.ListSOPs)
        v1.POST("/sops", h.CreateSOP)
        v1.PUT("/sops/:sid", h.UpdateSOP)
        v1.POST("/sops/:sid:activate", h.ActivateSOP)     // 🚪 需审批角色
        v1.POST("/sops/:sid:deprecate", h.DeprecateSOP)

        // 巡检与看板
        v1.POST("/inspections", h.TriggerInspection)
        v1.GET("/inspections/:rid", h.GetInspectionReport)
        v1.GET("/dashboards/health", h.GetHealthDashboard)
        v1.GET("/dashboards/capacity", h.GetCapacityDashboard)
        v1.GET("/dashboards/cost", h.GetCostDashboard)
    }
}
```

## 9. 人机闸门

### 9.1 🚪 G6 高风险运维操作（L3 双确认）

| 项 | 内容 |
|---|---|
| 触发 | 自愈 SOP 风险等级=high，或手工发起高风险操作 |
| 决策者 | 操作人（发起）+ 运维负责人（确认）= 双确认 |
| AI 干的 | 匹配 SOP、起草执行方案、评估影响、准备回滚 |
| 人干的 | 双确认后方可执行 |
| 产出 | 审批记录入 audit + HealingRun 进入 approved → executing |
| 拒绝路径 | 驳回/改方案/升级人工 |

### 9.2 G6 高风险操作清单

```
🟥 删除生产数据 / 清空数据库表 / 不可逆数据变更
🟥 资金操作（计费系统变更 / 退款 / 充值逻辑修改）
🟥 对外 DNS / 域名变更
🟥 生产环境安全组 / 防火墙规则变更
🟥 大规模节点缩容（> 30% 节点）
🟥 模型生产路由切换（影响线上推理质量）
```

> **未过 G6 的高风险操作一律禁止执行**——这是硬约束，由 Go 后端在 HealingRun 状态机中强制校验。

## 12. 落地优先级

| 优先级 | 范围 | 阶段 | 说明 |
|---|---|---|---|
| 🟧 P2 | 告警接入 + 去噪 + Incident 管理 | 阶段二 | 基础监控与研判闭环 |
| 🟧 P2 | 根因分析 Agent（基础版：拓扑+日志关联） | 阶段二 | Python 服务就绪后上线 |
| 🟧 P2 | 巡检基础版（定时健康检查） | 阶段二 | MVP 巡检 |
| 🟧 P2 | 健康/容量/成本看板 | 阶段二 | 联动板块8 |
| 🟩 P3 | 自愈预案库 + 低风险自动执行 | 阶段三 | 依赖板块4 SOP 就绪 |
| 🟩 P3 | 🚪G6 高风险管控（双确认流程） | 阶段三 | |
| 🟩 P3 | 容量预测增强（AI 时序预测） | 阶段三 | 需历史数据积累 |
| 🟩 P3 | 运维知识闭环（Incident→SOP 自动沉淀） | 阶段三 | 研判 MVP 数据积累后 |

### 关键里程碑

1. **监控闭环**：告警接入→去噪→Incident 创建→人工处理→复盘。替代传统告警系统的最小闭环。
2. **AI 研判**：根因分析 Agent 上线，告警自动关联+根因推荐+影响评估。
3. **受控自愈**：SOP 预案库就绪，低风险自动执行，🚪G6 高风险双确认。

> 阶段二先做基础监控与研判，把"看得见"做扎实；阶段三再做自愈，确保"敢放手"——因为自愈的前提是 SOP 可信、回滚可靠、闸门牢固。
