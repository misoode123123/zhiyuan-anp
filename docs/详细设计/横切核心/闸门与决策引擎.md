# 智源 ANP 平台 · 闸门与决策引擎（详细设计）

> **文档定位**：Go 后端 `gate` 模块的核心设计——分级自治 L0-L3、六大闸门 G1-G6、配置驱动的闸门引擎、决策工作流、高风险升级、AI 辅助决策、审计联动、降级机制。是平台治理"心脏"（规则治理中心）的**执行中枢**，也是 Agent 不得跨越的红线。
> **语言/栈**：Go + Gin + go-playground/validator + sqlx + viper + zap；模块化单体中的 `gate` package。
> **配套**：软件开发架构设计、AI 内核与 Agent 编排、规则治理中心、项目空间与多项目管理。

---

## 1. 设计目标与定位

闸门（Gate）是平台所有自动化流程中**强制的人工/规则检查点**。规则治理中心定义"应该遵守什么"，闸门引擎负责"在关键节点把住关"。Agent 的任何产出在到达下一阶段前，若触及闸门，必须暂停、经决策、留痕后方可放行。

| 目标 | 说明 |
|---|---|
| 不可绕过 | Agent 与自动化流程无法跳过闸门（引擎强制挂起） |
| 配置驱动 | 闸门启用/阈值/审批角色/SLA 均可配置，非硬编码 |
| 可插拔 | 新增闸门=注册配置，不改引擎 |
| 规则映射 | 规则（RaC）可映射为闸门的通过条件 |
| 留痕完备 | 每次决策（通过/拒绝/升级）不可篡改地入 audit |
| 自治分级 | L0-L3 决定哪些可自动、哪些需人审、哪些需双确认 |
| 可降级 | 闸门引擎故障时降级为"全人工"，绝不放任 |

🟥 **核心红线**：Agent 不得自行放行闸门、不得跳过闸门、不得删除决策记录。这三条在引擎、工具市场、audit 三层兜底（见 AI 内核文档 9.3）。

## 2. 两大维度：自治等级与闸门

平台治理由两个正交维度构成：**自治等级（L）**决定"谁来拍板"，**闸门（G）**决定"在哪卡点"。

### 2.1 自治等级 L0-L3

| 等级 | 名称 | 决策方 | 留痕 | 典型场景 |
|---|---|---|---|---|
| L0 | 全自动 | 系统自动 | 全程 | 实验项目、低风险常规编码 |
| L1 | 知会 | 系统执行，事后通知人 | 通知记录 | 常规 PR 合入（AI 预审通过） |
| L2 | 审批 | 人工审批通过才执行 | 审批记录 | 架构变更、发布、常规运维 |
| L3 | 双确认 | 两人独立确认 | 双签记录 | 高风险（生产数据、资金、对外发布） |

### 2.2 L 等级与 G 闸门的关系

- **L 等级是"默认审批强度"**：一个操作的 L 等级决定它是否需要人介入、需要几人。
- **G 闸门是"具体卡点"**：在流程特定位置（需求/架构/代码/安全/上线/运维）强制执行检查，其通过条件可包含 L 等级要求。
- **取严原则**：当某操作同时被 L 等级与 G 闸门约束时，**取更严者**。如 L1 默认知会，但触及 G5 上线闸门，则按 G5 要求人审。

```
操作的自治等级（项目空间配置 default_autonomy）
        │
        ▼
流程序列中的闸门（G1→G2→G3→G4→G5→G6）
        │
        ▼
取严：max(L要求, G要求) 作为该节点的实际审批强度
```

## 3. 六大闸门 G1-G6 定义

| 闸门 | 名称 | 触发条件 | 审批角色 | 默认 SLA | 自治默认 |
|---|---|---|---|---|---|
| 🚪G1 | 需求确认 | 需求规格起草完成、进入开发前 | 产品/需求负责人 | 1 工作日 | L2 |
| 🚪G2 | 架构评审 | 涉及架构变更、新模块、跨域影响 | 架构师 | 2 工作日 | L2 |
| 🚪G3 | 代码评审 | PR 创建、合入主干前 | 维护者/资深开发 | 4 小时 | L1（AI 预审通过可自动） |
| 🚪G4 | 安全合规 | 代码变更涉及安全敏感路径/数据/依赖 | 安全负责人 | 1 工作日 | L2 |
| 🚪G5 | 上线发布 | 发布到 staging/prod 前 | 发布负责人 | 2 小时 | L2（prod 升 L3） |
| 🚪G6 | 高风险运维 | 生产运维操作（扩缩容/重启/数据迁移） | 运维负责人 | 30 分钟 | L3（双确认） |

### 3.1 各闸门详细定义

#### 🚪G1 需求确认

```
触发：requirement.status = drafted AND 准备进入 dev
通过条件（全部满足）：
  ✓ 需求规格完整（验收标准非空）
  ✓ 规则引擎校验通过（需求模板规则）
  ✓ 优先级与预算已设定
  ✓ 产品负责人人工确认（L2）
失败动作：退回需求工作台，附评审意见
留痕：决策记录入 audit，关联需求 ID
```

#### 🚪G2 架构评审

```
触发：dev 识别为"架构变更"（影响 > N 模块 或 新增外部依赖 或 新 schema）
通过条件：
  ✓ 架构方案文档（Agent 起草）
  ✓ ADR 已提交
  ✓ 架构师人工评审通过（L2）
  ✓ AI 辅助：影响面分析（Python 给摘要，人定夺）
失败动作：退回方案修订
```

#### 🚪G3 代码评审

```
触发：PR 创建（pipeline 检测到 MR 事件）
通过条件：
  ✓ CI 全绿（lint/test/coverage/scan）
  ✓ AI 预审报告（Python 评审 Agent 产出）
  ✓ 规则合规校验通过
  ✓ 人审：核心模块≥1人，高风险≥2人；低风险经AI预审可 L1 自动合入
失败动作：PR 阻塞，附评审意见与规则违规项
```

⭐ G3 是 AI 协作的核心闸门：AI 预审强制，人审按风险分级。这是"双层评审"的技术实现。

#### 🚪G4 安全合规

```
触发：代码变更触及安全敏感路径（认证/加密/数据访问）或新增依赖
通过条件：
  ✓ SAST/SCA 扫描无高危
  ✓ 数据分级合规（无越权访问 restricted 数据）
  ✓ 密钥扫描无泄漏
  ✓ 安全负责人确认（L2）
失败动作：阻塞，安全告警入合规中心
```

#### 🚪G5 上线发布

```
触发：pipeline 达到发布阶段
通过条件：
  ✓ staging 验收通过
  ✓ 变更单 + 回滚预案就绪
  ✓ 发布窗口合规（规则引擎校验时间/冻结期）
  ✓ 发布负责人审批（L2）；prod/对外发布升 L3 双确认
  ✓ 灰度策略已配置
失败动作：阻塞发布
```

#### 🚪G6 高风险运维

```
触发：生产运维操作（扩缩容/重启/数据迁移/配置变更）
通过条件：
  ✓ 运维方案（AI 研判 Agent 起草）
  ✓ 影响面评估（Python 给摘要）
  ✓ 运维负责人 + 第二确认人 双签（L3）
  ✓ 回滚预案就绪
失败动作：操作阻塞，转人工
```

### 3.2 闸门触发判定流程

```
流程节点产出 / Agent 到达 gate 节点
  │
  ▼
gate.EvaluateTrigger(space, event) ── 查 GateDefinition 触发条件
  │
  ├─ 不触发 → 放行
  │
  ▼ 触发
创建 ApprovalInstance（状态 PENDING）
  │
  ▼
异步调 Python：请求 AI 辅助决策（风险摘要/影响面/建议）
  │
  ▼
通知审批人（按角色解析）── 等待 ──▶ 人在前端审批
  │
  ▼
决策：通过 / 拒绝 / 升级
  │
  ▼
留痕：DecisionRecord 入 audit + gate
  │
  ▼
通过 → 通知 agent-runtime.ResumeAtGate / 推进流程
拒绝 → 通知 Agent 回滚/重做
```

## 4. 闸门引擎设计

### 4.1 引擎职责

```
┌──────────────────────────────────────────────────┐
│  GateEngine（核心）                               │
│                                                   │
│  ① 触发评估 EvaluateTrigger                       │
│     · 事件驱动（来自 pipeline/PR/告警）            │
│     · 条件评估（规则映射）                         │
│                                                   │
│  ② 实例管理 ApprovalInstance 生命周期              │
│     · PENDING → APPROVED/REJECTED/EXPIRED         │
│                                                   │
│  ③ 审批人解析 ResolveApprovers                    │
│     · 按角色 + ABAC 解析谁有权审                   │
│                                                   │
│  ④ SLA 计时与超时升级                              │
│     · 超时未审 → 升级上级/转人工                   │
│                                                   │
│  ⑤ AI 辅助编排 RequestAIAssist                    │
│     · gRPC 调 agent-runtime 给风险摘要             │
│                                                   │
│  ⑥ 决策留痕 RecordDecision                        │
│     · 不可篡写入 audit                            │
│                                                   │
│  ⑦ 放行/回滚回调                                  │
│     · 通知流程引擎/Agent 恢复                     │
└──────────────────────────────────────────────────┘
```

### 4.2 配置驱动与可插拔

闸门不是硬编码 if-else，而是**配置驱动**。GateDefinition 存库，引擎按配置执行。

```go
// gate/definition.go
type GateDefinition struct {
    ID              uuid.UUID `db:"id"`
    GateCode        string    `db:"gate_code" validate:"oneof=G1 G2 G3 G4 G5 G6"` // 六大闸门编码
    Name            string    `db:"name"`
    ProjectSpaceID  uuid.UUID `db:"project_space_id"` // ⭐ 作用空间（uuid.Nil=全局）
    Enabled         bool      `db:"enabled"`
    TriggerEvent    string    `db:"trigger_event"`    // 触发事件类型
    TriggerCondition json.RawMessage `db:"trigger_condition"` // 条件表达式（CEL/JSONLogic）
    RequiredRoles   []string  `db:"required_roles"`   // 审批角色
    MinApprovers    int       `db:"min_approvers"`    // 最少审批人数（L3=2）
    AutonomyFloor   int       `db:"autonomy_floor"`   // 审批强度下限 0-3
    SLASeconds      int       `db:"sla_seconds"`      // 审批 SLA
    EscalationPolicy json.RawMessage `db:"escalation_policy"` // 超时升级策略
    RuleMappings    []uuid.UUID `db:"rule_mappings"`  // 映射的规则 ID（通过条件）
    AIAssistEnabled bool      `db:"ai_assist_enabled"` // 是否请求 AI 辅助
    IsDestructive   bool      `db:"is_destructive"`   // 关联操作是否破坏性
    Version         int       `db:"version"`
    CreatedAt       time.Time `db:"created_at"`
    UpdatedAt       time.Time `db:"updated_at"`
}
```

⭐ `TriggerCondition` 用条件表达式（CEL 或 JSONLogic），支持如 `"change.affected_modules > 3 || change.new_external_dependency == true"`。这让闸门触发可由规则治理中心动态配置，而非改代码。

### 4.3 规则映射到闸门

规则（RaC）可映射为闸门的**通过条件**：

```
规则治理中心：定义 Rule R1="PR 不得引入高危依赖"
              │
              ▼
映射到 G3 闸门的 rule_mappings: [R1]
              │
              ▼
G3 评估时：rule.Validate(artifact, [R1])
              │
              ├─ 通过 → 该条件满足
              └─ 不通过 → G3 阻塞，附违规详情
```

一个闸门可映射多条规则，**全部通过**才算该条件满足；规则与人工审批是 AND 关系。

## 5. 决策工作流

### 5.1 完整工作流

```
①起草（Draft）
  · Agent 或人触发闸门 → 创建 ApprovalInstance
  · AI 起草决策辅助信息（变更摘要/风险/影响面/建议）
  │
  ▼
②评估（Evaluate）
  · 规则自动校验（调 Go rule）
  · AI 辅助：Python 给风险摘要/影响面/建议置信度
  · 自动判定部分：规则全过 → 标记"自动条件满足"
  │
  ▼
③审批（Approve）—— 按 L 等级
  · L0/L1：自动或知会
  · L2：单审批人确认
  · L3：双确认（两人独立）
  │
  ▼
④留痕（Record）
  · DecisionRecord 入 audit（不可篡改）
  · 记录：谁/何时/依据/规则结果/AI建议/最终决策
  │
  ▼
⑤执行（Execute）
  · 通过 → 回调流程/Agent 恢复（ResumeAtGate）
  · 拒绝 → 回滚/重做
  · 执行结果回写
```

### 5.2 L3 双确认流程

```
ApprovalInstance（L3）
  │
  ▼
第一确认人 A 审批 ── 通过
  │ （状态：FIRST_APPROVED，仍不可执行）
  ▼
第二确认人 B 审批（须与 A 不同人，且角色合格）
  │
  ├─ 通过 → 双签完成 → 执行
  └─ 拒绝 → 终止
  │
  ▼ SLA 超时未获双签
  自动转人工紧急处理（不自动放行）
```

🟥 L3 场景下，**单方确认永远不可执行**——引擎在 Execute 前强制校验 `approvals.count(distinct person) >= MinApprovers`。

## 6. 高风险升级

### 6.1 升级触发

| 触发 | 动作 |
|---|---|
| 操作标记为高风险（如 prod 发布、数据迁移、资金） | 自动升至 L3 |
| AI 辅助判定风险高（置信度低/影响面大） | 建议升级（人最终定夺） |
| 规则引擎命中"强制升级"规则 | 自动升级 |
| SLA 超时 | 升级上级/转人工 |
| 审批人主动点"升级" | 进入 L3 流程 |

### 6.2 高风险操作清单（默认 L3）

| 操作 | 原因 |
|---|---|
| 对外正式发布 | 影响外部用户 |
| 生产数据库 schema 变更 | 数据风险 |
| 生产数据批量修改/删除 | 不可逆 |
| 资金/计费相关 | 资损 |
| 安全配置变更（鉴权/加密/防火墙） | 安全风险 |
| 删除审计记录之外的任何生产数据 | 合规 |

⭐ 此清单可由规则治理中心动态配置，项目空间可叠加（不可缩减）。

## 7. AI 辅助决策

闸门决策的**最终拍板权在人**，但 AI 提供"参谋"：风险摘要、影响面、建议、置信度。

### 7.1 AI 辅助内容

| 内容 | 来源 | 用途 |
|---|---|---|
| 变更摘要 | Python Agent（读 PR/变更单） | 审批人快速理解 |
| 风险评分 | Python（规则 + 模型推理） | 量化风险 |
| 影响面分析 | Python（依赖图 + 历史记忆） | 评估波及范围 |
| 建议决策 | Python（基于规则与历史） | 供参考，不构成授权 |
| 置信度 | Python | 低置信度→建议升级 |
| 历史类似案例 | L2/L3 记忆召回 | 类比参考 |

### 7.2 集成链路

```
ApprovalInstance 创建
  │
  ▼
gate.RequestAIAssist(gRPC → agent-runtime)
  · 任务类型：risk_assessment
  · 输入：变更内容、规则上下文、空间记忆
  │
  ▼
Python 评审/研判 Agent 运行
  · 调规则引擎校验
  · 召回历史案例
  · 模型推理风险
  │
  ▼
返回：RiskAssessment{score, impact, recommendation, confidence, citations}
  │
  ▼
写入 ApprovalInstance.ai_assist（供审批人查看）
  │
  ▼
人审时参考 AI 建议，但独立决策
```

🟥 **AI 建议不构成授权**：即使 AI 建议"通过"，仍需人工审批方为有效（见开发标准 10.3）。AI 建议只是信息输入。

## 8. 与审计的关系

### 8.1 留痕契约

每次闸门决策产生一条不可篡改的 `DecisionRecord`，同步入 `audit`。

```go
// gate 与 audit 的契约
type DecisionRecord struct {
    DecisionID    uuid.UUID
    InstanceID    uuid.UUID
    GateCode      string    // G1-G6
    ProjectSpaceID uuid.UUID
    Decision      string    // approved/rejected/escalated/expired
    DecidedBy     []uuid.UUID // 决策人（L3 可多人）
    DecidedAt     time.Time
    Rationale     string    // 决策理由（人填）
    RuleResults   []RuleResult // 规则校验结果
    AIAssistRef   *uuid.UUID // AI 辅助报告引用
    ArtifactRef   string    // 决策对象（PR/需求/变更单）
    AutonomyLevel int
    TraceID       string
}
```

### 8.2 审计不可篡改保证

- `audit` 表只允许追加（INSERT），**禁止 UPDATE/DELETE**（DB 权限层兜底）。
- agent-runtime 对 audit 表**仅有追加权限**，无删改。
- 关键决策记录可加哈希链（前一条 hash 纳入后一条计算），防篡改。

### 8.3 审计回溯场景

| 场景 | 用审计 |
|---|---|
| 事故复盘 | 查 G5 决策记录 → 谁批准了这次发布 |
| 合规检查 | 导出某空间全部闸门决策 |
| 规则违规追责 | 查为何 G3 放行了违规 PR |
| Agent 行为审计 | 查 Agent 触发了哪些闸门、AI 给了什么建议 |

## 9. 降级机制

### 9.1 故障降级

| 故障 | 降级动作 |
|---|---|
| 闸门引擎自身故障 | **全闸门转人工**：所有自动化暂停，操作必须人手动（宁严勿松） |
| AI 辅助不可用 | 审批继续，但标记"无 AI 辅助"，审批人自负全责 |
| 规则引擎不可用 | 闸门降级为"纯人工判断"（规则校验跳过，留痕标注） |
| 通知系统故障 | SLA 时钟照走，超时升级 |
| agent-runtime 不可用 | 闸门仍可用（人审独立于 Agent），但无 AI 辅助 |

🟥 **故障默认收紧，不放宽**：闸门引擎故障时绝不让操作自动通过，而是转人工——这是"守边界"约束在故障态的体现。

### 9.2 紧急放行（break-glass）

极少数情况（生产事故抢修）需绕过常规审批：

```
紧急放行流程：
  · 操作人申请 break-glass（附理由）
  · 需 ≥ 2 名授权人员在线确认（即使平时 L2，紧急也需双签）
  · 强制记录全程（录像/命令留痕）
  · 事后 24h 内必须补正式决策记录与复盘
  · 每次 break-glass 自动触发安全告警入合规中心
```

⭐ break-glass 不是"无闸门"，而是"简化但留痕的紧急闸门"——仍有双签与强制复盘。

// gate/http.go
func RegisterRoutes(r *gin.Engine, h *Handlers, mw *Middlewares) {

        gates.POST("/instances/:instanceId/escalate", h.Escalate) // 升级

        gates.GET("/decisions", h.ListDecisions)                  // 决策历史

// gate/service.go
type Service interface {
    // 评估触发（pipeline/dev/requirement 调用）
    EvaluateTrigger(ctx context.Context, event GateEvent) (*ApprovalInstance, error)

    GetInstance(ctx context.Context, id uuid.UUID) (ApprovalInstance, error)

## 12. 与 Agent / 规则引擎的联动

### 12.1 完整联动时序

```
① Agent（Python）执行到 Plan 的 gate 节点
        │ LangGraph interrupt_before
        ▼
② agent-runtime 发 TaskEvent(GATE_REACHED)
        │ gRPC StreamEvents
        ▼
③ Go 后端收到 → gate.EvaluateTrigger
        │ 创建 ApprovalInstance(PENDING)
        ▼
④ gate.RequestAIAssist → gRPC → Python risk_assessment Agent
        │ 返回 RiskAssessment
        ▼
⑤ 写入 AIAssistReport，通知审批人
        │
        ▼
⑥ 人审：前端 SubmitVote（L2 单签 / L3 双签）
        │
        ▼
⑦ gate 记录 DecisionRecord → 同步 audit
        │
        ▼
⑧ 通过：gate.ResolveForAgent → gRPC ResumeAtGate → Agent 恢复
   拒绝：通知 Agent 回滚/重做
        │
        ▼
⑨ 执行结果回写 + audit
```

### 12.2 与规则引擎的联动点

| 时机 | gate 行为 | rule 行为 |
|---|---|---|
| 触发评估 | 加载 GateDefinition.rule_mappings | 提供规则集 |
| 自动校验 | 调 rule.Validate(artifact, rule_ids) | 返回每条规则通过/违规 |
| 通过条件 | 规则全过 = 自动条件满足 | — |
| 违规展示 | 附违规详情给审批人 | 提供违规说明 |
| 动态映射 | 规则治理中心改映射 → gate 热加载 | 规则版本变更通知 gate |

⭐ 规则与闸门解耦：规则定义"什么对什么错"，闸门定义"何时检查、谁来放行"。规则可被多个闸门复用。

### 13.1 SLA 计时

### 13.2 超时升级策略（EscalationPolicy）

```json
// 示例：G5 发布闸门升级策略
{
  "on_timeout": "escalate",
  "escalate_to_role": "release_manager_backup",
  "grace_seconds": 600,
  "max_escalations": 2,
  "on_max_exceeded": "auto_reject_and_notify",
  "notify_channels": ["im", "email", "oncall"]
}
```

| 阶段 | 动作 |
|---|---|
| 到 SLA 50% | 提醒审批人 |
| 到 SLA 100% | 按策略升级（转备选审批人） |
| 升级达上限 | 自动拒绝 + 通知（不自动放行） |

🟥 **超时绝不自动通过**：最坏情况是拒绝/转人工，绝不让高风险操作因"没人审"而放行。

## 14. 关键设计取舍

| 决策 | 选择 | 理由 | 后果 |
|---|---|---|---|
| 闸门配置 | 配置驱动（DB + 表达式） | 可动态调整、可插拔 | 表达式引擎引入（CEL） |
| L 等级与 G 关系 | 取严原则 | 安全优先 | 偶尔比预期严（可接受） |
| AI 建议 | 仅参谋，人定夺 | 责任在人 | AI 价值受限但不担责 |
| 故障降级 | 全转人工 | 宁严勿松 | 故障期效率下降 |
| 审计不可篡改 | 只追加 + 哈希链 | 合规可信 | 无修正，需谨慎录入 |
| 紧急放行 | break-glass 双签+复盘 | 兼顾应急与留痕 | 流程略重 |
