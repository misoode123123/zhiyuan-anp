# 智源 ANP 平台 · 项目空间与多项目管理（详细设计）

> **文档定位**：Go 后端 `workspace` 模块的核心设计——项目空间作为组织一切的基本单元与多租户隔离键、三类项目接入形态、项目模板、公共/私有资产、跨项目关联、RBAC+ABAC 权限、配额管理。是九大板块共用的**组织骨架**。
> **语言/栈**：Go + Gin + go-playground/validator + sqlx + viper + zap；模块化单体中的 `workspace` package。
> **配套**：软件开发架构设计、AI 内核与 Agent 编排、闸门与决策引擎、规则治理中心。

---

## 1. 设计目标与定位

项目空间（Project Workspace）是平台的**基本组织单元**，也是**多租户隔离键**。一切实体——需求、代码、规则、Agent 任务、流水线、记忆——都隶属于某个项目空间。

| 目标 | 说明 |
|---|---|
| 隔离基线 | `project_space_id` 贯穿所有 schema，是数据隔离的硬边界 |
| 灵活接入 | 支持全自动纳入 / 轻量接入 / 自举三类形态 |
| 可配置治理 | 项目空间可叠加规则包、流水线模板、Agent 配置 |
| 公私分明 | 公共资产（企业级）与项目私有资产边界清晰 |
| 权限精细 | RBAC（角色）+ ABAC（项目空间、数据分级）双控 |
| 配额可控 | 每空间独立的算力/token/存储配额 |

⭐ **核心不变量**：任何一条业务数据都必须能通过 `project_space_id` 追溯到所属空间；任何一次 API 调用都必须能解析出 `project_space_id` 上下文。这是平台多租户安全的基石。

## 2. 核心概念

### 2.1 项目空间

```
企业（Enterprise / 平台）
  │
  ├── 项目空间 A（Project Space）── 产品线/团队/独立产品
  │     ├── 项目 A-1（Project）── 具体可交付项目
  │     │     ├── 需求集
  │     │     ├── 代码仓库（GitLab 绑定）
  │     │     ├── 流水线
  │     │     └── 项目记忆
  │     ├── 项目 A-2
  │     └── 空间级配置（规则包/模板/Agent/配额/成员）
  │
  ├── 项目空间 B
  │     └── ...
  │
  └── 公共资产（跨空间，企业级）
        ├── 企业级规则库
        ├── 企业记忆（Knowledge）
        └── 公共模板
```

### 2.2 层级关系

| 层级 | 实体 | 说明 |
|---|---|---|
| L0 | Enterprise/Platform | 平台本身，管理全局配置与公共资产 |
| L1 | Project Space | 多租户隔离单元，拥有独立成员/配额/规则包 |
| L2 | Project | 具体交付项目，绑定代码仓库与流水线 |
| L3 | 资产 | 需求/任务/规则/记忆/产物，归属某 Project，间接归属 Space |

🟥 **隔离边界在 L1（Project Space）**：跨 Space 的数据访问默认禁止；Project 间（同 Space 内）可有条件互通。

## 3. 三类项目接入形态

不同项目以不同"重量"接入平台。

### 3.1 形态对比

| 维度 | A 全自动纳入 | B 轻量接入 | C 自举（dogfooding） |
|---|---|---|---|
| 适用 | 平台内新研发项目 | 存量外部系统 | 平台自身开发 |
| 代码托管 | 平台内置 GitLab | 外部仓库，仅 Webhook 关联 | 平台自己的仓库 |
| 接管范围 | 全生命周期（需求→运维） | 仅需求 + 运维 | 全生命周期 |
| 流水线 | 平台流水线全自动 | 对接外部 CI 或只看结果 | 平台流水线 |
| 规则/闸门 | 全套 | 精简子集（需求+运维闸门） | 全套（最严，做示范） |
| Agent | 全套 Agent | 仅需求/运维 Agent | 全套 |
| 数据同步 | 双向实时 | 单向上报 + Webhook 回流 | 双向实时 |

### 3.2 形态 A：全自动纳入

```
新建项目空间 → 选模板 → 绑定 GitLab 仓库组 → 预置规则包/流水线/Agent
  → 需求工作台 → 研发工作台 → 测试中心 → 发布中心 → 运维中心 全自动打通
```

### 3.3 形态 B：轻量接入（存量系统）

```
外部系统 → 注册为项目空间 → 仅开通：
  ├── 需求工作台（接收需求、排期、状态回写）
  └── 运维中心（接收监控告警、巡检、人工运维）
代码与 CI 仍在原系统，平台经 Webhook/API 被动感知
```

⭐ 形态 B 解决"存量系统不想大改但想要 AI 治理"的场景。典型如：已有 Jira + 老系统，只想要 AI 需求澄清与运维巡检。

### 3.4 形态 C：自举（dogfooding）

```
平台开发自身 → 作为项目空间接入 → 用平台的 Agent/闸门/规则 管控平台开发
  → 平台自己的 PR 也过 G3、平台自己发布也过 G5
```

🟥 形态 C 是平台可信度的证明：自己用自己的闸门，且配置最严。这是治理可信的关键。

## 5. 多租户隔离实现

### 5.1 隔离策略对比

| 策略 | 本平台选择 | 说明 |
|---|---|---|
| 独立数据库 | 否 | 成本高，起步不采用 |
| 独立 schema | 部分采用 | 按域分 schema（非按租户） |
| 行级过滤 + 强制键 | 🟥 主策略 | 所有表带 project_space_id，ORM/中间件强制注入 |

### 5.2 强制注入机制

```go
// workspace/context.go
package workspace

// ContextKey 用于在 context.Context 中传递当前空间
type ContextKey struct{}

// WithSpace 把 project_space_id 注入 context
func WithSpace(ctx context.Context, spaceID uuid.UUID) context.Context {
    return context.WithValue(ctx, ContextKey{}, spaceID)
}

// FromContext 提取，所有 Service 层方法首步必调
func FromContext(ctx context.Context) (uuid.UUID, error) {
    v, ok := ctx.Value(ContextKey{}).(uuid.UUID)
    if !ok {
        return uuid.Nil, ErrNoSpaceContext
    }
    return v, nil
}
```

```go
// workspace/middleware.go
// Gin 中间件：从路由/JWT 解析 project_space_id 并注入 context
func SpaceResolver() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 优先级：路径参数 > 查询参数 > JWT claim
        spaceIDStr := c.Param("spaceId")
        if spaceIDStr == "" {
            spaceIDStr = c.Query("space_id")
        }
        spaceID, err := uuid.Parse(spaceIDStr)
        if err != nil {
            // 尝试从 JWT
            if claims, ok := c.Get("claims"); ok {
                spaceID = claims.(JWTClaims).DefaultSpaceID
            }
        }
        if spaceID == uuid.Nil {
            c.AbortWithStatusJSON(400, gin.H{"code": "no_space", "message": "project_space_id required"})
            return
        }
        // ⭐ 权限校验：用户是否属于该空间
        if !membershipService.CanAccess(c, spaceID) {
            c.AbortWithStatusJSON(403, gin.H{"code": "forbidden"})
            return
        }
        c.Request = c.Request.WithContext(workspace.WithSpace(c.Request.Context(), spaceID))
        c.Next()
    }
}
```

### 5.3 数据访问层的强制过滤

```go
// 所有 Repository 查询强制带 project_space_id（sqlx）
func (r *RequirementRepo) List(ctx context.Context, filter ReqFilter) ([]Requirement, error) {
    spaceID, err := workspace.FromContext(ctx)   // ⭐ 强制取空间
    if err != nil {
        return nil, err
    }
    const q = `SELECT * FROM requirement.requirements
               WHERE project_space_id = $1 AND status = ANY($2)
               ORDER BY created_at DESC LIMIT $3`
    return r.db.SelectContext(ctx, q, spaceID, pq.Array(filter.Status), filter.Limit)
}
```

🟥 **代码评审红线**：任何 SQL 查询若不带 `project_space_id` 条件，CI 静态检查即报错（自研 lint 规则）。这是隔离不被破坏的兜底。


## 6. 权限模型（RBAC + ABAC）

### 6.1 RBAC 角色定义

| 角色 | 空间级权限 | 项目级 |
|---|---|---|
| owner | 全部（含删除空间、转让） | 全部 |
| admin | 管理成员、配额、规则包、模板 | 项目级全部 |
| maintainer | 管理项目、流水线、闸门配置 | 负责项目全部 |
| developer | 创建需求/任务、提交 PR | 参与项目 |
| viewer | 只读 | 只读 |

### 6.2 ABAC 维度

```
访问决策 = f(角色, project_space_id, project_scope, data_classification, action, autonomy_level)
```

| 维度 | 示例 |
|---|---|
| 主体角色 | developer |
| 空间归属 | 必须在空间内 |
| 项目范围 | ProjectScope 限定 |
| 数据分级 | confidential 数据需更高角色 |
| 操作类型 | read/write/approve/publish |
| 自治等级 | L3 操作需双确认 |

### 6.3 权限决策器

```go
// workspace/authz.go
type Permission struct {
    Resource string // requirement/project/rule/gate/quota/...
    Action   string // read/write/delete/approve/...
}

func (s *AuthService) Can(ctx context.Context, userID uuid.UUID, perm Permission, attrs Attrs) bool {
    // 1. RBAC：角色 → 是否允许该 action
    role, ok := s.getRole(userID, attrs.SpaceID, attrs.ProjectScope)
    if !ok || !role Allows perm {
        return false
    }
    // 2. ABAC：数据分级、自治等级等
    if attrs.DataClassification == "restricted" && role != Owner && role != Admin {
        return false
    }
    if attrs.AutonomyLevel >= 3 && !s.hasDualConfirm(role, attrs) {
        return false
    }
    return true
}
```

### 6.4 Agent 权限令牌签发

```go
// workspace 为 Agent 签发短期权限令牌（见 AI 内核文档 4.3）
func (s *TokenService) IssueAgentToken(ctx context.Context, spaceID uuid.UUID, taskID uuid.UUID, scopes []string) (string, error) {
    claims := AgentClaims{
        TaskID:    taskID,
        SpaceID:   spaceID,           // ⭐ 锁定空间
        Scopes:    scopes,            // 受限作用域
        ExpiresAt: time.Now().Add(30 * time.Minute),
        IssuedAt:  time.Now(),
    }
    return s.signer.Sign(claims)
}
```

## 7. 项目模板机制

### 7.1 模板内容

一个项目模板预置了"开箱即用"的配置包：

```
模板 = 规则包（RaC 子集）
     + 流水线定义（阶段/步骤）
     + Agent 配置（启用哪些 Agent、默认自治等级）
     + 闸门配置（启用哪些闸门、SLA）
     + 配额策略（默认配额）
     + 知识种子（初始企业记忆引用）
```

### 7.2 内置模板示例

| 模板 | 适用 | 预置内容 |
|---|---|---|
| `standard-product` | 新产品研发 | 全套规则、全流水线、G1-G6、L1 自治 |
| `library` | 内部库/SDK | 精简规则、CI 流水线、G3+G5、L2 |
| `legacy-integration` | 存量系统接入(B) | 需求+运维规则、G1+G6、L2 |
| `experiment` | 实验性项目 | 最少规则、仅 G5、L0 自治（全自动） |
| `platform-self` | 平台自举(C) | 最严规则、全套闸门、L3 关键操作 |

### 7.3 模板实例化流程

```
选模板 → 创建项目空间
  │
  ▼
克隆规则包到空间（rule_pack 实例化）
  │
  ▼
预置流水线（pipeline 定义入库）
  │
  ▼
注册 Agent 配置（agent_config 入库）
  │
  ▼
配置闸门（gate 定义入库，启用对应 G1-G6）
  │
  ▼
应用配额策略（quota_usage 初始化）
  │
  ▼
种子知识挂载（引用企业记忆条目）
  │
  ▼
空间就绪，可创建项目
```

⭐ 模板让"新项目开箱即用治理"——这是平台降低治理门槛的关键体验。

## 8. 公共资产 vs 项目私有

| 资产类型 | 公共（企业级） | 项目私有 |
|---|---|---|
| 规则 | 企业红线规则（不可放宽） | 项目可叠加更严规则 |
| 知识记忆 | 企业记忆（L3，跨空间） | 项目记忆（L2，空间内） |
| 模板 | 公共模板（is_public=true） | 空间私有模板 |
| Agent 配置 | 全局默认 Agent | 项目级覆盖配置 |
| 流水线 | 全局流水线模板 | 项目流水线实例 |
| 闸门 | 全局最低闸门要求 | 项目可启用额外闸门 |

🟥 **单向收紧原则**：项目空间可在企业红线之上叠加更严规则/更多闸门，但**不得放宽全局红线**。规则引擎在加载时强制校验此约束。

## 9. 配额管理

### 9.1 配额检查点

```
Agent 任务启动前
  │
  ▼
workspace.CheckQuota(space_id, estimated_tokens)
  │
  ├─ 查 Redis 实时计数（当日 token/agent runs）
  ├─ 超额 → 拒绝（L0 任务降级人工，L1+ 提示用户）
  └─ 未超 → 预扣估算值（乐观锁）
  │
  ▼
任务执行中实时回写真实消耗
  │
  ▼
任务结束 → 修正预扣 → 落 PG（定时对账）
```

### 9.2 配额状态机

| 状态 | 触发 | 行为 |
|---|---|---|
| normal | 用量 < 80% | 正常 |
| warning | 80% ≤ 用量 < 100% | 通知 owner |
| exceeded | 用量 ≥ 100% | 拒绝新 AI 任务（人工流程不受影响） |
| throttled | 短时高频 | 限流（Redis 令牌桶） |

⭐ **配额不影响业务连续性**：超额只拒绝 AI 能力，Go 后端业务流程仍可用——这与"AI 故障不等于平台故障"一致。

```go
// workspace/http.go
func RegisterRoutes(r *gin.Engine, h *Handlers, mw *Middlewares) {
    v1 := r.Group("/api/v1")
    v1.Use(mw.Auth(), mw.SpaceResolver()) // ⭐ 所有业务路由经空间解析

    spaces := v1.Group("/spaces")
    {
        spaces.POST("", h.CreateSpace)              // 创建项目空间
        spaces.GET("", h.ListSpaces)                // 列出我可访问的空间
        spaces.GET(":spaceId", h.GetSpace)
        spaces.PATCH(":spaceId", h.UpdateSpace)
        spaces.POST(":spaceId/archive", h.ArchiveSpace)

        // 成员
        spaces.GET(":spaceId/members", h.ListMembers)
        spaces.POST(":spaceId/members", h.AddMember)
        spaces.PATCH(":spaceId/members/:userId", h.UpdateMemberRole)
        spaces.DELETE(":spaceId/members/:userId", h.RemoveMember)

        // 项目
        spaces.POST(":spaceId/projects", h.CreateProject)
        spaces.GET(":spaceId/projects", h.ListProjects)
        spaces.GET(":spaceId/projects/:projectId", h.GetProject)

        // 配额
        spaces.GET(":spaceId/quota", h.GetQuota)

        spaces.POST(":spaceId/templates", h.CreateTemplate)        // 空间私有模板
```

## 13. 关键设计取舍

| 决策 | 选择 | 理由 | 后果 |
|---|---|---|---|
| 隔离粒度 | 行级 project_space_id（非独立库/schema） | 运维简单、成本可控 | 需代码层强制（中间件+lint） |
| 模板机制 | 预置规则+流水线+Agent+闸门 | 开箱即用治理 | 模板维护成本 |
| 接入形态 | A/B/C 三类 | 覆盖新/旧/自举 | 配置分支复杂度 |
| 跨空间关联 | Portfolio（视图层，不破隔离） | 平衡协同与安全 | 关联只读 |
| 配额 | Redis 计数+PG 对账 | 实时+一致 | 对账窗口期 |
| 公共/私有 | 单向收紧（项目只能更严） | 治理底线不被绕过 | 灵活性受限（可接受） |
