# 智源 ANP 平台 · 板块 8 · 算力与资源中心（详细设计）

> **文档定位**：平台愿景"企业只投入算力"的落点。GPU/模型/Token/配额的统一管理枢纽——像水电一样按需调度、可计费、可优化。为全平台所有 AI 能力消费方（板块1/2/3/7/9）提供统一的算力池与模型网关，是 9 大板块的**资源地基**。
> **技术基线**：业务后端 Go（Gin + sqlx + zap）管理算力池/配额/计费/用量记录；**模型网关与路由基于 Python 侧 LiteLLM**（多供应商接入/故障切换/负载均衡），路由策略 Agent 与成本优化 Agent 由 **LangGraph** 承载，Go 经 **gRPC** 调用。持久化 PostgreSQL（schema: `compute`）+ Redis（实时配额缓存/限流）。
> **配套**：软件开发架构设计、开发标准与规范、总体方案 V1.0。
> **多租户**：所有数据带 `project_space_id` 作为路由键。

## 1. 定位与价值

### 1.1 定位

算力与资源中心是平台的**资源总线和模型网关**。一切 AI 能力消费（需求澄清、代码生成、测试生成、智能研判、技能市场调用）都经此获取算力与模型，不走直连。它把异构算力（自建 GPU + 弹性云）和多家模型（OpenAI/Anthropic/信创国产模型）抽象为统一接口，按任务智能路由，按用量精确计费，按配额严格管控。

⭐**核心设计原则**：模型/算力经 LiteLLM 网关解耦——上游消费方不感知具体供应商，只声明任务需求（难度/延迟/成本偏好），网关自动选最优路径。

```
消费方(板块1/2/3/7/9)          算力与资源中心               供应商
    │ "我要做代码生成"             │                         │
    │ ──────────────────────────▶ │                         │
    │  声明: 难度=高/延迟可容忍    │ 路由Agent选模           │
    │  预算=0.5元                  │ ──────────────────────▶ │ OpenAI/Claude/信创
    │                              │ ◀── 推理结果 + 用量 ──── │
    │ ◀── 结果 + 扣费记录 ──────── │                         │
    │                              │ 计费/配额/成本看板       │
```

## 3. 功能模块

### 3.1 算力池管理

- **自建 GPU 池**：管理本地 GPU 集群（型号/数量/拓扑），统一纳管调度。
- **弹性云**：对接公有云 GPU（阿里云/华为云/腾讯云），按需弹性扩缩。
- **统一调度**：自建优先（成本低），溢出弹云；任务优先级驱动调度。
- **健康监控**：GPU 利用率/温度/显存/故障率实时监控，联动板块7运维中心。

### 3.2 模型网关与路由

⭐核心能力，基于 Python 侧 **LiteLLM** 实现：

- **多模型接入**：统一接入 OpenAI / Anthropic / 信创模型（通义/文心/智谱/DeepSeek），标准化接口。
- **按任务路由**：路由 Agent 根据**任务难度 × 成本 × 延迟**三维决策自动选模型。
- **故障切换**：主模型不可用时自动切换备用模型（≤30 秒）。
- **负载均衡**：多实例/多 Key 池化轮转，突破单 Key 限流。
- **缓存**：相同 prompt 命中缓存（Redis / 向量缓存），降低重复调用成本。
- **信创路由**：涉密/敏感数据场景强制路由到信创模型（规则驱动）。

### 3.3 配额与计费

- **配额管理**：按部门/项目空间分配 Token 配额、GPU 时配额，周期可配（月/季）。
- **实时扣减**：每次模型调用实时扣减配额，超配额硬拦截。
- **计费核算**：按模型定价（input/output 分别计价）精确计费，支持内部结算。
- **预算预警**：接近阈值（80%/90%/100%）自动告警。

### 3.4 成本看板

- **实时看板**：Token 消耗/算力消耗实时可见，按部门/项目/模型/服务多维度拆解。
- **趋势分析**：历史消耗趋势，预测未来成本。
- **异常告警**：消耗突增/异常趋势自动发现（联动板块7）。
- **账单导出**：按周期生成账单，支持导出（内部结算凭证）。

### 3.5 弹性与抢占

- **优先级队列**：任务分优先级（high/medium/low），高优保障资源。
- **抢占机制**：资源紧张时低优任务可被抢占（保存 checkpoint），保障高优。
- **自动扩缩容**：基于队列深度/GPU 利用率自动扩缩容算力池。
- **SLA 保障**：高优任务 SLA 保障率 ≥ 99%。

## 5. AI Agent 设计

> 路由 Agent 和成本优化 Agent 在 **Python 侧**（agent-runtime 服务，LangGraph + LiteLLM），Go 经 gRPC 调用。Agent 产出遵守五约束。

### 5.1 Agent 清单

| Agent | 职责 | 输入 | 输出 | 自治级 |
|---|---|---|---|---|
| 路由 Agent (RoutingAgent) | 按任务难度×成本×延迟选最优模型 | 任务描述+难度+预算+延迟要求+约束 | model_id + endpoint + 路由理由 | L0-L1 |
| 成本优化 Agent (CostOptAgent) | 识别浪费、建议缓存/降档/合并调用 | 用量数据+调用模式分析 | 优化建议清单（降档/缓存/合并） | L1 |

### 5.2 路由 Agent 决策逻辑

```
[invoke_request] ─▶ [parse_task] ─▶ [check_constraints]
                                         │
                               ┌─────────┴──────────┐
                               ▼                    ▼
                       [is_xinchu_required?]   [is_cache_hit?]
                               │                    │
                          yes  │               yes  │
                               ▼                    ▼
                      [filter_xinchu]         [return_cache]
                               │
                               ▼
                      [score_models]  ← 难度×成本×延迟×可用性
                               │
                               ▼
                      [select_best] ─▶ [emit_route]
```

- **约束检查链**：信创约束 → 预算约束 → 延迟约束 → 白名单约束 → 缓存检查 → 评分选模。
- **故障切换**：主模型调用失败/超时 → 自动重试备用模型（最多 2 次切换）。
- 出站模型调用经 LiteLLM 统一接口。

### 5.3 gRPC IDL 概要（compute_agent.proto）

```protobuf
service ComputeAgentService {
  // 模型路由决策
  rpc RouteModel(RouteRequest) returns (RouteResponse);
  // 成本优化分析
  rpc AnalyzeCost(CostAnalysisRequest) returns (CostAnalysisResponse);
  // 弹性调度建议
  rpc SuggestScaling(ScalingRequest) returns (ScalingResponse);
}

message RouteRequest {
  string project_space_id = 1;
  string trace_id = 2;
  string task_type = 3;          // codegen / chat / embedding / analysis
  string difficulty = 4;         // high / medium / low
  float  budget = 5;             // 预算上限(元)
  string latency_pref = 6;       // realtime / standard / tolerant
  bool   xinchu_required = 7;    // 信创约束
  string prompt_preview = 8;     // prompt 摘要(用于缓存检查)
}

message RouteResponse {
  string model_id = 1;
  string endpoint_id = 2;
  string reason = 3;             // 路由理由
  bool   cache_hit = 4;
  float  estimated_cost = 5;
  float  estimated_latency_ms = 6;
  string model_version = 7;
}
```

### 5.4 透明度要求

每次路由决策记录：候选模型列表、评分明细、选中理由、约束命中情况、缓存命中情况——全部落 UsageRecord 可追溯。

```go
// internal/compute/api/router.go

func RegisterComputeRoutes(r *gin.Engine, h *ComputeHandler, auth middleware.AuthMiddleware) {
    // 统一调用入口(独立鉴权, APIKey 或 Bearer)
    r.POST("/api/v1/compute/invoke", auth.RequireAPIKey(), h.InvokeModel)
    r.POST("/api/v1/compute/invoke:stream", auth.RequireAPIKey(), h.InvokeModelStream)
    r.GET("/api/v1/compute/models", h.ListModels)
    r.GET("/api/v1/compute/models/:mid", h.GetModel)

    // 项目空间级管理
    v1 := r.Group("/api/v1/projects/:psid/compute")
    v1.Use(auth.Require(), auth.RequireTenant())
    {
        // 算力池
        v1.GET("/pools", h.ListPools)
        v1.POST("/pools", h.CreatePool)
        v1.PUT("/pools/:pid", h.UpdatePool)
        v1.GET("/pools/:pid/health", h.GetPoolHealth)

        // 模型管理
        v1.POST("/models", h.RegisterModel)               // 🚪 需审批
        v1.PUT("/models/:mid", h.UpdateModel)
        v1.POST("/models/:mid:activate", h.ActivateModel) // 🚪 需审批
        v1.POST("/models/:mid:deprecate", h.DeprecateModel)
        v1.GET("/routing-policies", h.ListRoutingPolicies)
        v1.PUT("/routing-policies", h.UpdateRoutingPolicies)

        // 配额与计费
        v1.GET("/quota", h.GetQuota)
        v1.POST("/quota:request", h.RequestQuota)
        v1.PUT("/quota", h.AdjustQuota)                    // 🚪 需审批
        v1.GET("/usage", h.GetUsage)
        v1.GET("/cost", h.GetCostDashboard)
        v1.GET("/billing", h.GetBilling)
    }
}
```

## 8. 规则与约束

> 本板块**读**规则治理中心（板块4）的规则资产，不自己维护规则。

### 8.1 知识资产消费清单

| 资产类型 | 内容 | 用途 | 强制度 |
|---|---|---|---|
| Rule | 成本预算规则 | 配额管控依据 | 🟥 强制 |
| Rule | 模型白名单（信创） | 涉密场景强制路由信创模型 | 🟥 强制 |
| Rule | 数据驻留规则 | 敏感数据不出境，约束模型/地域选择 | 🟥 强制 |
| Process | 新模型接入流程 | 需评审+审批+测试 | 🟥 强制 |
| Process | 配额追加流程 | 需审批 | 🟥 强制 |
| Knowledge | 模型能力基准评测 | 路由评分依据 | 🟧 应遵循 |

### 8.2 Agent 五约束落地

| 约束 | 算力场景落地 |
|---|---|
| 可校验 | 路由决策可复现（route_reason + 评分明细可审计） |
| 可追溯 | 每次调用 UsageRecord 全量记录（caller/model/tokens/cost/trace_id） |
| 可回滚 | 配额扣减可回补（调用失败/退款） |
| 守边界 | 消费方不得直连模型供应商，必须经网关 |
| 宥权限 | 路由 Agent 不可超预算、不可越白名单、不可绕信创约束 |

### 8.3 信创合规约束

```
🟥 涉密/敏感数据场景:
   - 数据分级标记(来自板块5安全合规中心)
   - 路由 Agent 强制过滤 is_xinchu=true 模型
   - 禁止调用境外供应商(OpenAI/Anthropic)
   - 数据驻留: 仅路由到本地部署/信创云

🟧 一般场景:
   - 无信创约束, 路由 Agent 按性价比自由选择
```

## 9. 人机闸门

### 9.1 审批清单

| 闸门 | 触发场景 | 约束 | 自治级 |
|---|---|---|---|
| 🚪新模型接入 | 注册新 ModelEndpoint | 需评审（能力/安全/合规）+ 审批 | L2 |
| 🚪模型激活 | 模型进入生产路由 | 审批 + 灰度验证 | L2 |
| 🚪配额追加 | 部门超预算申请追加 | 审批（部门负责人→平台管理员） | L2 |
| 🚪大规模扩容 | 算力池扩容 > 20% | 审批 + 财务确认 | L2-L3 |
| 路由选择 | 日常模型路由 | 自动 | L0-L1 |
| 成本优化建议 | 降档/缓存建议 | 知会（采纳可选） | L1 |
| 抢占调度 | 低优被高优抢占 | 自动 + 知会 | L1 |

### 9.2 闸门执行机制

- Go 后端在模型接入/配额变更操作前强制校验审批状态。
- 未审批的新模型不进入路由候选池（`is_whitelisted=false` 的模型不参与路由）。
- 所有审批记录入 audit，含决策人、时间、依据。

## 12. 落地优先级

| 优先级 | 范围 | 阶段 | 说明 |
|---|---|---|---|
| 🟥 P0 | 模型网关（LiteLLM）+ 统一调用入口 | 阶段一 | **地基**，所有 AI 能力经此调度 |
| 🟥 P0 | 基础模型接入（2-3 个主流模型） | 阶段一 | 覆盖 chat/codegen/embedding |
| 🟥 P0 | 配额管理（Token 配额 + 超额拦截） | 阶段一 | 成本管控底线 |
| 🟥 P0 | 用量记录 + 基础成本看板 | 阶段一 | 消耗可见 |
| 🟧 P1 | 路由 Agent（难度×成本×延迟选模） | 阶段一末 | 智能 routing |
| 🟧 P1 | 信创模型接入 + 信创路由约束 | 阶段一末 | 合规要求 |
| 🟧 P1 | 故障切换 + 负载均衡 | 阶段一末 | 高可用 |
| 🟩 P2 | 成本优化 Agent（缓存/降档建议） | 阶段二 | 降本增效 |
| 🟩 P2 | 弹性抢占调度 | 阶段二 | 保障高优 |
| 🟩 P2 | 算力池管理（自建+弹性云） | 阶段二 | GPU 纳管 |
| 🟩 P3 | 账单系统 + 内部结算 | 阶段三 | 精细化运营 |

### 关键里程碑

1. **地基就绪**：模型网关 + 统一调用入口 + 配额管理 + 基础看板。所有 AI 能力消费方接入。
2. **智能路由**：路由 Agent 上线，按任务自动选模，信创约束生效。
3. **降本增效**：成本优化 Agent + 缓存 + 弹性调度，成本节省 ≥ 25%。

> ⭐本板块是阶段一（P0）地基——没有统一算力与模型网关，所有 AI 能力板块都无法运转。必须最先就绪。
