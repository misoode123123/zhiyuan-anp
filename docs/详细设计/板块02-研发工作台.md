# 智源 ANP 平台 · 板块 2 · 研发工作台（详细设计）

> **文档定位**：AI 与技术人员协同写代码的**主战场**——AI 承担主体编码，人聚焦架构、难点、规范、评审。是研发主链路的**第二环**，消费需求工作台产出的需求规格，产出可测 PR 交付测试中心。
> **技术基线**：业务后端 Go（Gin + sqlx + zap）；架构设计/编码/评审/知识沉淀由 **Python 侧 LangGraph Agent** 承载，Go 经 **gRPC** 调用并做调度。GitLab 集成在 **Go 侧**（REST + Webhook）。持久化 PostgreSQL（schema: `dev`）+ MinIO（产物）+ Redis（Agent 上下文/任务队列）。
> **配套**：软件开发架构设计、开发标准与规范、总体方案 V1.0。
> **多租户**：所有数据带 `project_space_id` 作为路由键。

## 1. 定位与价值

### 1.1 定位

研发工作台是平台的**生产力核心**。它把"需求规格"变成"可测试的代码变更（PR）"。AI 担任主体编码，人不再陷在逐行写代码，而是聚焦架构决策、难点攻关、规范把关、代码评审。

```
需求规格(板块1)           研发工作台                    测试中心(板块3)
  │  confirmed 规格         │  架构→拆任务→编码→评审      │
  │ ──────────────────────▶ │ ──────────────────────────▶ │
  │  可行性回传              │  多Agent并行+G2/G3闸门      │
  │ ◀────────────────────── │ ◀────────────────────────── │
```

### 1.2 价值（可量化目标）

| 价值点 | 指标 | 18 个月目标 |
|---|---|---|
| 产能跃升 | 代码 AI 生成占比 | 50–60%（36 个月 70%+） |
| 质量保障 | PR 一次通过率（免返工） | ≥ 60% |
| 评审有效 | 评审缺陷发现率 | 高于纯人工评审 |
| 人效 | 人均交付故事点/人天 | 提升 1.5–2× |
| 规范落地 | 规则合规 PR 占比 | ≥ 95%（CI 强制） |
| 资产沉淀 | ADR/踩坑库增长率 | 持续上升 |

|---|---|---|---|

## 3. 功能模块

### 3.1 任务级 AI 编码（核心）

- 从需求规格拆出**原子研发任务**（DevTask），每个任务有明确输入/输出/验收。
- 给一个任务，编码 Agent 自主完成闭环：**读代码 → 理解上下文 → 改 → 写测试 → 自检 → 提 PR**。
- Agent 产出遵守五约束（可校验/可追溯/可回滚/守边界/守权限）：每次变更可单独 revert，不得跨模块改他域，不得跳闸门。

### 3.2 代码资产与 ADR

- **代码资产**：代码库索引、组件库、公共模块的元数据，供 Agent 检索复用。
- **ADR（架构决策记录）**：每次重大技术决策落 `docs/adr/NNNN-标题.md`，背景/决策/理由/后果；Agent 起草，人评审。
- 资产与 ADR 进版本管理（GitLab），有评审、有版本。

### 3.3 AI 代码评审

- PR 一创建，评审 Agent 自动跑：**Review + 规则校验 + 安全扫描**，产出评审报告。
- 交叉审查：模块 A 的 Agent 产出由模块 B 的评审 Agent 审（避免自审盲区）。
- 这是开发标准中"AI 预审 + 人审"双层评审的**AI 预审层**。

### 3.4 架构设计工作区

- 架构 Agent 起草方案（技术选型、模块划分、接口契约、数据模型、风险）。
- 人评审，重大变更过 G2。方案落 ADR + 草图（MinIO 存图）。
- 与板块4 架构标准对齐（读架构红线）。

### 3.5 技术知识沉淀

- 知识沉淀 Agent 从 PR/评审/踩坑中**抽取经验**，自动入库（最佳实践、坑、模式）。
- 沉淀的知识反哺编码/评审 Agent，越用越准（RAG 闭环）。

### 3.6 多 Agent 并行

- 多任务/多模块并行开发，每个编码 Agent 是独立实例，各自一个分支。
- 调度器（Go 侧）管：任务队列、实例数、算力配额、冲突检测（同文件并发改预警）。

### 4.1 主流程：从需求规格到可测 PR

```
┌──────────────────┐
│ 需求规格(板块1)   │ 事件 requirement.confirmed
└────┬─────────────┘

┌──────────────────┐

└────┬─────────────────────┘
     │

     └─ 高风险/核心模块 ─▶ 🚪 G3 代码人工审批 (L0)
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
              通过 ─▶ 合入主干         驳回 ─▶ 回编码Agent修

└──────────────────┘
```

### 4.2 单个编码 Agent 闭环（任务级）

```
[receive_task] ─▶ [scan_codebase] (检索代码资产/相关知识)
                       │
                       ▼
                 [implement] ──┬─▶ [write_tests] (覆盖正常/异常/边界)
                                │
                                ▼
                          [self_check] (lint/type/test 本地跑)
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
              全绿 ─▶ [create_pr]      失败 ─▶ [fix_loop] (≤ N 次)
                    │                            │
                    │◀───────────────────────────┘
                    ▼
              [hand_to_review] (评审Agent)
```

- **fix_loop 上限**：默认 ≤ 3 次自修复，仍失败则升级人工（不无限循环、不假装成功）。
- **守边界**：Agent 只改任务指定模块范围；跨模块需求拆成多任务。

### 4.3 GitLab 集成流程（Go 侧）

```
Go dev 模块 ──REST──▶ GitLab (建分支/提MR/查流水线)
GitLab ──Webhook──▶ Go dev 模块 (MR事件/流水线状态)
                          │
                          ▼
                   更新 PullRequest 状态 → 驱动评审/闸门
```

## 5. AI Agent 设计

> 所有 Agent 在 **Python 侧**（agent-runtime 服务，LangGraph 编排），Go 后端（`dev` 模块）经 gRPC 调用并做调度、配额、冲突检测。Agent 出站模型调用经模型网关（LiteLLM）。

### 5.1 Agent 清单

| Agent | 语言/运行时 | 职责 | Go 调用入口(gRPC) |
|---|---|---|---|
| 架构 Agent | Python/LangGraph | 出架构方案 + 草案 ADR + 接口契约 | `DesignArchitecture` |
| 编码 Agent（多实例） | Python/LangGraph + 外部工具 | 读码→改→写测试→自检→提 PR | `ExecuteCodingTask` |
| 评审 Agent | Python/LangGraph | 交叉审查 + 规则校验 + 安全扫描 | `ReviewPullRequest` |
| 知识沉淀 Agent | Python/LangGraph | 从 PR/评审抽取经验入库 | `DistillKnowledge` |

### 5.2 编码 Agent 的可插拔设计

编码 Agent 支持**多后端**，可接 Claude Code / Cursor / Devin 类工具，统一抽象：

```
CodingAgentInterface:
  - plan(task) -> steps
  - edit(steps) -> diff
  - test(diff) -> result
  - self_check(result) -> pass|fail
```

| 后端类型 | 适配方式 | 适用 |
|---|---|---|
| 内置 LangGraph | 平台自研编排 | 通用、可控 |
| Claude Code 类 | 工具适配器（Agent SDK） | 复杂推理编码 |
| Cursor 类 | 工具适配器 | IDE 协同场景 |
| Devin 类 | 工具适配器（沙箱） | 长时自主任务 |

> 编码工具"做薄"——可替换，平台护城河在调度/规则/知识/闸门，不在某个编码工具。

### 5.3 评审 Agent 的交叉审查

- 评审 Agent 与编码 Agent **实例隔离**：评审者未参与被审 PR 的编码，避免自审盲区。
- 评审维度：规范符合度、模块边界、测试覆盖、规则合规、安全扫描、性能、可维护性。
- 产出评审报告 + 结构化缺陷项（阻断/建议/疑问）。

### 5.4 gRPC IDL 概要（dev_agent.proto）

```protobuf
service DevAgent {
  rpc DesignArchitecture(ArchReq) returns (ArchResp);
  rpc ExecuteCodingTask(TaskReq) returns (stream TaskProgress); // 流式进度
  rpc ReviewPullRequest(ReviewReq) returns (ReviewResp);
  rpc DistillKnowledge(DistillReq) returns (DistillResp);
}
// 请求必带 project_space_id + trace_id + caller(标识Go模块)
```

### 5.5 调度器（Go 侧 `dev` 模块）

| 职责 | 说明 |
|---|---|
| 任务队列 | Redis 队列，按优先级/依赖调度 |
| 实例池 | 编码 Agent 实例数受算力配额约束（板块8） |
| 冲突检测 | 同文件/同模块并行改预警，必要时串行化 |
| 超时与熔断 | 单任务超时、失败重试上限、熔断降级为人工任务 |
| trace 透传 | Go→Python 同 traceId，跨语言可观测 |

## 8. 规则与约束

> 本板块**读**规则治理中心（板块4）的规则资产，是规则消费大户。

### 8.1 知识资产消费清单

| 资产类型 | 内容 | 用途 | 强制度 |
|---|---|---|---|
| Rule | 编码规范（Go/Python/TS） | 编码 Agent 遵循 + 评审校验 | 🟥 强制 |
| Rule | 架构标准/红线 | 架构方案校验 + G2 依据 | 🟥 强制 |
| Rule | 安全红线 | 评审安全扫描 + 编码约束 | 🟥 强制 |
| Rule | 依赖白名单 | SCA 校验，非白名单依赖阻断 | 🟥 强制 |
| Rule | 模块边界 | 守边界，跨模块改预警 | 🟥 强制 |
| Knowledge | 组件库/代码资产 | 编码 Agent 复用检索 | 🟧 应遵循 |
| Knowledge | 踩坑库/最佳实践 | 编码 Agent RAG + 评审参考 | 🟧 应遵循 |

### 8.2 规则校验点（CI 门禁 + 评审 Agent）

每个 PR 的 CI 必跑（开发标准第 11 节）：

1. lint / format / type-check
2. 单元测试 + 覆盖率门禁（≥ 80%，核心 ≥ 90%）
3. 安全扫描（密钥/依赖/SAST）
4. **规则合规校验**（调规则引擎）
5. API 契约一致性检查
6. 构建镜像 + 冒烟

任一失败阻断合入；评审 Agent 复用这些结果 + LLM 复核。

## 9. 人机闸门

### 9.1 🚪 G2 架构评审（L0，影响大系统必过）

| 项 | 内容 |
|---|---|
| 触发 | 架构 Agent 出方案 → 提交 G2 |
| 判定范围 | 是否影响大系统（跨服务/数据模型变更/新技术引入/公共接口） |
| 决策者 | 架构师（拍板方案）+ 规则架构师（红线） |
| AI 干的 | 起草方案 + 草案 ADR + 风险识别 |
| 人干的 | 评审方案合理性、红线合规、技术风险 |
| 简化路径 | 小改动/同模式变更可简化（轻量 G2 或免审，由规则判定） |
| 产出 | `approved` 方案 + ADR accepted + 启动任务拆解 |

### 9.2 🚪 G3 代码审批（L0-L1，分级）

| 路径 | 条件 | 决策 |
|---|---|---|
| 自动合入候选（L1） | 低风险 + AI 预审全绿 + 规则合规 + 测试达标 + 非核心模块 | 评审 Agent 批准后自动合入候选（可配人工观察期） |
| 人工审批（L0） | 高风险 / 核心模块 / 触发红线提示 / 涉及安全敏感 | 至少 1–2 人审，人拍板 |

| 项 | 内容 |
|---|---|
| AI 干的 | 评审报告、缺陷定位、修复建议 |
| 人干的 | 判断是否可合入、是否需返工 |
| 产出 | `merged` 或 `rejected`（驳回回编码 Agent 修复） |
| 留痕 | 合入人、依据规则、模型版本入 audit |

> G3 是"AI 预审 + 人审"双层评审的核心落地点。**谁采纳谁负责**——人工合入的 PR 由合入人承担最终责任；Agent 建议不构成授权。

### 9.3 禁止事项（平台 AI 协作规范落地）

- ❌ Agent 直接合入高风险 PR（必过 G3）。
- ❌ Agent 跨模块改他域（守边界）。
- ❌ Agent 跳过 CI/规则校验。
- ❌ Agent 一次性"大爆炸"提交（每次变更可单独 revert）。

## 12. 落地优先级

| 优先级 | 范围 | 阶段 | 说明 |
|---|---|---|---|
| 🟥 P0 | GitLab 集成 + 任务拆解 + 编码 Agent（单后端） + AI 评审 + G3 | 阶段二核心 | 最小可用闭环：规格→PR |
| 🟥 P0 | G2 架构评审 + ADR | 阶段二 | 大系统变更必须 |
| 🟧 P1 | 多 Agent 并行调度 + 冲突检测 | 阶段二 | 提速 |
| 🟧 P1 | 知识沉淀 Agent（RAG 闭环） | 阶段二末 | 越用越准 |
| 🟩 P2 | 多编码后端可插拔（Claude Code/Cursor/Devin） | 阶段二末–三 | 工具可替换 |
| 🟩 P2 | 架构设计工作区增强（草图/契约联动） | 阶段三 | 体验增强 |

### 关键里程碑

1. **MVP**：需求规格（板块1 G1 通过）→ 架构 Agent 出方案 → G2 → 拆任务 → 编码 Agent 出 PR → AI 评审 → G3 → 合入 → 触发测试中心（板块3）。
2. **闭环**：多任务并行 + 评审 Agent 交叉审查 + 知识反哺。
3. **增强**：多编码后端可插拔 + 知识沉淀 RAG 闭环。

> 与板块1的握手是 `requirement_id + spec_version`；与板块3的握手是 `pr_merged` 事件。编码工具可替换，平台护城河在调度/规则/知识/闸门。
