# 智源 ANP 平台 · 软件开发架构设计（详细设计）

> **文档定位**：定义平台的软件架构——架构原则、技术选型、模块/服务划分、数据架构、部署架构、集成、安全、可观测、演进路线。是所有板块详细设计的**技术基线**。
> **技术基线**：**Go（平台业务后端）+ Python（AI/Agent 运行时）混合**，两层经 gRPC 解耦。
> **配套**：总体方案 V1.0、开发展开计划、开发标准与规范。

---

## 1. 架构目标与原则

**目标**：可扩展、模型无关、多租户安全、高可用、全链路可观测、可持续演进。

| 原则 | 含义 |
|---|---|
| 分层解耦 | Go 业务后端与 Python AI 运行时经 gRPC 分离，各自演进 |
| 模块化单体起步 | Go 单部署单元、内部按域强隔离；复杂度达标再拆服务 |
| 模型/算力解耦 | 所有模型调用经网关，模型与 GPU 可平滑替换 |
| 契约先行 | API 用 OpenAPI、跨语言用 gRPC IDL，前后端与 Agent 依契约 |
| 规则内嵌 | 治理（规则、闸门、审计）是一等公民，非旁路 |
| 可插拔 | Agent、模型、工具、流水线步骤皆可替换 |
| 零信任 + 可观测 | 默认拒绝、最小权限；日志/指标/链路全覆盖 |
| 演进优于完美 | 架构随阶段演进，每步可回退 |

## 2. 总体逻辑架构（软件模块视角）

```
┌──────────────────────────────────────────────────────────────┐

├──────────────────────────────────────────────────────────────┤
│  Go 模块化单体 backend（Gin，按域分 package）                   │
│  ┌──────────┐┌──────────┐┌──────────┐┌──────────┐┌──────────┐ │
│  │workspace ││requirement││  dev     ││  rule    ││  gate    │ │
│  │ 项目空间 ││ 需求     ││ 研发编排 ││ 规则引擎 ││ 闸门引擎 │ │
│  └──────────┘└──────────┘└──────────┘└──────────┘└──────────┘ │

        ▼                   ▼                   ▼
```

## 3. 技术架构（选型分层）

| 层 | 选型 | 备注 |
|---|---|---|
| 前端 | Next.js + TypeScript + Tailwind + shadcn/ui | 工作台 SPA，统一组件库 |
| **平台业务后端** | **Go** + Gin + go-playground/validator + sqlx(或 GORM) + viper + zap | 模块化单体；高并发骨干 |
| **AI/Agent 运行时** | **Python** 独立微服务 + LangGraph + LiteLLM/模型 SDK | AI 生态；经 gRPC 供 Go 调用 |
| 跨语言通信 | gRPC（IDL）+ HTTP/REST | Go↔Python、前端↔Go |
| 关系数据 | PostgreSQL（多 schema） | 每域一个 schema |
| 向量数据 | pgvector（起步）→ Milvus | 起步零额外组件 |
| 缓存/队列 | Redis +（演进）Kafka/NATS | 起步 Redis 即够 |
| 对象存储 | MinIO（自建）/ S3 兼容 | 文档、产物、日志 |
| 代码托管/CI | GitLab 自托管 | 仓库+流水线+评审一体 |
| 容器/编排 | Docker Compose（起步）→ Kubernetes | 渐进 |
| 可观测 | Prometheus + Loki + Tempo + Grafana | 指标/日志/链路（trace 跨 Go/Python） |
| 密钥 | Vault / SOPS | 密钥不落代码 |

> **为什么 Go + Python 混合**：Go 拿到高并发、强类型、部署简单（单二进制）；Python 拿到不可替代的 AI 生态（LangGraph/LiteLLM/SDK/RAG）。两者解耦，业务迭代用 Go 的效率，AI 迭代用 Python 的生态，互不拖累。
>
> 信创：模型换国产（通义/文心/DeepSeek/Qwen/智谱）；数据库可选国产 PG 兼容（openGauss）；组件均私有化可部署。

## 4. 应用架构与服务划分

### 4.1 策略：Go 模块化单体 + Python AI 独立服务

- **Go 后端**：单个部署单元，按**限界上下文**分 package；package 间只通过内部接口/事件通信，禁止跨包直接查库。
- **Python AI 运行时**：独立部署的微服务，承载 Agent 编排与模型调用；Go 经 gRPC 调用。
- **拆分信号**：某 Go 模块在负载/发布节奏/团队归属上显著分化时，拆为独立 Go 服务。

### 4.2 模块/服务清单

| 模块/服务 | 语言 | 限界上下文 | 核心职责 |
|---|---|---|---|
| `workspace` | Go | 项目空间 | 项目空间 CRUD、成员、配额 |
| `requirement` | Go | 需求 | 需求池、规格、澄清记录 |
| `dev` | Go | 研发编排 | 任务拆解、编码调度、PR 元数据 |
| `rule` | Go | 规则 | 规则 CRUD、结构化、校验、阻断 |
| `gate` | Go | 闸门 | 闸门定义、审批流、决策留痕 |
| `pipeline` | Go | 流水线 | 阶段编排、对接 GitLab CI |
| `knowledge` | Go + Python | 知识 | Go 管元数据；Python 做 RAG/向量 |
| `audit` | Go | 审计 | 全链路留痕、合规追溯 |
| `observability` | Go | 可观测 | 指标/日志/链路聚合 |
| `agent-runtime` | **Python** | Agent 运行时 | Orchestrator、LangGraph、工具市场、记忆 |
| `model-gateway` | **Python**(LiteLLM) | 模型网关 | 模型路由、成本、限流、故障切换 |

### 4.3 通信规则

- **Go 包内**：调用内部 Service 接口（不直连他域 DB）。
- **Go ↔ Python**：gRPC（IDL 契约），同步调用 + 异步回调。
- **跨域副作用**：领域事件（Redis Stream 起步，演进到消息中间件）。

## 5. 数据架构

### 5.1 所有权与隔离

- 每个限界上下文**拥有自己的 schema**（起步同一 PG 实例）。
- 跨域查询走 API/事件，**禁止跨 schema JOIN**。
- `project_space_id` 是贯穿所有域的**多租户路由键**。

### 5.2 存储选型矩阵

| 数据类型 | 存储 | 示例 |
|---|---|---|
| 事务型业务数据 | PostgreSQL | 需求、规则、项目空间、闸门 |
| 向量/语义检索 | pgvector / Milvus | 知识库、需求查重 |
| 文档/产物 | MinIO | 上传文档、生成产物、日志归档 |
| 缓存/会话/队列 | Redis | Agent 上下文、任务队列 |
| 时序指标 | Prometheus | 算力、成本、SLO |

### 5.3 数据流（一次需求的关键路径）

```
requirement(规格) ──事件──▶ dev(任务) ──事件──▶ pipeline(编码/测试)
      │                          │                      │
      └──── rule 校验 ◀──────────┴──── gate 决策 ◀──────┘
                            │
                      audit 留痕（全链路，跨 Go/Python 同 traceId）
```

## 6. 部署架构

### 6.1 环境分层

| 环境 | 用途 | 部署 |
|---|---|---|
| dev | 开发联调 | 本地 / Compose |
| test | 自动化测试 | Compose / 单节点 K8s |
| staging | 预发布（数据脱敏） | K8s |
| prod | 生产 | K8s 多副本 |

### 6.2 部署拓扑（生产）

```
            入口（LB / Ingress）
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
   前端（CDN）         API 网关(Go)
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
      Go backend       agent-runtime   model-gateway
      (多副本)         (Python)        (Python/LiteLLM)
            │             │             │
        ┌───┴───┐     ┌───┴───┐         │
        ▼       ▼     ▼       ▼         ▼
      PG主从  Redis  向量库  MinIO    模型API/GPU池
```

### 6.3 CI/CD

- 代码 → GitLab CI（lint/test/build/scan）→ 镜像（Go 二进制 + Python 镜像）→ staging 自动部署 → 🚪G5 审批 → prod 灰度。
- 平台自身发布也走自己的闸门（dogfooding）。

## 8. 安全架构

- **认证**：OIDC/SSO；服务间 mTLS（演进期）。
- **授权**：RBAC（角色）+ ABAC（项目空间、数据分级），默认拒绝。
- **密钥**：Vault/SOPS 管理，运行时注入，不落仓。
- **数据安全**：敏感字段脱敏、最小权限、AI 对话防泄密（出站过滤）。
- **审计**：所有 AI 产出与人工决策落 `audit`，不可篡改、可追溯。
- **零信任**：每个请求都鉴权，Go/Python 服务间不默认信任。

## 9. 可观测架构

| 维度 | 工具 | 内容 |
|---|---|---|
| 指标 | Prometheus + Grafana | QPS、延迟、Agent 成功率、算力/成本 |
| 日志 | Loki | 结构化日志，按 trace 关联 |
| 链路 | Tempo | **跨 Go/Python** 调用链（同 traceId 透传） |
| 告警 | Alertmanager → 闸门/运维 | 异常进研判 Agent |
| 业务看板 | 自研 | 交付周期、AI 占比、规则覆盖率 |

**Trace 贯穿**：一次需求从澄清到上线，Go 与 Python 共享同一 traceId，便于审计与根因分析。

## 10. 高可用与扩展

- **无状态服务**：Go backend、Python agent-runtime/model-gateway 水平扩缩。
- **状态数据**：PG 主从、Redis 哨兵/集群、向量库副本。
- **弹性**：K8s HPA；模型网关按配额限流。
- **降级**：模型不可用自动切备用；Agent 失败降级为人工任务；Python 运行时故障时 Go 后端可降级直连模型或转人工。
- **容灾**：关键数据定期备份 + 跨可用区（演进期）。

## 11. 架构演进路线

| 阶段 | 形态 | 触发 |
|---|---|---|
| 阶段一 | Go 模块化单体 + Python agent-runtime/gateway 独立服务 | MVP |
| 阶段二 | 高负载 Go 域拆服务（如 gateway Go 化、knowledge 独立） | 负载/团队分化 |
| 阶段三 | 全面服务化 + 服务网格 + 多区容灾 | 规模化 |

## 12. 架构决策记录（ADR）

- **ADR-001 起步采用 Go 模块化单体**：理由=避免过早分布式、单二进制部署简单；后果=需严格 package 边界纪律。
- **ADR-002 模型调用统一走网关**：理由=模型无关、成本可控；后果=多一跳延迟（可接受）。
- **ADR-003 规则与闸门作为一等模块**：理由=治理内嵌、可审计；后果=所有 Agent 产出必须过校验。
- **ADR-004 平台业务后端选用 Go**：理由=高并发/强类型/部署简单/性能；后果=AI 生态弱（见 ADR-005）。
- **ADR-005 AI/Agent 运行时用 Python 独立服务 + gRPC 解耦**：理由=LangGraph/LiteLLM/SDK 生态不可替代；后果=多一个运行时（用 gRPC 契约 + trace 透传治理）。

> 所有 ADR 存于 `docs/adr/`，重大架构变更必须先有 ADR。
