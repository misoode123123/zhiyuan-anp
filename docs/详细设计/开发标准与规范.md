# 智源 ANP 平台 · 开发标准与规范（详细设计）

> **文档定位**：定义平台自身（及平台上所有项目）的开发标准——编码、分支、提交、评审、测试、API、文档、安全、发布，以及平台独有的 **AI 协作规范**。
> **配套**：软件开发架构设计、开发展开计划。

---

## 1. 标准体系总览

| 类别 | 规范 | 强制度 |
|---|---|---|
| 编码 | 语言规范 + 项目约定 | 🟥 强制（CI 门禁） |
| 版本 | 分支 + 提交信息 | 🟥 强制 |
| 评审 | AI 预审 + 人审清单 | 🟥 强制 |
| 测试 | 覆盖率 + 必测场景 | 🟥 强制门禁 |
| API | OpenAPI 契约 + 版本 | 🟥 强制 |
| 文档 | ADR + 模块文档 | 🟧 应遵循 |
| 安全 | 安全编码 + 密钥 + 脱敏 | 🟥 强制 |
| 发布 | 变更 + 闸门 + 回滚 | 🟥 强制 |
| **AI 协作** | 人机分工 + Agent 约束 + prompt/规则版本 | 🟥 强制（平台特色） |

## 2. 编码规范

### 2.1 通用

- 格式化工具入 CI，**不允许"格式之争"进评审**。
- 命名清晰优于注释；注释解释"为什么"，不解释"是什么"。
- 函数/模块单一职责；圈复杂度上限（建议 ≤ 10）。
- 不留死代码、TODO 必带 issue 号。

### 2.2 Go（平台业务后端）

- `golangci-lint` 检查 + `gofmt`/`goimports` 格式化；强类型。
- 数据校验用 `go-playground/validator`（struct tag）；Gin 路由薄、逻辑在 Service 层。
- 错误显式处理（`error` 返回）；`context.Context` 贯穿用于超时/取消/链路透传。
- 并发优先 `errgroup`/goroutine + channel；避免 goroutine 泄漏。
- package 边界纪律：跨域只调 Service 接口，不直连他域 schema。

### 2.3 Python（AI/Agent 运行时）

- `ruff` + `black` 格式化；类型注解（`pyright`）。
- 仅用于 Agent 编排（LangGraph）、模型调用（LiteLLM/SDK）、RAG 等 AI 计算；作为**独立微服务**，与 Go 后端经 **gRPC** 通信，不承载业务事务。
- 模型/Prompt 调用必须经网关，不直连供应商。

### 2.4 TypeScript（前端）

- `eslint` + `prettier`；`strict` 模式。
- 组件统一 shadcn/ui；状态管理统一方案；API 调用统一经生成客户端。

### 2.4 SQL/数据

- 迁移用工具（如 Alembic）版本化，**禁止手改库**。
- 所有表带 `project_space_id`（多租户路由键）、审计字段（created/updated/by）。

## 3. 分支与提交规范

### 3.1 分支策略：Trunk-Based（主）

- 主干 `main` 始终可发布；短分支（≤ 2 天）+ 频繁合入。
- 大特性用 feature flag 解耦发布与合并。
- 里程碑版本打 tag。

### 3.2 提交信息：Conventional Commits

```
<type>(<scope>): <subject>

<body>

<footer>
```

- type：`feat / fix / refactor / test / docs / chore / perf`
- scope：模块名（如 `rule`、`requirement`）
- 破坏性变更：`BREAKING CHANGE:` 或 type 后加 `!`
- 关联：footer 写 `Refs: #需求号`。

### 3.3 提交粒度

- 一个提交一件事；可被 `revert` 单独回退。
- AI 生成的提交需标注 `Co-authored-by: <agent>`。

## 4. 代码评审标准

### 4.1 双层评审：AI 预审 + 人审

1. **AI 预审（强制）**：PR 一经创建，评审 Agent 自动检查规范、安全、规则合规、潜在缺陷，产出评审报告。
2. **人审（分级）**：低风险 PR 经 AI 预审通过可自动合入（🚪L1）；高风险/核心模块至少 1–2 人审（🚪G3）。

### 4.2 评审清单（Checklist）

- [ ] 符合编码规范与模块边界
- [ ] 有测试且覆盖关键路径与异常
- [ ] 不违反规则库（安全/合规/架构红线）
- [ ] 无硬编码密钥/配置
- [ ] API 变更有契约、有版本
- [ ] 性能无明显退化
- [ ] 必要时附 ADR

> 评审反馈对事不对人；AI 评审意见同样需人确认后再改。

## 5. 测试标准

### 5.1 测试金字塔

```
        /\
       /UI\          少（关键链路自动化）
      /----\
     / 集成 \         中（模块间契约）
    /--------\
   /   单元    \      多（覆盖率 ≥ 80%）
  /____________\
```

### 5.2 门禁

- 单元覆盖率 ≥ 80%（核心模块 ≥ 90%），CI 强制。
- 关键路径必有**异常与边界用例**（AI 生成时强制覆盖）。
- PR 必须全绿方可合入；失败即阻断（硬质量门）。

---

## 6. API 契约规范

- **契约先行**：先写 OpenAPI，再实现；前端/Agent 客户端由契约生成。
- **版本**：URL 前缀 `/api/v1`；破坏性变更升版本，旧版灰度下线。
- **命名**：RESTful 资源命名；字段 `snake_case`（后端）/ 输出层按需转换。
- **错误**：统一错误模型 `{code, message, trace_id}`，HTTP 语义正确。
- **幂等**：写操作支持幂等键。

---

## 7. 文档规范

- **ADR**：架构决策存 `docs/adr/NNNN-标题.md`，格式：背景/决策/理由/后果。
- **模块文档**：每个模块 README，含职责、接口、依赖、负责人。
- **API 文档**：由 OpenAPI 自动生成。
- **变更日志**：CHANGELOG.md 按版本记录。

> 文档与代码同生命周期；过时文档视为缺陷。

---

## 8. 安全编码规范

- 密钥/凭证经 Vault/SOPS，**永不入库**；CI 扫密钥泄漏。
- 输入校验（Go: validator struct tag；Python AI 运行时: pydantic）、输出编码、参数化查询（禁拼接 SQL）。
- 最小权限：服务账号仅授必要权限。
- 依赖白名单 + 漏洞扫描（SCA）；高危依赖阻断。
- 日志不打印敏感数据（脱敏）。

---

## 9. 发布与变更规范

- 生产变更必经 🚪G5（上线审批）；高风险升级 🚪L3 双确认。
- 每次发布有**变更单 + 回滚预案 + 检查清单**（AI 起草、人确认）。
- 灰度发布 + 指标守护；异常自动暂停/回滚。
- 变更窗口规则由规则治理中心下发。
- 对外发布、资金操作、生产数据变更属高风险（见闸门等级）。

---

## 10. AI 协作规范（平台特色 ⭐）

> 本平台自身由 AI 驱动开发，故必须定义"人与 Agent 如何协同"。本节是普通研发标准中不存在的、平台独有规范。

### 10.1 人机分工矩阵

| 环节 | Agent 干 | 人把关 |
|---|---|---|
| 需求 | 澄清、起草规格、查重、估时 | 范围/优先级/预算（G1） |
| 架构 | 起草方案、ADR | 架构评审（G2） |
| 编码 | 写代码、写测试、自检 | 核心模块/高风险审（G3） |
| 评审 | 自动预审、规则校验 | 采纳与否 |
| 发布 | 起草变更单、灰度执行 | 上线审批（G5） |
| 运维 | 巡检、研判、低风险自愈 | 高风险操作（G6） |

### 10.2 Agent 产出五项约束（必须）

1. **可校验**：产出必须能被测试/规则校验，否则不交付。
2. **可追溯**：标注依据了哪些规则、参考了哪些知识、用了哪个模型版本。
3. **可回滚**：每次变更可单独 revert；禁止"一次性大爆炸"提交。
4. **守边界**：不得跨越模块边界直接改他域；不得跳过闸门。
5. **守权限**：不得访问无权限数据、不得直接动生产、不得绕过审计。

### 10.3 责任归属

- **谁采纳谁负责**：人工合入/批准的 Agent 产出，由批准人承担最终责任。
- Agent 不得作为责任主体；Agent 建议不构成授权。

### 10.4 Prompt 与规则即资产

- Prompt、规则、Agent 配置**同代码一样进版本管理**（Git），有评审、有版本。
- 规则变更走规则治理中心生命周期（草稿→评审→发布）。
- Prompt 变更附评测结果（回归集），防"改一处退一片"。

### 10.5 透明度

- Agent 必须说明：本次产出**依据哪些规则**、**参考哪些知识**、**置信度如何**。
- 不确定时主动声明并升级人工，禁止"假装确定"。

### 10.6 禁止事项

- ❌ Agent 直接合入高风险 PR（必须过 G3）。
- ❌ Agent 直接发布生产（必须过 G5）。
- ❌ Agent 执行高风险运维操作（必须过 G6）。
- ❌ Agent 访问与任务无关的敏感数据。
- ❌ 删除审计记录。

---

## 11. 质量门禁（CI 检查清单）

每个 PR 的 CI 必跑：

1. lint / format / type-check
2. 单元测试 + 覆盖率门禁
3. 安全扫描（密钥/依赖/SAST）
4. 规则合规校验（调规则引擎）
5. API 契约一致性检查
6. 构建镜像 + 冒烟

全绿方可合入；任一失败阻断。

---

## 12. 标准的演进

- 本标准本身进版本管理，变更经评审。
- 新规则/新禁令由规则治理中心下发后，自动并入相应门禁。
- 定期回顾：哪些标准被频繁绕过 → 要么强化，要么修订。

---

*本规范适用于平台自身及平台上所有项目；项目空间可叠加更严规则，不得放宽全局红线。*
