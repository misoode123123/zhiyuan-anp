# 板块03 · 测试与质量中心（详细设计）

> **文档定位**：定义平台测试与质量中心的完整设计——从"人写人跑"转向"AI 生成 / 自动执行 / 持续回归"，质量左移、内嵌流水线。覆盖用例自动生成、自动化执行、覆盖率与缺口分析、缺陷智能分析、测试数据脱敏、回归智能选取。
>
> **技术基线**：业务后端 **Go**（Gin + sqlx + zap）；Agent 编排与模型调用在 **Python** 侧（LangGraph + LiteLLM），Go 经 **gRPC** 调用。
>
> **配套**：软件开发架构设计、开发标准与规范（§5 测试标准）、板块02-研发、板块04-规则治理中心、板块06-发布中心。
>
> **核心原则**：🟥 **测试失败 = 自动阻断**（硬质量门），PR 全绿方可合入。

---

## 1. 定位与价值

### 1.1 问题

传统测试：人写用例、人跑回归、覆盖率靠感觉、缺陷靠经验、测试数据靠拷库。结果是：测试滞后于开发、回归跑不全、缺陷漏测、测试数据含真实敏感信息。

### 1.2 定位

测试与质量中心将测试从**阶段末尾的验证活动**前移为**贯穿研发的持续质量保障**：

| 转变 | 从 | 到 |
|---|---|---|
| 用例来源 | 人写 | AI 生成（正常+异常+边界）+ 人审核 |
| 执行时机 | 手动/定期 | CI 触发 + PR 触发 + 定时全量 |
| 回归范围 | 全量或拍脑袋 | 按变更影响面智能选取 |
| 覆盖率 | 只看百分比 | 反向指出未测路径与缺口 |
| 缺陷处理 | 新建工单流转 | AI 根因分析 + 去重 + 智能指派 |
| 测试数据 | 拷贝生产 | 合成/脱敏数据，按分级管控 |

### 1.3 价值

- **左移**：在 PR 阶段就阻断缺陷，而非到测试环境才发现。
- **降本**：回归选取减少 50%+ 无关用例执行。
- **增效**：AI 生成异常/边界用例，覆盖人容易遗漏的场景。
- **可量化**：覆盖率、缺口、缺陷密度全量可视。

## 2. 用户与角色

| 角色 | 职责 | 自治等级 |
|---|---|---|
| 开发工程师 | 写单测、修复失败用例、确认缺口 | L2（可触发局部回归） |
| 测试工程师 | 审核AI用例、管理测试计划、分析缺陷 | L2 |
| 质量负责人（QA Lead） | 制定测试策略、质量门禁配置、质量报告 | L2 |
| 项目经理 | 查看质量看板、质量风险决策 | L1（只读+审批） |
| AI 用例生成Agent | 自动生成用例草案 | L1（产出需人确认） |
| AI 执行编排Agent | 编排测试执行、收集结果 | L2（CI 触发，自动跑） |
| AI 根因分析Agent | 分析失败根因、去重、指派建议 | L1（建议需人确认） |

> 自治等级：L0 纯人工 → L1 AI 建议人确认 → L2 AI 执行人监督 → L3 AI 自主人监督。

## 5. AI Agent 设计

> 所有 Agent 在 **Python 侧**（LangGraph 编排），Go 后端经 **gRPC** 调用。每个 Agent 产出须满足五项约束（可校验/可追溯/可回滚/守边界/守权限）。

### 5.1 用例生成 Agent（TestCaseGenAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `TestCaseGenService.Generate(req) → resp` |
| **输入** | 需求规格、接口契约(OpenAPI)、代码diff、已有用例(去重) |
| **输出** | `[]TestCaseDraft`（含 title/steps/expected/type/category/priority/rationale） |
| **工具** | 静态分析器（AST解析）、契约解析器、知识RAG（测试模式库）、模型推理 |
| **编排图** | `StateGraph`: 分析输入 → 推导场景 → 分类(正常/异常/边界) → 去重 → 打分排序 → 输出 |
| **模型** | 经 LiteLLM 网关，默认高能力模型（场景推导）；轻量去重用小模型 |
| **约束** | 每条用例必须标注 `rationale`（为什么测这个）；必须覆盖异常+边界（硬约束） |
| **自治等级** | L1：产出为草案，需人审核确认 |

```
LangGraph 编排（用例生成）:

   ┌─────────┐     ┌──────────┐     ┌──────────┐     ┌─────────┐
   │ 输入分析 │────▶│ 场景推导  │────▶│ 分类展开  │────▶│ 去重排序 │
   │ 契约+代码│     │ 模型推理  │     │正常/异常/ │     │ vs已有  │
   │ 需求规格 │     │          │     │边界       │     │ 用例库  │
   └─────────┘     └──────────┘     └──────────┘     └────┬────┘
                                                          │
                                                          ▼
                                                    ┌──────────┐
                                                    │ 输出草案集│
                                                    │ +rationale│
                                                    └──────────┘
```

### 5.2 执行编排 Agent（TestExecutionAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `TestExecutionService.Execute(req) → stream resp` |
| **输入** | 用例集（或回归选取结果）、环境信息、执行策略 |
| **输出** | 每个 TestCase 的执行结果（pass/fail/skip/block + 日志 + 产物） |
| **工具** | 测试执行器（go test / pytest / playwright）、并行调度器、结果收集器 |
| **编排图** | `StateGraph`: 解析策略 → 分片 → 并行执行 → 结果汇总 → Flaky检测 → 输出 |
| **自治等级** | L2：CI 触发后自动执行，人监督 |

### 5.3 根因分析 Agent（RootCauseAgent）

| 属性 | 说明 |
|---|---|
| **语言/框架** | Python / LangGraph |
| **Go 调用入口** | gRPC `RootCauseService.Analyze(req) → resp` |
| **输入** | 失败用例列表、堆栈日志、代码diff、历史缺陷库 |
| **输出** | `[]RootCauseReport`（root_cause / dedup_hash / similar_defects / suggested_assignee / fix_suggestion） |
| **工具** | 日志分析器、堆栈解析器、blame 分析、向量检索（历史缺陷相似度）、模型推理 |
| **编排图** | `StateGraph`: 收集失败上下文 → 堆栈解析 → 相似检索(去重) → blame定位 → 根因推断 → 指派建议 |
| **自治等级** | L1：分析报告为建议，缺陷创建/指派需人确认 |

### 5.4 gRPC 接口定义（IDL 摘要）

```protobuf
// agent_runtime/proto/testing.proto

service TestCaseGenService {
  rpc Generate(TestCaseGenRequest) returns (TestCaseGenResponse);
}

service TestExecutionService {
  rpc Execute(ExecutionRequest) returns (stream ExecutionEvent);
}

service RootCauseService {
  rpc Analyze(RootCauseRequest) returns (RootCauseResponse);
}

message TestCaseGenRequest {
  string project_space_id = 1;
  string source_type      = 2;  // "requirement" | "contract" | "diff" | "code"
  string source_ref       = 3;  // 需求ID / OpenAPI URL / commit_sha / 文件路径
  string test_type_hint   = 4;  // "unit" | "integration" | "api" | "e2e"
  int32  max_cases        = 5;
}

message TestCaseGenResponse {
  string  agent_run_id     = 1;
  string  model_version    = 2;
  float   confidence       = 3;
  repeated TestCaseDraft cases = 4;
  repeated string rules_applied = 5;  // 引用了哪些测试规则
}

message TestCaseDraft {
  string  title       = 1;
  string  description = 2;
  string  test_type   = 3;   // unit / integration / api / e2e
  string  category    = 4;   // normal / exception / boundary
  string  priority    = 5;   // p0 / p1 / p2 / p3
  string  steps       = 6;   // JSON 或自然语言步骤
  string  expected    = 7;
  string  rationale   = 8;   // 为什么测这个（五约束：可追溯）
  string  pre_condition = 9;
}
```

## 8. 规则与约束

> 规则来源：板块04-规则治理中心。本中心是规则**消费者**。

### 8.1 规则消费清单

| 规则ID | 类型 | 内容 | 强制度 |
|---|---|---|---|
| RULE-TEST-001 | Rule（强制） | 所有新接口必须有异常+边界用例 | 🟥 强制（门禁） |
| RULE-TEST-002 | Rule（强制） | 单元覆盖率 ≥ 80%（核心模块 ≥ 90%） | 🟥 强制（门禁） |
| RULE-TEST-003 | Rule（强制） | P0用例失败即阻断PR | 🟥 强制 |
| RULE-TEST-004 | Process（应遵循） | AI 生成用例必须经人审核方可入库 | 🟧 应遵循 |
| RULE-TEST-005 | Process（应遵循） | Flaky 用例连续3次不稳定自动隔离 | 🟧 应遵循 |
| RULE-TEST-006 | Knowledge（参考） | 常见测试模式库（注入/超时/并发/编码） | 📘 参考 |
| RULE-TEST-007 | Rule（强制） | 测试数据不得使用生产真实数据（须脱敏/合成） | 🟥 强制 |

### 8.2 质量门判定逻辑

```
  质量门(QualityGate) = AND(
      所有P0用例通过,
      所有P1用例通过率 == 100%,
      行覆盖率 ≥ 80% (核心≥90%),
      无新增 critical 缺陷,
      新接口有异常+边界用例,
      CI 全部步骤通过
  )

  任一条件不满足 → 🟥 gate_passed = false → 阻断PR合入
```

| `test.defect.created` | 出 | defect_id, severity, module_id |
| `test.coverage.updated` | 出 | commit_sha, line_coverage, meets_gate |
| `pr.merged` / `pr.updated` | 入 | pr_id, commit_sha, changed_files |
| `rule.test.updated` | 入 | rule_id, rule_content |

## 11. KPI

| KPI | 定义 | 目标 | 说明 |
|---|---|---|---|
| 单元覆盖率 | 被测代码行/总代码行 | ≥80%（核心≥90%） | 行覆盖率，CI 门禁 |
| 测试通过率 | 通过用例/执行用例 | ≥99%（P0/P1=100%） | 按优先级分层 |
| PR 阻断率 | 被质量门阻断的PR比例 | 5%~15% | 太低=门禁松；太高=质量差 |
| 回归选取缩减率 | (1-选取用例/全量) | ≥50% | 不影响漏测率前提下 |
| 缺陷漏测率 | 线上缺陷/(线上+测试发现) | ≤5% | 越低越好 |
| AI 用例采纳率 | 人审核通过/AI生成总数 | ≥60% | 采纳率过低=生成质量差 |
| 平均缺陷修复时长 | 从new到verified | ≤24h(critical) | 按severity分 |
| 缺陷去重率 | 去重合并/总缺陷 | ≥30% | 反映Agent去重效果 |
| 测试执行时长 | batch 执行时间 P95 | ≤10min | CI 体感 |
| 覆盖率缺口闭环率 | 已补测缺口/总缺口 | ≥80%/月 | 持续消缺 |

## 12. 落地优先级

### 12.1 阶段规划

| 阶段 | 范围 | 优先级 | 依赖 |
|---|---|---|---|
| **P1-阶段一（MVP）** | 用例CRUD + 手动+CI触发执行 + 覆盖率采集门禁 + 基础质量报告 | 🟥 必须 | 板块02 研发、GitLab CI |
| **P1-阶段二** | AI 用例生成（正常+异常+边界） + 回归智能选取 + 根因分析Agent | ⭐ 核心 | 板块04 规则、Python agent-runtime |
| **P2-阶段三** | 缺陷智能分析（去重/指派） + 覆盖率缺口反向分析 + 测试数据合成 | 🟧 增强 | 板块05 安全（脱敏规则） |
| **P3-阶段四** | 自适应测试（历史数据学习） + 质量预测 + Flaky自愈 | 📘 演进 | 板块08 算力 |

### 12.2 MVP 必须项（阶段一验收标准）

- [ ] TestCase CRUD + 版本管理
- [ ] CI Webhook 接入 → 自动触发测试执行
- [ ] 覆盖率采集 + 80% 门禁阻断（🟥硬门）
- [ ] PR 评论（通过/失败 + 覆盖率）
- [ ] 基础质量报告（对接板块06）
- [ ] Go struct 全部实现 + sqlx 持久化

### 12.3 技术债红线

- 阶段一可不做 AI 生成（先手动），但质量门（覆盖率+测试通过）必须就位。
- 阶段一回归可全量，但回归选取框架必须预留。
- 测试数据脱敏规则**阶段一就必须接入**（RULE-TEST-007 🟥强制），不因 MVP 放松安全。
