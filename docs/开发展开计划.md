# 智源 ANP 平台 · 开发展开计划

> 从方案到代码的落地指南：如何动手把「智源」平台建出来。
> 配套文档：[`企业AI原生研发平台方案.md`](企业AI原生研发平台方案.md)（总体方案 V1.0）

## 0. 总策略：三条原则

1. **垂直切片，不做水平大饼**：先打通一条最窄的端到端闭环（需求→编码→部署→反馈），再横向扩展板块。不要把 9 个板块各建一半。
2. **混合栈，薄自研厚开源**：模型网关用 LiteLLM、流水线用 GitLab、应用搭建可借 Dify/扣子组件；自研只做核心四件套——Orchestrator（编排）、规则引擎（RaC）、闸门引擎、项目空间。
3. **第一天就 dogfooding**：平台用自己的需求工作台管自己的开发；第一个项目空间就是"智源平台本身"。

## 1. MVP 定义（最小可用闭环，约 10 周）

只做让一条链路跑通的最小集：

| 模块 | MVP 范围 |
|---|---|
| 项目空间 | 1 个空间（平台自身） |
| 需求工作台 | 对话提需求→生成规格→🚪G1 人工确认 |
| 规则治理中心 | 最小版：录几条硬规则，编码时违规阻断 |
| 研发工作台 | 接 Claude Code/Cursor 做 AI 编码 + AI Review |
| 测试 | AI 生成单测 + CI 执行 |
| 部署 | GitLab CI + 一个环境→🚪G5 上线审批 |
| 算力 | 先用模型 API（不建 GPU 池） |

**验收标准**：用这个 MVP 真实交付 1 个试点小项目（建议选一个内部工具）。

## 2. 推荐起步技术栈

| 组件 | 选型 | 理由 |
|---|---|---|
| 仓库 | monorepo（pnpm workspaces / nx） | 统一管理前后端 + Agent |
| 后端 | Python + FastAPI | AI 生态最全，Agent 框架丰富 |
| 前端 | Next.js + Tailwind + shadcn/ui | 快速出可用界面 |
| Agent 编排 | LangGraph | 显式多 Agent 状态机 |
| 模型网关 | LiteLLM | 统一 100+ 模型接口，模型无关 |
| 向量库 | pgvector（起步）→ Milvus | 起步零额外组件 |
| 规则/元数据 | PostgreSQL | 结构化规则 + 关系 |
| 代码托管/CI | GitLab 自托管 | 流水线 + 评审一体 |
| 编码代理 | Claude Code | 开发平台本身就用它 |
| 部署 | Docker Compose（起步）→ Kubernetes | 渐进 |

> 信创场景：模型换国产（通义/文心/DeepSeek/Qwen），其余开源组件均支持私有化。

## 3. Monorepo 结构

```
智源-ANP平台/
├── docs/                  # 方案与计划文档
├── platform/              # 平台自身代码
│   ├── backend/           # FastAPI 后端
│   │   ├── agents/        # 需求/编码/测试/评审 Agent
│   │   ├── gateway/       # LiteLLM 模型网关
│   │   ├── rules/         # 规则引擎 (RaC)
│   │   ├── workspaces/    # 项目空间
│   │   └── gates/         # 闸门引擎
│   ├── frontend/          # Next.js 工作台
│   ├── infra/             # 部署/基础设施
│   └── packages/          # 共享类型/SDK
├── pilots/                # 试点项目
└── README.md
```

## 4. 开发节奏（对齐方案阶段一，细化到开发动作）

- **阶段 0 · 准备（1–2 周）**：仓库、CI、模型 API、基础环境、dogfooding 起点。
- **阶段 1 · MVP 垂直切片（8–10 周）**：打通最小闭环 + 1 个试点交付。
- **阶段 2 · 补全板块（持续）**：多项目空间、测试/安全/发布/运维中心、AI 能力市场。
- **阶段 3 · 自主化深化**：运维自愈、成本调度、规模化。

## 5. 第一个 Sprint（2 周）具体任务

1. 建 monorepo + GitLab CI 骨架。
2. LiteLLM 网关跑通，调通第一个模型。
3. FastAPI 后端骨架 + PostgreSQL + 项目空间 CRUD。
4. Next.js 前端骨架 + 需求工作台对话页（最小可用）。
5. 规则中心最小后端（规则 CRUD + 一条阻断规则 demo）。
6. dogfooding：把"开发 MVP 本身"作为第一个项目空间录进需求工作台。

**Sprint 验收**：能在需求工作台用对话生成一条需求规格，并被一条规则校验。

## 6. 开发避坑

- 别先造完美的规则 DSL / 多租户——先够用。
- 别等所有板块就绪才试点——MVP 跑通就上试点。
- Agent 效果是迭代出来的——别追求一次到位。
- 闸门先走简单人工审批流——别先造复杂引擎。
- 模型先走 API——别一上来就建 GPU 池。

## 7. 今天就能做的第一步

1. 定 MVP 范围 + 选 1 个试点项目。
2. 建 monorepo 骨架。
3. 搭 LiteLLM 网关，跑通第一次模型调用。
4. 用需求工作台录第一条需求（dogfooding 起点）。

## 8. 里程碑与 Go/No-Go

| 里程碑 | 时间 | Go 标准 |
|---|---|---|
| M0 环境就绪 | 第 2 周末 | 网关调通模型、CI 绿、仓库就绪 |
| M1 MVP 闭环 | 第 10 周末 | 端到端跑通 + 试点项目交付 |
| M2 多项目空间 | 第 16 周末 | 第 2 个真实项目接入 |
