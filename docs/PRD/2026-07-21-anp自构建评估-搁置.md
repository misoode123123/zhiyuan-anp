# anp 自构建(self-build)评估 · 搁置

> 日期：2026-07-21 ｜ 状态：**搁置，后期再议**
> 触发：8 人真用起来讨论中，问「anp 能否通过自己（anp 的 appdeploy 引擎）构建部署自己」。

## 背景

anp 平台的 appdeploy 引擎（docker build→run→端口→URL）用于构建部署产出应用。
讨论：anp 自己注册为 app，用自己的 appdeploy 构建部署自己（self-build / 自举 / dogfooding 终极形态）。

## 正面（价值）

1. **dogfooding 终极**——用自己的引擎部署自己，最强自我验证（引擎连自己都能部署，用户 app 更没问题）。
2. **全流程统一**——需求→编码→审批→发布→部署全在 anp 闭环，免人工 tar/scp。
3. **验证引擎成熟**——anp 复杂多服务，能构建自己 = appdeploy 引擎到生产级。
4. **发布即部署**——自动化，迭代快。

## 反面（风险）

1. **模式不匹配（技术阻塞）**——appdeploy 是「单 Dockerfile→单容器」；anp 是 docker-compose 多服务（backend+frontend+postgres+agent-runtime+nginx）。要改造 appdeploy 支持 compose（大改）或 anp 单容器化（不现实）。
2. **部署中断 + 单点故障**——构建部署自己 = 重建正在运行的 anp 容器。build 失败/新容器起不来 → 平台自己挂，**无法用平台自救**（SSH 人工救，2026-07-20 部署事故教训）。
3. **数据持久化**——appdeploy `docker run` 要挂 volume（PG 数据/repos/opencode 会话），否则重建丢数据。
4. **回滚难**——新挂旧删，回滚要保旧镜像。
5. **生产=开发环境**——8 人 + 你都在用，自部署失败全员断。

## 决策：搁置，后期再议

anp 自构建技术可行但风险高（单容器 vs 多服务 compose 不匹配 + 部署中断单点）。暂搁置。

## 后期建议（分层）

- **现在**：anp 走 appdeploy 流程到「合并」（dogfooding 价值已实现，流程留痕），**部署人工 tar/scp 兜底**（现状，可控）。
- **中期**：anp 自部署到**独立 staging 实例**（不动开发用的 prod anp），验证后人工推 prod。
- **长期**：anp 自己 CI/CD（push→自动测试→staging→prod 灰度），self-build 的安全形态。

## 重新评估条件（后期重启讨论当）

- appdeploy 改造支持 compose（或 anp 单容器化）
- staging 环境就绪（anp 自部署不影响 prod）
- 回滚机制完善（保旧镜像 / 蓝绿部署）
- 数据持久化方案确认（volume 挂载）
