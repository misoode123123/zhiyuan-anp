# 需求-代码核对门禁 · P0 修复 PRD

> 日期：2026-07-20 ｜ 状态：**待评审**
> 上游设计：[`../superpowers/specs/2026-07-17-协作需求驱动开发-设计.md`](../superpowers/specs/2026-07-17-协作需求驱动开发-设计.md)（期1 核对门禁）
> 触发：2026-07-20 测试巩固时探现状，发现期1 门禁「装了但没接电源」——2 个 P0 让其形同虚设。

---

## 1. 目标

让「提交前 AI 核对代码 vs 需求验收标准」这道门禁**真正生效**：开发者没实现需求时**真能拦住提交**。当前接口壳子和 AI 核对调用都在，但 2 个 P0 使其实际不拦。

## 2. 问题（实证于 `internal/appdeploy/handler.go`）

### 🔴 P0-1：读错代码源（`handler.go:282, 319-339`）

- **现状**：`Submit` 调 `readRepoCode(a.RepoDir)`，读的是**主 repo 工作区**。
- **问题**：开发者在 `dev-<user>` 分支（worktree `repo/.worktrees/<user>/`）写的代码，Submit **看不到**——AI 核对的是 main 当前代码，不是开发者刚写的那份。等于保安检查的是隔壁仓库的货。
- **对照**：`Merge`（`handler.go:307`）和 `codews.ensureWorktree` 都用 `dev-<sanitizeID(user)>` 分支 + `.worktrees/<sanitizeID(user)>/`，唯独 `Submit` 没用。

### 🔴 P0-2：核对基准不可信（`handler.go:277-281, 344-346`）

- **现状**：`Title` / `AcceptanceCriteria` 由前端 request body 传入，Submit **不查 requirement 表、不接 req_id**。
- **问题**：前端可传空/假标准。`checkRequirement` 在 `code=="" && criteria==""` 时直接 `return true`（"跳过核对"）→ 门禁被绕过。等于保安别人说合格他就信，自己不验。

### 🟡 附带 P1（修 P0 时一并，否则门禁仍可绕过）

1. **AI 失败静默放行**（`handler.go:344-388`）：apiKey 空 / 请求构造失败 / HTTP 失败 / 无响应 / 解析失败，共 **5 处 `return true`**。设计意图是「不匹配拦」，现状是「AI 不可用就放」。
2. **Submit 通过不自动登记变更**（`handler.go:292`）：只回「可点登记变更」，实际靠开发者再调 RegisterChange——两步手工、无关联约束，可单独调 RegisterChange 绕过门禁。

---

## 3. 方案

### 3.1 P0-1：读 worktree 代码

- `Submit` 取 `X-User`（与 `Merge` 一致，空则 `anonymous`），算 worktree 路径 `<a.RepoDir>/.worktrees/<sanitizeID(user)>/`。
- `readRepoCode` 改读该 worktree 路径（**复用 `sanitizeID`**，命名与 codews/Merge 严格对齐）。
- worktree 不存在 → 400「先认领需求/打开工作台生成工作分支」。

### 3.2 P0-2：从 requirement 读基准

- `Submit` body 加 `req_id`（**必填**，关联需求）。
- `Handler` 加依赖 `reqRepo *requirement.Repository`（改 `NewHandler` 签名 + `main.go` 装配；`reqRepo` 在 main.go 已构造）。
- 从 requirement 表读 `title` + `acceptance_criteria`，**不再信前端 body 的 title/criteria**。
- `acceptance_criteria` 为空 → 400 拒绝（「需求无验收标准，请先补全」），不跳过。

### 3.3 P1：AI 失败不放行 + 自动登记变更

- `checkRequirement` 失败路径改为返回 `(false, 原因)`，由 Submit 转 **503**，不静默放行。区分：
  - apiKey 未配置 → 503「AI 未配置，请联系管理员」
  - AI 调用/解析失败 → 503「AI 核对失败，请重试」
- Submit 全 ✅ → **自动登记 `change_request`**（pending, source_id=req_id, application_id=aid），返回 change_id。复用现有 RegisterChange 逻辑。

---

## 4. 接口契约

`POST /project-spaces/:id/apps/:aid/submit`

**请求**：`{ "req_id": "<需求ID, 必填>" }`

| 场景            | HTTP | 业务码 | 响应                                             |
| --------------- | ---- | ------ | ------------------------------------------------ |
| 全 ✅           | 200  | —      | `{passed:true, details, change_id}` 自动登记变更 |
| 有 ❌           | 409  | 40930  | 差异清单                                         |
| 验收标准为空    | 400  | 40031  | 请先补全验收标准                                 |
| 应用/需求不存在 | 404  | 40420  | —                                                |
| worktree 不存在 | 400  | 40032  | 先认领/打开工作台                                |
| AI 失败         | 503  | 50301  | 配置或重试                                       |

> 前端配套：编码 tab 的「提交」按钮改为传 `req_id`（当前选中的需求），不再传 title/criteria。

---

## 5. 验收标准

1. 开发者在 `dev-<user>` 写代码，Submit 时 AI 读到的是 **worktree 代码**（不是 main）。
2. 验收标准**从 requirement(req_id) 读**，前端只传 req_id。
3. 代码未实现验收标准 → **被拦（409 + 差异清单）**。
4. AI 失败 → **不放行**（503）。
5. 全 ✅ → **自动登记 change_request**（关联 req_id），不再两步手工。
6. 验收标准为空 → 拒绝（不跳过核对）。

## 9. 风险

- worktree 路径须与 `codews.ensureWorktree` 命名**严格一致**（sanitizeID），否则读不到代码 → 改动时对齐 + 测试覆盖。
- `readRepoCode` 截断（15 文件 / 1200 字 / 总 8000 字）大 repo 可能漏关键文件 → 本 PRD 不改，留调参。
- AI 核对准确性（误判）→ 本 PRD 不含申诉通道，靠 503 重试 + 后续申诉 PRD 兜底。
