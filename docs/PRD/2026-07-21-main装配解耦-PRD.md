# main.go 装配解耦（减少 8 人后端合并冲突）

> 日期：2026-07-21 ｜ 状态：**实施中（P1）**
> 上游：[8人团队工程化完善方案](../2026-07-18-8人团队工程化完善方案.md) P1-7
> 触发：8 人改后端都改 main.go（new 22 handler + 依赖 + Register 集中），合并冲突集中点。

## 1. 现状（main.go:102-236）

- main new 22 handler（workspace/dev/requirement/conversation/config/rule/standard/docs/change/qa/release/compute/auth/ops/security/capability/attendance/appdeploy）+ 它们的 store/svc + 跨模块依赖组装。
- 跨模块依赖在 main 编织：requirement.Svc 需 devAgent/appDeployStore；appdeploy.Handler 需 reqRepo；release.Handler 需 appDeployHandler。
- 8 人加/改任一模块都要改 main → 冲突集中。

## 2. 目标

main.go 只做：① 组装共享依赖（db/config/共享 store + 跨模块枢纽 devAgent/reqRepo/appDeployHandler）② 调各模块 `Register(v1, deps...)`。模块内部 new（store/svc/handler）移到模块自己，8 人改模块不碰 main。

## 3. 模式

- 每模块暴露**模块级** `Register(r gin.IRouter, deps...)`：内部 new 自己 store/svc/handler + `h.Register(r)`。
- 共享 store（db/config/auth/change/compute/rule/standard/codetask/appDeploy/qa）main new（含 seed）；跨模块枢纽（devAgent/reqRepo/appDeployHandler）main new + 传递。
- 跨模块依赖保持现有 interface 解耦（requirement 用 `AppResolver` interface，不直接 import appdeploy，避免循环）。
- 复杂模块签名长（接多个共享），但 main 行数大减、模块自包含。

## 4. 分批迁移

- **P1 简单模块**（无跨模块枢纽依赖）：config/rule/standard/docs/change/auth/compute/security/attendance/capability/ops/workspace。各加 `Register(v1, store...)`。
- **P2 复杂模块**（跨依赖）：dev/requirement/conversation/qa/appdeploy/release。`Register(v1, 共享..., 跨模块枢纽)`；appdeploy.Register 返 `*Handler` 供 release。

## 5. 风险

- 22 模块 Register 签名 + main 重构。每批 be:test 全绿 + 启动验证。
- 跨模块依赖循环：保持 interface（AppResolver）解耦，模块不互相 import 业务类型。
- appdeploy→release 的 appDeployHandler：appdeploy.Register 返 handler，release.Register 接收。

## 6. 验收

- main.go 从 new 22 handler → 组装共享 + 调 Register（行数大减）。
- be:test 全绿；启动正常（登录/各中心）。
- 8 人改模块 handler 内部不碰 main（仅加新模块/新共享 store 才改 main）。
