> 触发：dogfooding 暴露——认领需求后**不会自动建 worktree**（手动 git worktree add 才行）、合并后**不清理**，流程断在手工补。

---

## 1. 目标

把「认领需求 → 编码 → 提交核对 → 审批 → 合并 → 交付」链条上的**自动化联动**接通，让开发者不用手动 `git worktree add`、合并后状态自动收敛。

## 2. 缺口（实证，dogfooding + 探索报告）

| #   | 缺口                   | 现状                                                                                                                                              | 位置                                    |
| --- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| 1   | **认领不建 worktree**  | `Assign` handler 只写 assignee，不接触 codews；worktree 由单独的 `POST /apps/:aid/workspace` 触发。开发者认领后还要手动开工作台/手动 git worktree | `requirement/handler.go:194-204` Assign |
| 2   | **合并不收敛**         | `Merge` 只 `git merge --no-ff`，不释放 assignee、不置需求 delivered、不清 worktree                                                                | `appdeploy/handler.go:296-317` Merge    |
| 3   | **Merge 无 G3 前置**   | Merge 不检查 `change_request.status='approved'`，任何人任何时候能 merge（Promote 倒是有 grandfather 闸门）                                        | `appdeploy/handler.go:296`              |
| 4   | **24h 超时释放未实现** | 全后端无 cron/scheduler，认领后不提交永不释放                                                                                                     | main.go 无 cron                         |

## 3. 方案

### 3.1 认领自动建 worktree（#1）

- `Assign` handler 成功后，调 `codews.Manager.Ensure(appID, repoDir, user, "opencode")` 建/复用 `dev-<user>` worktree（设计 5.2/5.3）。
- requirement handler 加 `codeWS *codews.Manager` 依赖（改 `NewHandler` + main.go 装配）。assign 时按需求的 `application_id` 解析 app 的 repo_dir（无 application_id 则跳过 worktree，只认领）。

### 3.2 合并后收敛（#2）

- `Merge` 成功后：① `requirement.Repository.Release(req_id)` 释放认领；② `UpdateStatus(req_id, "delivered")`；③ 清理 `.worktrees/<user>/`（`git worktree remove`）。
- Merge handler 加 `reqRepo` 依赖（已有，P0 加过）+ 合并成功后联动。

### 3.3 Merge 加 G3 前置（#3）

- Merge 前检查 `change.HasApproved(appID)`，未审批 → 409「需先审批变更」。
- 对齐 Promote 的 grandfather 模式（设计 4-⑥⑦）。

### 3.4 24h 超时释放（#4，标后续）

- 需 scheduler（cron）。当前 main.go 无 cron 基础设施。
- **本 PRD 不实现**，列为后续（引入 cron 框架后做）。认领超时暂时靠人工 Release。

## 4. 接口变更

- `POST /project-spaces/:id/requirements/:rid/assign`：成功后**自动建 worktree**（响应多 `worktree_url`）。
- `POST /project-spaces/:id/apps/:aid/merge`：
  - 加 `req_id` body 参数（合并哪条需求的变更）
  - 无 approved 变更 → 409
  - 成功 → 合并 + 释放认领 + delivered + 清 worktree，响应 `{merged, released, worktree_cleaned}`

## 5. 验收

1. 认领需求 → **自动建 dev-<user> worktree**（不用手动 git）。
2. Merge 未审批的变更 → **409 拦截**。
3. Merge approved 变更 → 合并 main + **释放 assignee + 需求 delivered + 清 worktree**。
4. 单测覆盖：assign 建工作台、Merge G3 前置、Merge 后收敛。
