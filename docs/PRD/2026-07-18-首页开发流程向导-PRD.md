# PRD:首页「开发流程向导 + 我的任务」

| 版本 | 日期 | 作者 | 状态 |
|---|---|---|---|
| v1.0 | 2026-07-18 | ANP 团队 | 待评审 |

## 1. 背景

当前首页(`/`)是统计数字(成员/应用/需求/变更/发布)+ 3 个静态链接卡片。问题:
- 开发者打开首页**不知道该做什么**(没有个人待办)
- **流程不可见**(需求→上线的全链路无可视化)
- 各操作模块(需求/编码/审批/部署)散落,需手动找

## 2. 目标

- **流程可见**:首页顶部图形化展示开发全流程(需求→上线)
- **任务驱动**:每个人看到自己权限内各阶段的**待办任务**,点击直达处理
- **一键处理**:任务项链接到对应操作模块

## 3. 用户角色与场景

| 角色 | 核心场景 |
|---|---|
| 业务 | 提需求,看需求是否被认领/开发 |
| 开发者 | 认领需求、编码、测试、核对、登记变更 |
| 审批人/负责人 | 审批变更、上线 |

## 4. 功能需求

### F1 图形化流程向导(首页顶部)

横向流程图,展示开发全流程 8 个节点:

```
①需求 → ②认领 → ③编码 → ④测试 → ⑤核对 → ⑥登记 → ⑦审批 → ⑧上线
 💬      👤      🧑‍💻     🧪     🔒      📝     ✅      🚀
 [3]    [2]    [1]                              [1]
```

- 每节点:**图标 + 名称 + 我的待办数角标**([N])
- 当前用户有该阶段待办 → 节点高亮(蓝)+ 角标
- 无待办 → 灰显
- 点击节点 → 跳转对应模块

| 节点 | 跳转 |
|---|---|
| ①需求 | `/requirements` |
| ②③④⑤⑥(开发类) | `/applications`(选应用进编码 tab) |
| ⑦审批 | `/approvals` |
| ⑧上线 | `/applications` |

### F2 我的任务列表(流程图下方)

当前用户在各阶段的**待办清单**,按阶段分组:

| 阶段分组 | 任务数据来源 | 「去处理」链接 |
|---|---|---|
| 待认领 | 需求 `assignee` 为空 | `/requirements`(认领) |
| 我的开发中 | 需求 `assignee=当前用户`,status=developing | `/workspace?app=<归属应用>` |
| 待核对 | 已编码待提交核对(开发中 + 有代码) | `/workspace` |
| 待我审批 | 变更 `status=pending` | `/approvals` |
| 待上线 | 变更 `status=approved` | `/applications`(上线) |

任务卡片字段:
- 标题(需求标题 / 变更摘要)
- 状态标签(priority P0/P1、developing、pending、approved)
- 归属应用
- 「去处理」按钮(跳模块)

### F3 权限/角色过滤

按当前用户角色,显示对应阶段:

| 阶段 | 业务 | 开发 | 审批 |
|---|---|---|---|
| ①需求 | ✅ | | |
| ②认领 | | ✅ | |
| ③编码 | | ✅ | |
| ④测试 | | ✅ | |
| ⑤核对 | | ✅ | |
| ⑥登记 | | ✅ | |
| ⑦审批 | | | ✅ |
| ⑧上线 | | | ✅ |

非自己角色的阶段:灰显/隐藏,角标不计数。

## 6. 数据接口

### 新增:`GET /project-spaces/:id/my-tasks`(按 `X-User`)

返回当前用户各阶段待办聚合:

```json
{
  "toClaim":   [{id, title, priority, application_id}],   // 待认领需求
  "myDev":     [{id, title, status, application_id}],     // 我的开发中
  "toApprove": [{id, kind, status, source_id}],           // 待我审批(pending 变更)
  "toRelease": [{id, status, source_id}]                  // approved 待上线
}
```

实现:聚合 requirement(按 assignee/status)+ change(按 status)。

### 现有(保留):`GET /project-spaces/:id/overview`(统计数字,首页底部保留)

## 7. 权限矩阵

角色判定:用户在项目空间的成员角色(auth.rbac)。首页按角色过滤阶段(F3)。
- 无角色/匿名:只读统计,无任务

## 8. 验收标准

| 编号 | 验收点 |
|---|---|
| AC1 | 首页顶部显示 8 节点流程图,有待办的节点高亮+角标 |
| AC2 | 角标数 = 当前用户该阶段真实待办数 |
| AC3 | 点击节点跳转对应模块 |
| AC4 | 我的任务列表按阶段分组,显示真实待办 |
| AC5 | 任务「去处理」跳转正确(认领→需求、编码→workspace、审批→approvals) |
| AC6 | 不同角色看到不同阶段(业务只看需求、审批只看审批/上线) |
| AC7 | 待办完成(认领/审批)后,列表实时刷新(角标-1) |

## 9. 实现分期

- **期1【前端为主】**:流程图向导(静态节点+链接)+ 我的任务(前端聚合现有 requirements/changes 接口,按 assignee/status 过滤)。快速上线。
- **期2【后端聚合】**:`/my-tasks` 聚合接口(一次请求拿全部待办,优化性能)+ 角色权限过滤(接 RBAC)。

## 10. 非功能需求

- **性能**:首页加载 < 2s(任务聚合接口 < 500ms)
- **响应式**:流程图窄屏可横滑;任务列表自适应
- **实时**:任务完成(认领/审批/上线)后首页刷新(轮询 or 返回触发)
- **扩展**:节点可配置(未来加"设计""评审"等阶段)

## 11. 风险与边界

- **角色判定**:需接现有 RBAC(auth.member.role);期1 可简化(所有人看全阶段),期2 接角色
- **任务聚合性能**:多表 join;期2 用专用接口优化
- **"待核对"界定**:已编码但未提交核对的判定(需状态机细化),期1 可并入"我的开发中"

## 12. 关联

- 复用:需求(assignee)、变更(status)、应用(deploy)等已有数据
- 依赖:协作式需求驱动开发(认领/核对/审批 流程,见 `docs/superpowers/specs/2026-07-17-协作需求驱动开发-设计.md`)
