# 真实鉴权 · 撤 X-User 模拟回退 + SSO

> 日期：2026-07-20 ｜ 状态：**✅ P0 已落地并验证（撤 X-User 回退 + 多用户密码 + 登录守卫）**
> 上游：[8人团队工程化完善方案](../2026-07-18-8人团队工程化完善方案.md) P1-2
> 触发：8 人团队可用第一道门。当前鉴权体系完整,但前后端 X-User 模拟回退让任何人传 `X-User: admin` 即可伪装,鉴权形同虚设。

---

## 1. 现状（核实于 2026-07-20）

**已有（体系完整）**：

- `User` 表（id/name/email/status/password_hash）+ bcrypt 密码
- `auth_session` token（`tok_xxx`，7 天，DB 存可吊销）— `Login`/`ValidToken`/`Logout`
- `AuthUser` 中间件（Bearer token → user_id）+ `Require`/`AutoRequire` RBAC
- 权限矩阵 `OpRoles`（6 操作 × 5 角色：business/dev/rule_architect/gatekeeper/admin）
- 登录页 `/login`（name+password → token）+ 用户管理 `/users` + 成员管理 `/members`
- admin/admin123 seed（`EnsurePassword`）

**🔴 唯一漏洞：X-User 模拟回退（前后端）**：

- 后端 `AuthUser`（middleware.go:32-37）：无 Bearer token → 回退 `X-User` 头（默认 anonymous）。**任何人传 `X-User: admin` 即伪装 admin**。
- 前端 `installAuthInterceptor`（api.ts:69-71）：无 token → 带 `X-User: currentUser()`（默认 admin）。
- `login` 页"跳过,以游客继续（X-User 模拟）"链接。
- token 是 DB session（非 JWT），可吊销；8 人规模够用，不必强上 JWT。

## 2. 目标

1. **撤 X-User 模拟回退**：无 token → 401（后端）/ 跳登录（前端）。强制真实登录。
2. 8 人 8 账号：admin 创建用户 + 分空间角色（`/users` `/members` 已有）。
3. SSO（钉钉/企微 OIDC）作企业进阶。

## 3. 方案

### P0 撤模拟回退（核心，小改动，breaking）

- **后端 `AuthUser`**：无 Bearer token → 401「未登录」（删 X-User 回退）；token 无效 → 401。`/auth/login` + `/healthz` 白名单放行。
- **前端 `installAuthInterceptor`**：无 token 不带 X-User（请求自然 401 → 守卫跳登录）。
- **前端登录守卫**：Shell/layout 检查 `isLoggedIn()`，未登录跳 `/login`（除 /login 本身）。
- **删 login"跳过游客"**链接。
- 公开接口白名单：`/auth/login`、`/healthz`（放行）；其余强制 token。

### P1 SSO/OIDC（企业，可选）

- 后端 `/auth/sso/{provider}/redirect` + `/callback`（钉钉/企微）。OIDC 换用户信息 → 自动建/找 User + 签 token。
- 前端 login 加"SSO 登录"按钮。
- 配置：`sso_dingtalk_client_id/secret` 等（system_config 或 .env）。

### P2 JWT（可选，不急）

- DB session → JWT（无状态）。但失吊销能力（要黑名单）。
- **建议保留 DB session**：8 人规模 DB session 够（可吊销、每请求查 DB 开销小）；除非 P1 SSO 要 JWT，否则不换。

## 4. 改动清单（P0）

- `internal/auth/middleware.go`：`AuthUser` 撤 X-User 回退、无 token 401、白名单 login/healthz
- `internal/auth/middleware_test.go`（新增）：无 token 401、有 token 放行、login 白名单
- `platform/frontend/lib/api.ts`：`installAuthInterceptor` 无 token 不带 X-User
- `platform/frontend/app/_components/shell.tsx`（或 layout）：登录守卫
- `platform/frontend/app/login/page.tsx`：删"跳过游客"

## 5. 验收

1. 未登录访问任意 API → 401；前端跳 /login。
2. `/auth/login` + `/healthz` 无 token 可访问（白名单）。
3. 登录后（token）正常访问；`X-User` 头不再起作用（即使传也忽略）。
4. `pnpm be:test` 绿（含新 middleware_test）。

## 6. 风险

- **breaking**：撤回退后未登录全 401。需前端守卫 + token 持久（localStorage 已有）。本地 dev 也要登录（admin/admin123）。
- **anonymous 依赖**：submit/merge 等用 X-User 取 user。撤后必须登录（合理，8 人都要账号）；后端从 token 解 user（已有 CtxUserID）。
- **现有 session 失效**：DB session 不变，已登录用户不受影响。

## 7. 建议

先做 **P0**（撤回退，小改动立即真实鉴权），验证 8 人用户名密码登录闭环。SSO（P1）/JWT（P2）按企业需求再定。
