# 协作式需求驱动开发 + 需求-代码核对门禁

> 日期:2026-07-17 ｜ 状态:待评审

## 1. 背景与目标

编码 tab 已支持"选需求 → opencode 协助编码 → 自动测试",但存在根本缺口:

- **需求-代码是否匹配无保证**:代码写完就能提交,没人核对"是否真实现了需求"。这是之前反复调整(dispatch/serve/逐个/增量)都没解决的根因——缺一道"代码 vs 需求"的核对。
- **多人协作会冲突**:多开发者同 repo 主线编码,互改/覆盖;同一需求可能被多人重复开发。

**目标**:建立"**标准需求为准绳、核对门禁保证、多人协作不冲突**"的开发流程。

## 2. 核心理念

1. **标准需求是唯一准绳**——需求规格(尤其验收标准)是开发与核对的唯一基准。
2. **需求-代码核对门禁**——提交前 AI 逐条核对"代码实现 vs 需求验收标准",不匹配**不让提交**。
3. **人协助 AI**——开发者协助 opencode 编码/测试(交互式,看过程、可介入,非全自动)。
4. **多人协作不冲突**——需求认领互斥(防重复)+ git worktree 隔离(防代码冲突)+ 合并集中处理。

## 3. 角色与职责

| 角色 | 职责 |
|---|---|
| 业务 | 提标准需求(标题/用户故事/**验收标准**) |
| 开发者 | 认领需求 → 协助编码 → 测试 → 提交(过核对门禁) |
| 审批人 | 审批变更(G3 闸门) |
| 平台 | 核对门禁 + worktree 管理 + 合并 |

## 4. 端到端流程

```
① 业务提【标准需求】(验收标准 = 核对基准)
        ↓
② 开发者【认领需求】(互斥:已被他人认领则拒绝)
        ↓ 认领成功 → 平台自动建该开发者的 worktree(dev-<user> 分支)
③ 开发者【协助 opencode 编码】
        在自己 worktree 内,按需求逐项、基于现有代码增量(看过程/可介入)
        ↓
④ 【测试】自动(qa 按验收标准跑 HTTP)+ 人工(操作 test 环境)
        ↓
⑤ 开发者点【提交】→ ★ 需求-代码核对门禁 ★
        AI 读 worktree 代码 + 逐条核对验收标准
        ├ 有 ❌未实现/⚠️偏离 → 拦截(列差异)→ 回 ③ 修正
        └ 全 ✅ → 允许提交(登记变更,关联需求)
        ↓
⑥ 【变更审批】(G3,审批人 approve/reject)
        ↓
⑦ approved →【合并】dev-<user> → main(冲突则提示+解决)
        ↓
⑧ 【上线 prod】→ 需求 delivered,释放 assignee
```

## 5. 各环节设计

### 5.1 标准需求
- 字段:标题、用户故事、**验收标准(结构化数组,核对基准)**、优先级、计划版本
- 验收标准每条应是**可核对的陈述**(如"GET / 返回 200""游戏能暂停"),便于 AI 核对。
- 新增字段:`assignee`(认领人)、`assigned_at`(认领时间)。

### 5.2 需求认领(互斥)— 防重复分配
- 接口 `POST /requirements/:rid/assign`:写入 `assignee=当前用户`。
- **互斥规则**:若 `assignee` 非空且 ≠ 当前用户 → **拒绝**,提示"需求已被 {assignee} 认领"。
- 前端:需求列表显示认领人;**已认领的需求他人不可选**(灰显 + "X 开发中");未认领的可「认领开发」。
- **释放**:合并主线 / 需求 delivered → 清空 assignee;**超时释放**:认领后 24h 未提交 → 自动释放(防占着不干)。

### 5.3 协助编码(opencode + worktree)
- 认领时,codews 为该 `app+user` 创建/复用 worktree:`repo/.worktrees/<user>`,分支 `dev-<user>`。
- opencode cwd = 该 worktree(隔离,不互改)。
- 协助式:开发者协助 AI 按需求逐项编码(交互、看过程);prompt 强约束**基于现有代码增量、不重写**。
- 子任务 checklist 作为逐项引导(非硬绑定)。

### 5.4 测试
- **自动**:qa 模块按验收标准生成用例 + 对着 test URL 跑(HTTP/状态)。
- **人工**:开发者访问 test URL 实际操作(功能/交互/视觉)。
- 两者都过才建议提交。

### 5.5 ★ 需求-代码核对门禁(核心)
- **时机**:开发者点「提交」(平台可控点;opencode 内部 commit 拦不住,但"提交/登记变更"是平台动作)。
- **输入**:worktree 当前代码(关键文件内容)+ 需求验收标准(逐条)。
- **AI 核对**:调 GLM,对每条验收标准判断:
  - ✅ 已实现(指出在哪个文件/函数)
  - ❌ 未实现
  - ⚠️ 部分实现/偏离(说明差异)
- **拦截规则**:存在任意 ❌ → **拒绝提交**,返回差异清单 → 开发者回 opencode 修正;全 ✅(允许 ⚠️ 但需说明)→ 允许提交。
- **人工兜底**:AI 核对可误判,允许开发者「申诉」(填理由)→ 仍进入审批,由审批人最终把关。

### 5.6 变更审批(G3)
- 过核对门禁的提交 → 登记 ChangeRequest(pending,关联需求 source_id)。
- 审批人 approve/reject。
- (已有 G3 闸门 + grandfather:登记过变更的应用 promote 需 approved。)

### 5.7 合并上线
- approved → `dev-<user>` 合并到 `main`。
- **冲突**:git merge 报冲突 → 平台返回冲突文件/内容 → 开发者解决(或 AI 协助合并)→ 再合。
- 合并后 → 可上线 prod → 需求 `delivered`,释放 assignee,清理 worktree。

## 6. 多人协作(三层防冲突)

| 层 | 机制 | 防什么 |
|---|---|---|
| 需求层 | assignee 认领互斥 + 超时释放 | 同一需求不被多人重复开发 |
| 代码层 | git worktree(dev-<user>)隔离 | 不同开发者代码不互改 |
| 合并层 | dev→main 合并 + 冲突处理 | 集中处理不可避免的冲突 |

主线 `main` 受保护:opencode 不直接在 main 编码,只接受合并。

## 7. 数据模型变更

- `requirement` 加列:`assignee TEXT`、`assigned_at DATETIME`(migration 幂等加列)。
- worktree 元数据:`repo/.worktrees/<user>/`(文件系统,codews 管理;无需建表)。

## 8. 接口设计

| 接口 | 作用 |
|---|---|
| `POST /requirements/:rid/assign` | 认领需求(互斥),触发建 worktree |
| `POST /requirements/:rid/release` | 释放认领(手动) |
| `POST /apps/:aid/submit` | **提交 + 核对门禁**(AI 核对,不匹配拦) |
| `POST /apps/:aid/merge` | dev-<user> → main 合并(冲突返回) |
| (codews 内部) | Ensure 时按 user 建/复用 worktree |

## 9. 异常处理

| 场景 | 处理 |
|---|---|
| 认领超时(24h 未提交) | 自动释放 assignee,他人可认领 |
| 核对失败(有 ❌) | 拦截 + 差异清单,回编码修正;支持申诉(人工审) |
| 合并冲突 | 返回冲突详情,开发者/AI 解决后重试 |
| 编码失败 | 提示错误,开发者介入修正 |
| AI 核对误判 | 申诉通道 + 审批人终审兜底 |

## 10. 分期实现

- **期1【核对门禁】(核心,先做)**:`/submit` 接口 + AI 核对(代码 vs 验收标准)+ 拦截。直接实现"不匹配不让提交"。
- **期2【多人协作】**:需求 assignee 认领互斥 + codews worktree 隔离 + 合并接口。
- **期3【标准需求模板】**:验收标准结构化模板(利于核对准确)。

## 11. 边界与风险

- **AI 核对准确性**:可能误判(漏检/误报)→ 申诉 + 人工审批兜底;prompt 要求"指出实现位置"降低误判。
- **worktree 资源**:每开发者一目录,需清理(合并/释放时);大 repo worktree 占空间。
- **合并冲突复杂度**:大改动/同文件多改,冲突难自动解 → 人工 + AI 协助。
- **opencode 内部 commit**:平台拦不住 opencode 内 commit,门禁放"提交/登记变更"(平台动作),不依赖拦 commit。

## 12. 与现有能力的关系(复用)

- 需求规格、验收标准(requirement):已有 → 作核对基准。
- 编码 tab 选需求 + opencode 协助 + 逐项:已有 → 接 worktree。
- 自动测试(qa):已有 → 提交前跑。
- 变更审批(G3)+ grandfather 闸门:已有 → 核对门禁后接审批。
- codews 多 user session:已有 → 扩展为多 user worktree。
