# 智源 ANP · 测试与迁移巩固方案（P0 收尾 + P1 起点）

> 日期：2026-07-20 ｜ 状态：**T1/T2/T3 已完成并验证（覆盖率 21.8%→36.4%），顺带修复 7 处切 PG 生产 bug；T4 迁移工具化进行中**
> 上游：[`2026-07-19-工程化落地执行计划.md`](2026-07-19-工程化落地执行计划.md) 的 **P0-3（迁移）+ P0-4（测试）** 深化收尾，并新增前端测试。
> 触发：切 PostgreSQL + OpenAPI 扩 33 接口之后，处于「快速扩接口」的巩固拐点——再往前推业务功能而不补护栏，债务会爆。

---

## 0. 背景与目标

切 PG、扩接口、前端 18 路由全驱动后端——**广度已成型，但护栏几乎为零**：

- 后端业务核心写路径（`release/qa/rule/workspace/requirement/conversation`）**零单测**，总体覆盖率估算 ~10–15%，CI 覆盖率**只上传 artifact、无门禁**。
- CI 的 Go 版本配 `1.22`，与 `go.mod` 的 `go 1.25.0` **不一致**（本地 1.26.5 能编，CI 一跑就爆）。
- 前端**没有测试框架**，`pnpm fe:test` 指向不存在的 script，根 `pnpm test` 必失败。
- 迁移是**手写幂等式、仅 up、非事务**，多人改表无标准流程（P0-3 当时评估「随切 PG 一并做」，PG 已切，该兑现）。

**目标（2 周）**：任何 push 都有「测试护栏 + Go 版本对齐 + 覆盖率门禁」，迁移可 up/down，CI 真能拦住回退。

### 1.2 迁移器现状（`internal/db/migrate.go`）

- 手写：embed `migrations/pg/*.up.sql`，按文件名排序，`schema_migrations` 表记录版本，已应用则跳过（幂等）。
- **仅 up**：代码只读 `.up.sql`，`.down.sql` 虽存在但从不执行。
- **非事务**：每个迁移文件一次 `ExecContext`，半途失败会留半成品。
- **仅 PG**：SQLite 驱动保留但 `Migrate` 不支持。

### 1.3 CI 现状（`.github/workflows/ci.yml`）

- ⚠️ **Go 版本不一致**：backend job 与 openapi job 都 `go-version: "1.22"`，`go.mod` 是 `1.25.0`。
- 覆盖率：`go test -coverprofile=coverage.out` → 上传 artifact，**无阈值门禁步骤**。
- 前端 job：lint + prettier + tsc + build，**无 test 步骤**（因 frontend 无 test script）。

### 1.4 前端测试现状（`platform/frontend`）

- `package.json` **无 `test` script**，无 `vitest/jest/playwright` 依赖。
- 根 `package.json` 的 `fe:test` → `pnpm --filter frontend test` **必失败**。
- 唯一测试 `lib/devstep.test.ts`：standalone（`npx tsx` 手跑 + `console.log` 断言），未纳入任何框架。

## 2. 任务卡

### T1 后端核心写路径补单测

- **改动**：为零测试的业务核心包补 store/service 单测，优先级 `release > qa > rule > workspace > requirement > conversation > config`。模式仿 `internal/change/store_test.go`（SQLite 内存 + 自建单表，自包含、不依赖外部 PG）。每包覆盖：关键写路径 + 边界/异常（状态机迁移、空值、重复操作、权限边界）。
- **验证**：`pnpm be:test` 全绿；`pnpm be:cover` 总体覆盖率 ≥ 30%。
- **验收**：核心写路径均有异常/边界用例；覆盖率门禁（T2）通过。
- **风险**：测试表 schema 手写，可能和 PG 迁移 schema 漂移（类型 `DATETIME` vs `TIMESTAMP`、缺约束/默认值）。**缓解**：T1 内逐表核对测试建表语句与 `000001_init.up.sql` 对齐，漂移处留 `TODO`；PG 方言验证留给后续集成测试 / dogfooding（本次不引入 PG 依赖，保住「单测快、自包含」）。

### T2 CI 门禁 + Go 版本对齐

- **改动**：
  1. ci.yml 两处 `go-version: "1.22"` → `"1.25"`（对齐 go.mod；本地 1.26.5 兼容）。
  2. 加覆盖率阈值检查（`go tool cover -func` 取总覆盖率 + 脚本断言 ≥30%，或接 codecov/covergate），起步 30%、只升不降。
  3. 前端 job 加 `test` 步骤（依赖 T3 完成）。
- **验证**：本地 `act` 或推特性分支触发；故意降覆盖率验证被拦。
- **验收**：CI 全绿；覆盖率 < 30% 时 job 失败。
- **风险**：门禁太严阻塞迭代。**缓解**：先「warn 不 block」跑一轮观察，再切硬门禁；阈值阶梯爬坡。

### T3 前端测试框架落地

- **改动**：`platform/frontend` 装 `vitest`（必要时 `@testing-library/react`）；加 `test` script；根 `fe:test` 跑通；把 `lib/devstep.test.ts` 从 standalone 迁入 vitest；补 1–2 个纯函数/组件测试示范。e2e（playwright）留后续，本次不做。
- **验证**：`pnpm fe:test` 绿；CI 前端 job 含 test 步骤（T2.③）。
- **验收**：框架落地 + devstep 测试纳入 CI + ≥1 个示范测试。
- **风险**：Next 16 + React 19 与训练数据差异。**缓解**：`platform/frontend/AGENTS.md` 已警示——写测试前先读 `node_modules/next/dist/docs/` 对应章节， heed deprecation。

### T4 迁移工具化（golang-migrate）

- **改动**：引入 `golang-migrate/migrate`，embed 用其 `source/iofs`；**保留 `Migrate(ctx, db) error` 签名不变**（内部改实现，上层 `main.go` 无感）；支持 up/down + 单迁移文件事务包裹；Makefile 加 `migrate-up` / `migrate-down`。
- **验证**：空库 `migrate up` → 表结构与 `000001_init.up.sql` 完全一致；`migrate down` 回滚干净；现有 dev 库用 `migrate force` 接入。
- **验收**：迁移可 up/down；多人改表各加独立编号文件（`000002_*.up.sql`）。
- **风险**：动运行中库，破坏性最高。**缓解**：①独立分支；②保留旧 `Migrate` 签名；③空库全量验证；④对现有库 `force` 兜底（沿用 P0-3 预案）。

## 3. 落地顺序与依赖

```
T2.① Go 版本对齐   ← 最先（5 分钟，立刻让 CI 可信）
   │
   ├─> T1 后端补单测 ─┐
   │                  ├─> T2.② 覆盖率门禁（等覆盖率拉到 30% 再启用 block）
   └─> T3 前端测试框架 ─┘
                          │
T4 迁移工具化（独立分支，可与 T1/T3 并行，最后合） ◄──── 高风险，单独评估
```

- T1 与 T3 可并行（后端/前端互不依赖）。
- T2 门禁要等 T1 把覆盖率拉到 30% 才切硬门禁，否则会先红。
- T4 高风险，独立分支 + 空库验证，不阻塞 T1/T3。

## 6. 测试暴露并已修复的生产 bug（切 PG 遗漏）

补单测过程中暴露 schema/SQL 与生产 PG 不兼容的 bug，已全部修复（生产 PG-7 切换前清雷）：

| #   | 严重度 | 位置                                            | 问题                                                                                                                              | 修复                                                                               |
| --- | ------ | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| 1   | 高     | `requirement` 表                                | 迁移转换遗漏 `application_id/priority/fixed_version` 三列，repository.go 已引用 → 生产 INSERT/SELECT 直接报 column does not exist | init.up.sql 补列 + 新增 `000002_requirement_columns` up/down（IF NOT EXISTS 幂等） |
| 2   | 高     | `auth/membership.go:49` Roles                   | 动态 WHERE 用 `?`，PG 只认 `$N` → MyTasks 按空间查角色失败                                                                        | `?`→`$2`                                                                           |
| 3-7 | 高     | `attendance/security/ops/capability` 的 ListBy* | 动态拼 WHERE 系统性混用 `?`（SQLite 双兼容掩盖、PG 报错）                                                                         | 统一改 `?` + 执行前 `sqlx.Rebind(sqlx.DOLLAR, q)` 自动编号                         |

> 根因：切 PG 时静态 SQL 的 `$N` 改了，但**动态拼接 WHERE 的 `?`** 漏扫。共 6 处 `?` + 1 处 schema 缺列。
