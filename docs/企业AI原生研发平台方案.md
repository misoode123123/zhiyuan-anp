# 第一篇　市场格局与战略定位

## 1. 市场格局：现有企业级 AI 开发平台全景

目前并不存在一个"开箱即用、完全满足愿景"的单一平台。市场产品分布在四个不同层，能力彼此割裂。

### 1.1 四大类平台矩阵

| 类别 | 代表产品 | 解决的问题 | 差距 |
|---|---|---|---|
| **A. AI 应用/智能体搭建平台** | 扣子 Coze（字节，Apache 2.0 开源）、Dify、FastGPT、蚂蚁数科 Agentar、n8n | 拖拽/自然语言搭 AI 智能体或 RAG 应用并部署 | 只覆盖"建应用"，不覆盖研发/测试/运维全流程；治理偏弱 |
| **B. AI 编码代理（Agentic Coding）** | Devin、Cursor、Factory AI、Claude Code、GitHub Copilot Workspace | AI 写代码、跑测试、提 PR | 是"工程师的工具"，业务用不了；不碰需求/部署/运维 |
| **C. AI 原生 SDLC 平台** | LTIMindtree SDLC AI Radar 等厂商套件 | 一个平台内研发各阶段 AI 化 | 多为服务商私有方案、绑定人力；2026 年大多停在"AI 辅助"而非"自主" |
| **D. DevOps / 平台工程底座** | GitLab、Jenkins、ArgoCD、Kubernetes | 流水线、部署、运维工程底座 | 本身无 AI，需上层大脑驱动 |

### 1.2 行业结论（两条硬约束）

1. 不要追求"全程无人工"，而要设计**分级自治 + 关键节点人工闸门**。2026 年头部成绩约 55% 提速、90%+ 测试覆盖率、人在回路；完全自主"AutoOps"仍属愿景。
2. 平台必须**内生治理**（制度、流程、标准、合规、审计），把 AI 嵌入既有治理之内而非绕过——这是受监管行业的胜出路径。

## 2. 平台定位与核心理念

### 2.1 愿景拆解

| # | 原话 | 工程语言 |
|---|---|---|
| 1 | 覆盖需求→研发→部署→运维全流程 | 端到端 AI-SDLC 流水线 |
| 2 | 用户全程参与需求提出 | 业务侧对话式需求工作台（非技术人可用） |
| 3 | 分不同应用板块 | 多工作台/多租户的应用矩阵 |
| 4 | 业务部门直接跟 AI 对话 | 自然语言即接口，AI 担任业务分析师 |
| 5 | AI 嵌入各应用前端及底层 | Agent SDK + Copilot 注入 + 底层能力市场 |
| 6 | 企业只投入算力、制定研发规则 | 算力即基础设施 + 规则即代码（RaC） |
| 7 | 各部门录入制度/流程/标准，AI 驱动一切，关键节点人决策 | 知识治理中心 + 分级自治闸门 |

### 2.2 产品定位

> 「智源」是企业自己的 AI 软件工厂操作系统：业务部门说话提需求，AI 在制度与规则护栏内自主完成研发、测试、部署、运维，人在关键闸门把关；企业只需供算力、立规矩、养知识。

### 2.3 五大设计原则

1. **Rule-as-Code（规则即代码）**：制度/流程/标准结构化入库，成为 AI 不可违反的硬约束。
2. **Gated Autonomy（分级自治）**：按风险分级，低风险全自动、高风险强制人工闸门。
3. **Human-in-the-Loop, Not-in-the-Weeds**：人只在决策点出现，不陷在执行细节。
4. **Knowledge-First（知识资产化）**：企业文档是驱动 AI 行为的"操作依据"，而非"参考资料"。
5. **Model-Agnostic（模型无关）**：算力与模型解耦，可接 Claude/GPT，也可接通义/文心/DeepSeek/Qwen，避免锁定、便于信创。

# 第二篇　总体方案（立项级）

## 3. 项目目标（可量化）

| 维度 | 现状基线（典型） | 18 个月目标 | 36 个月目标 |
|---|---|---|---|
| 需求→上线周期 | 6–12 周 | ≤ 3 周 | ≤ 1 周（标准需求） |
| 代码 AI 生成占比 | < 10% | 50–60% | 70%+ |
| 自动化测试覆盖率 | 20–40% | 85%+ | 90%+ |
| 业务侧自主提出并推进的需求比 | < 5% | 40% | 70% |
| 生产变更人工介入率 | ~100% | 50%（低风险全自动） | 30% |
| 制度/标准数字化覆盖率 | < 20% | 70% | 95% |

## 4. 价值假设

1. **研发产能跃升**：人从"写代码"解放到"立规矩+做决策"。
2. **业务敏捷**：业务直接对话提需求，省去需求传递损耗。
3. **治理前置**：制度/合规变成 AI 硬约束，从"事后审计"变"事前阻断"。
4. **资产沉淀**：制度、流程、经验变成可复用 AI 资产，越用越值钱。
5. **去锁定**：模型/算力解耦，信创可平滑替换。

## 5. 项目范围与边界

**✅ 在范围内**：9 大板块、AI-SDLC 流水线、规则治理中心、模型/算力管理、AI 能力市场；覆盖企业内部研发与业务系统的 AI 化。

**❌ 不在范围（V1 不做）**：自研基础大模型（采购/接入）；硬件 GPU 制造（采购算力）；完全无人工的 L4 自治生产运维（关键节点必须留人）。

## 6. 总体架构设计

采用 **六层 + 两横切** 分层架构：底层供算力、中间放智能与知识、上层是各业务板块，横切贯穿安全与可观测。

```
┌─────────────────────────────────────────────────────────────────────────┐
│  ⑤ 应用板块层 (Business Workspaces) —— 面向不同部门/角色的工作台           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │需求工作台│ │研发工作台│ │测试中心  │ │发布中心  │ │运维中心  │ …(治理) │
│  │(业务部门)│ │(技术/AI) │ │          │ │          │ │          │        │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘        │
├─────────────────────────────────────────────────────────────────────────┤
│  ④ 协同与决策层 (Orchestration & Governance)                              │
│   人机协同总线 · 审批闸门引擎 · 决策看板 · 变更CAB · 成本/算力调度           │
├─────────────────────────────────────────────────────────────────────────┤
│  ③ AI 研发流水线层 (AI-SDLC Pipeline) —— 平台核心，多Agent接力              │
│   需求→架构→编码→评审→测试→安全→部署→运维→反馈  闭环                       │
├─────────────────────────────────────────────────────────────────────────┤
│  ② 知识与规则资产层 (Knowledge & Rule Assets)                            │
│   企业知识库(RAG) · 制度库 · 流程库 · 标准库 · 研发规范 · 领域本体 · 记忆      │
├─────────────────────────────────────────────────────────────────────────┤
│  ① 模型与智能层 (Model & Agent Runtime)                                  │
│   多模型网关/路由 · Agent运行时 · 工具市场 · 上下文/记忆 · 向量检索           │
├─────────────────────────────────────────────────────────────────────────┤
│  ⓪ 基础设施与算力层 (Infra & Compute)                                    │
│   GPU/算力池 · 模型推理服务 · 对象存储 · K8s · 容器 · 网络                  │
├─────────────────────────────────────────────────────────────────────────┤
│ 横切：安全与合规（鉴权/脱敏/审计/密钥） · 可观测性（日志/指标/Trace/成本）   │
└─────────────────────────────────────────────────────────────────────────┘
```

各层职责：

| 层 | 职责 | 关键组件 |
|---|---|---|
| ⓪ 算力层 | 企业"只投入算力"的落点 | GPU 池、vLLM/SGLang 推理、K8s、对象存储 |
| ① 智能层 | 大脑，模型无关 | 模型网关、多模型路由、Agent Runtime、工具市场 |
| ② 知识规则层 | 企业"立规矩"的落点 | 制度/流程/标准库、研发规范、领域本体 |
| ③ 流水线层 | AI 驱动的研发主线 | 9 个阶段 Agent 接力 |
| ④ 协同决策层 | "关键节点人决策"的落点 | 审批闸门、CAB、决策看板 |
| ⑤ 应用板块层 | "分应用板块、业务对话"的落点 | 多工作台矩阵 |

## 7. 组织与角色设计

| 角色 | 职责 | 来源转型 |
|---|---|---|
| **规则架构师** | 把制度/标准结构化为可执行规则 | 资深架构师/流程专家 |
| **AI 编排师** | 维护 Agent 编排、提示工程、效果调优 | 高级研发 + 算法 |
| **闸门负责人** | 守住关键决策点、审批 SLA | 技术管理/业务负责人 |
| 研发工程师 | 从"写代码"→"审 AI、攻难题、定规范" | 现有研发 |
| 业务部门 | 从"提 Word 需求"→"对话提需求+验收" | 现有业务 |

**RACI 抽样**（一次需求上线）：

| 活动 | 业务 | 研发 | 规则架构师 | 闸门负责人 | AI |
|---|---|---|---|---|---|
| 需求确认(G1) | **A/R** | C | I | I | R(起草) |
| 架构评审(G2) | I | **A** | R | C | R(起草) |
| 上线审批(G5) | C | C | I | **A/R** | R(起草) |

## 8. 投入估算框架（按规模量级）

假设：研发 200 人企业、年新增/迭代需求 ~300 个。

| 类目 | 估算（年） | 说明 |
|---|---|---|
| 算力（GPU 推理） | 中等规模 | 自建 GPU 池 + 弹性云；可先租后建 |
| 平台研发团队 | 15–25 人 | 含编排师/规则架构师/平台研发 |
| 模型 API 费用 | 与算力二选一/混合 | 国产模型可大幅降低 |
| 开源/商业工具 | 少量 | Dify/扣子开源免费；编码工具按席位 |
| 培训与变革 | 一次性 + 持续 | 容易被低估，建议单列 |

> 投入产出锚点：若产能提升 50%+，相当于等效扩容 ~100 人力，ROI 通常在 12–18 个月转正。

## 9. 度量体系（三层）

- **业务层**：需求交付周期、业务满意度、AI 交付占比。
- **研发层**：AI 代码占比、测试覆盖率、缺陷逃逸率、变更失败率。
- **平台层**：算力利用率、Token 成本/需求、规则覆盖率、闸门通过率、Agent 成功率。

## 10. 三阶段实施路线

- **阶段一（0–6 月）**：算力+智能层地基、规则治理中心、需求工作台、1–2 试点。
- **阶段二（6–18 月）**：研发/测试/发布/运维全流水线、AI 能力市场。
- **阶段三（18–36 月）**：全部门知识数字化、运维自愈、规模化。

```
阶段一(0-6月) ── 地基与试点            阶段二(6-18月) ── 全流程打通
├ 算力+模型网关+Agent运行时           ├ 编码/测试/部署流水线
├ 制度规则治理中心(⭐先建)            ├ 发布中心+运维中心
├ 需求工作台(1个业务试点)            ├ AI能力市场+前端Copilot
└ 1-2个试点项目端到端跑通            └ 覆盖核心业务线

阶段三(18-36月) ── 规模化与自主化
├ 全部门制度/标准全面数字化入库
├ AI 嵌入所有业务系统前端
├ 运维自愈、成本智能调度
└ 持续优化、形成企业AI资产飞轮
```

## 11. 总体治理原则

- **规则先行**：任何 AI 行为必须有对应规则可循。
- **闸门分级**：低风险放行、高风险留人。
- **全链路留痕**：每一次 AI 产出与人工决策可追溯、可审计。
- **可降级**：AI 故障时能切换到人工流程，业务不中断。

# 第三篇　核心机制设计

## 12. AI 内核设计（平台大脑）

### 12.1 多 Agent 接力 + 总控编排

流水线不是一个大模型干到底，而是专业 Agent 分工接力，由 Orchestrator 总控调度：

```
                  ┌───────────────────────────────┐
   业务自然语言 ──▶│  Orchestrator 总控 Agent       │
                  │  (规划/拆解/调度/汇总/记忆)     │
                  └───────────────┬───────────────┘
            ┌──────────────┬──────┴───────┬──────────────┐
            ▼              ▼              ▼              ▼
      ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
      │需求分析  │  │架构设计  │  │编码实现  │  │测试生成  │  …
      │ Agent    │  │ Agent    │  │ Agent(s) │  │ Agent    │
      └──────────┘  └──────────┘  └──────────┘  └──────────┘
            │             │             │             │
            └─────────────┴──────┬──────┴─────────────┘
                                 ▼
                     ┌───────────────────────┐
                     │  共享上下文 / 项目记忆  │
                     └───────────────────────┘
```

每个 Agent 遵循 **感知→检索知识/规则→规划→工具调用→自检→产出** 循环，并强制读取规则资产层。

### 12.2 模型路由（成本与能力平衡）

| 任务类型 | 推荐模型档位 |
|---|---|
| 需求澄清、对话、轻量总结 | 小/快模型（Haiku 档、Qwen-Turbo、DeepSeek） |
| 复杂架构、长上下文编码 | 旗舰模型（Claude Opus/Sonnet 档、DeepSeek-R1、Qwen3-Max） |
| 代码补全、单测 | 专用编码模型 / 本地小模型 |
| 安全审计、合规审查 | 独立模型 + 规则引擎交叉验证 |

路由策略走模型网关，按"任务难度 × 成本 × 延迟"动态选模——这是"企业只投算力"的成本优化关键。

### 12.3 记忆与上下文

- **项目记忆**：架构决策、命名约定、历史踩坑——沉淀复用。
- **短期上下文**：当前任务中间产物，走向量检索 + 摘要压缩。
- **企业记忆**：制度/流程/标准是最高优先级"硬记忆"，AI 行为必须与之对齐。

## 13. AI 驱动的研发全流程（需求 → 上线）

```
需求     架构     编码      评审      测试      安全      部署      运维
 │        │        │        │        │        │        │        │
[需求分析]→[架构设计]→[编码实现]→[代码评审]→[测试生成]→[安全扫描]→[发布]→[巡检/自愈]
 │        │        │        │        │        │        │        │
🚪G1     🚪G2      ▲       🚪G3     ▲        🚪G4     🚪G5     🚪G6
需求     架构     自动     代码     自动     安全     上线     高风险
确认     评审     合入候选 审批     回归     审批     审批     操作

  ▲ 自动合入候选：低风险变更可自动；高风险升级到 G3 人工审
  ◀──── 反馈闭环：线上问题/缺陷回流为知识，优化下一轮 ────▶
```

| 阶段 | AI 做什么 | 闸门（人决策什么） |
|---|---|---|
| 需求分析 | 澄清、转规格、查重、估工期 | **G1**：范围/优先级/预算确认 |
| 架构设计 | 选型、画架构、技术方案 | **G2**：架构评审（影响大的系统） |
| 编码实现 | 写代码、自检、提 PR | 低风险自动；高风险升级 G3 |
| 代码评审 | AI Review + 规则校验 | **G3**：核心模块/高风险代码人工审 |
| 测试生成 | 生成用例/脚本、跑回归 | 自动，失败阻断 |
| 安全扫描 | SAST/依赖/密钥/合规 | **G4**：高危/合规风险人工审 |
| 发布 | 灰度、变更单、回滚预案 | **G5**：上线审批（CAB） |
| 运维 | 巡检、告警研判、自愈 | **G6**：高风险操作（删数据/资金/对外）人工 |

## 14. 关键节点人工决策机制（分级自治）

### 14.1 自治等级

| 自治等级 | 含义 | 示例 |
|---|---|---|
| **L0 全自动** | 无需人介入 | 单测运行、文档生成、巡检采集 |
| **L1 知会** | 自动执行，事后通知 | 低风险 PR 合入、非生产部署 |
| **L2 审批** | 必须人工批准 | 上线发布、架构变更 |
| **L3 双确认** | 多人/多角色确认 | 生产数据变更、对外发布、资金操作 |

### 14.2 闸门引擎三要素

每个闸门配置：**触发条件 + 审批角色 + SLA + 留痕审计**。闸门决策本身被 AI 辅助（AI 给风险摘要、影响面、建议），但**拍板权在人**。

## 15. AI 嵌入机制（嵌入各应用前端及底层）

1. **前端 Copilot 注入**：统一 SDK，任意前端可挂载"对话助手/智能表单/智能搜索"，能力来自 AI 能力市场。
2. **底层能力市场**：平台能力封装为"技能（Skill）/ API"，业务应用按需调用。
3. **数据/业务侧 Agent**：财务、人力、客服等业务系统嵌入领域 Agent，共用同一套治理与算力。

```
        各业务应用前端 (Web/App/小程序)
              │ 挂载
              ▼
       ┌──────────────┐         ┌──────────────────┐
       │ Copilot SDK  │◀───────▶│ AI 能力市场(Skill)│
       └──────┬───────┘         └────────┬─────────┘
              │                          │ 统一调用
              └──────────┬───────────────┘
                         ▼
              模型与智能层 + 治理(横切)
```

## 16. 项目空间与多项目管理

**定位**：9 大板块共同的数据组织维度。所有项目都从同一个「需求工作台」进需求、走同一条流水线，但每个项目是一个独立的**项目空间**——数据隔离、规则可定制、底座共享。这是平台"一个入口管所有项目"的根本机制。

### 16.1 项目空间（Project Workspace）

平台里每个项目都是一个项目空间，是组织一切的基本单元。需求工作台、研发工作台等所有板块都遵循"**先选项目空间，再操作**"。

```
                    智源 ANP 平台（统一底座）
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   项目空间 A            项目空间 B          项目空间 C
  (如 客服系统)        (如 财务系统)       (平台自身迭代)
        │                   │                   │
   各自的：             各自的：            各自的：
   · 需求池             · 需求池            · 需求池
   · 规则包             · 规则包            · 规则包
   · 知识库             · 知识库            · 知识库
   · Agent配置          · Agent配置         · Agent配置
   · 闸门等级           · 闸门等级          · 闸门等级
   · 算力配额           · 算力配额          · 算力配额
        │                   │                   │
        └──── 共享：模型网关、算力池、公共规则、公共组件、公共Agent ────┘
```

### 16.2 三类项目接入形态

| 项目类型 | 例子 | 接入需求管理 | 接入深度 |
|---|---|---|---|
| **A. 平台用 AI 新研发的项目** | 从 0 到 1 做的客服系统 | ✅ 全自动纳入 | 全流程：需求→研发→测试→部署→运维 |
| **B. 企业存量系统** | 已经在跑的老 ERP/CRM | ✅ 可接入 | 轻接入：只接管需求 + 运维；存量代码渐进式纳入 |
| **C. 平台自身（自举/dogfooding）** | ANP 平台自己的迭代 | ✅ 自己也是用户 | 全流程——平台用自己的需求工作台管自己的需求 |

> **关键澄清**：「接入需求管理」≠「必须用平台重写」。存量系统可只接需求入口和运维，代码不必动；后续迭代再渐进纳入 AI 研发流水线。避免"要么全干、要么不干"的僵局。

### 16.3 关键设计要点

1. **项目模板**：新建项目时选模板（如"后台管理系统""数据应用""对外小程序"），自动带预设的规则包、流水线、Agent 配置——开箱即用。
2. **需求跨项目关联**：一个大需求可能涉及多个系统（如"全渠道客户视图"涉及 CRM + 客服 + BI），支持需求拆分、跨空间关联，提供**项目群/产品组合**视图。
3. **公共资产 vs 项目私有**：公共规则（全公司安全红线）、公共组件、公共 Agent 在所有空间共享；项目私有规则只在自己空间生效。
4. **权限与可见性**：业务方只看到自己有权限的项目空间需求池；平台管理员有全局视图。

### 16.4 与 9 大板块的关系

项目空间是所有板块的**数据组织维度**——需求池、规则包、知识库、Agent 配置、闸门策略、算力配额都挂在项目空间下。换言之，板块 1–9 描述"做什么"，项目空间描述"在哪个项目里做、用什么配置做"。两者正交，共同构成平台的完整运行模型。

# 第四篇　分板块详细设计

> 统一模板：定位 / 角色 / 功能模块 / 核心流程 / Agent 设计 / 数据与规则 / 闸门 / 板块接口 / KPI / 落地优先级。

## 板块 1 · 需求工作台（业务入口，最关键）

**定位**：业务部门唯一主入口。不会技术的人用自然语言把业务问题变成可执行需求，并全程跟踪到上线。

**角色**：业务需求方（主力）、产品经理、需求分析师 Agent、规则架构师（评审）。

**功能模块**：
1. 对话式需求采集：多轮澄清，AI 主动追问歧义、补全边界。
2. 需求规格自动生成：用户故事、验收标准、流程图、线框原型。
3. 需求查重与关联：检索历史需求/现有功能。
4. 工期与算力预估：基于历史数据。
5. 需求池与优先级看板：业务自助排序、排期。
6. 全程进度透视：业务看到"进度卡片+人话"，非技术 ticket。

**核心流程**：
```
业务对话 ─▶ [需求澄清] ─▶ [生成规格] ─▶ 🚪G1需求确认 ─▶ [排期入池]
                                              │
                                   (业务确认范围/优先级/预算)
                                              ▼
                                        进入研发流水线
              ◀──── 任意时刻可对话变更/补充 ────┤
              ◀──── 全程进度卡片透视 ──────────┘
```

**Agent 设计**：需求澄清 Agent（追问异常/权限/数据来源）、需求规格 Agent（套需求模板标准）、需求查重 Agent（向量+语义）、预估 Agent（回归历史工时）。

**数据与规则**：读需求模板标准、业务术语表、领域本体、合规红线；写需求库、澄清经验。

**闸门**：🚪 G1 需求确认（业务确认范围/优先级/预算，AI 起草人拍板）。

**板块接口**：出需求规格→研发工作台；入研发可行性/验收结果。

**KPI**：澄清轮次、规格一次通过率、查重命中率、需求→上线周期、业务满意度。

**落地优先级**：**P0（阶段一必做，试点首选）**。

---

## 板块 2 · 研发工作台（核心生产力）

**定位**：AI 与技术人协同写代码主战场。AI 承担主体编码，人聚焦架构、难点、规范、评审。

**角色**：研发工程师、架构师、编码 Agent 群、代码评审 Agent。

**功能模块**：
1. AI 编码（任务级）：给一个任务，Agent 自主"读代码→改→自检→提 PR"。
2. 代码资产与知识：代码库、组件库、架构决策记录（ADR）。
3. AI 代码评审：自动 Review + 规则校验 + 安全扫描。
4. 架构设计工作区：AI 出方案、人评审。
5. 技术知识沉淀：踩坑、最佳实践自动入库。
6. 多 Agent 并行：多任务/模块并行开发。

**核心流程**：
```
需求规格 ─▶ [架构Agent:出方案] ─▶ 🚪G2架构评审 ─▶ [任务拆解]
                                                      │
            ┌─────────────────────────────────────────┤
            ▼              ▼              ▼            ▼
        [编码Agent]   [编码Agent]   [编码Agent]  ...(并行)
            └──────────────┴──────┬───────┴────────────┘
                                  ▼
                        [AI代码评审+规则校验]
                                  │
                    低风险 ─▶ 自动合入候选
                    高风险 ─▶ 🚪G3 代码人工审批
```

**Agent 设计**：架构 Agent（出方案+ADR）、编码 Agent 多实例（Claude Code/Cursor/Devin 类，遵循规范）、评审 Agent（交叉审查）、知识沉淀 Agent（抽取经验）。

**数据与规则**：读编码规范、架构标准、安全红线、依赖白名单；写 ADR、组件库、踩坑库。

**闸门**：🚪 G2 架构评审（影响大系统必过）；🚪 G3 代码审批（核心/高风险人工审，低风险自动）。

**板块接口**：入需求规格（板块1）、架构规范（板块4）；出可测 PR→测试中心（板块3）。

**KPI**：AI 代码占比、PR 一次通过率、评审缺陷发现率、人均产能、返工率。

**落地优先级**：**P0（阶段二核心）**。

---

## 板块 3 · 测试与质量中心

**定位**：测试从"人写人跑"变成"AI 生成、自动执行、持续回归"，质量左移。

**角色**：QA 工程师、测试 Agent、研发（消费结果）。

**功能模块**：用例自动生成；自动化执行（单测/集成/接口/UI，CI 触发）；覆盖率与缺口分析；缺陷智能分析（根因/去重/指派）；测试数据与脱敏；回归智能选取。

**核心流程**：
```
PR 产出 ─▶ [用例生成Agent] ─▶ [执行] ─▶ 全过?
                                      ├ 是 ─▶ 进入安全扫描/发布
                                      └ 否 ─▶ [缺陷分析] ─▶ 打回编码Agent修复
```

**Agent 设计**：用例生成 Agent（正常+异常+边界）、执行编排 Agent（环境/并行/重试）、根因分析 Agent（日志/堆栈）。

**数据与规则**：读覆盖率标准、必测场景、脱敏规则；写用例库、缺陷模式库。

**闸门**：测试失败 = 自动阻断（硬质量门）。

**板块接口**：入 PR（板块2）、测试标准（板块4）；出质量报告→发布中心（板块6）。

**KPI**：测试覆盖率、缺陷逃逸率、自动化执行率、平均修复时长。

**落地优先级**：**P1（阶段二）**。

---

## 板块 4 · 制度与规则治理中心（平台心脏 ⭐）

**定位**：愿景"各部门制定制度/流程/标准录入，AI 据此驱动一切"的落点。把治理从"墙上文件"变成"AI 硬约束"。是平台护城河与数据资产。

**角色**：各部门制度 Owner（业务/IT/安全/法务/财务）、规则架构师、合规审计员。

**功能模块**：
1. 制度/流程/标准录入工作台：文档上传、结构化录入、对话式录入。
2. 规则结构化引擎（Rule-as-Code）：自然语言制度→可执行规则（条件/动作/生效范围）。
3. 规则版本与生命周期：草稿→评审→发布→废弃，全程版本化。
4. 规则生效范围映射：每条规则绑定"在哪些环节、对哪些 Agent 生效"。
5. 规则校验与阻断：违规即阻断 + 提示违反哪条规则。
6. 合规审计与追溯：任意产出可查"依据了哪些规则"。
7. 规则冲突检测：新旧冲突告警。

**核心流程**：
```
部门制度Owner ─录入(文档/对话)─▶ [规则结构化引擎]
        规则架构师 ──评审──▶ [发布]
                                  │ (版本+生效范围)
                                  ▼
                          ┌───────────────┐
                          │  规则库(RaC)   │
                          └───────┬───────┘
        ┌─────────────────────────┼─────────────────────────┐
        ▼                         ▼                         ▼
   约束编码Agent             约束测试Agent             约束运维Agent
   (板块2必须遵守)           (板块3必须覆盖)           (板块7必须合规)
        │                         │                         │
        └───────── 违规即阻断 + 触达对应人工闸门 ───────────┘

   规则更新 ─▶ 版本升级 ─▶ 历史 AI 产出可追溯(按当时规则版本审计)
```

**规则三层模型（数据模型核心）**：
```
Rule {
  id, 版本, 类型(强制|应遵循|参考),
  生效范围[环节, Agent, 业务域],
  条件(when), 动作(then: 允许/阻断/告警/需审批),
  来源(制度文档+条款), Owner, 生效期, 审计追溯链
}
```

**人机闸门**：规则发布需评审+审批；高风险规则（放宽安全策略等）升级多角色确认。

**板块接口**：被所有板块读取（向下约束所有 AI 行为）；出合规报告给安全合规中心（板块5）、审计追溯给闸门负责人。

**KPI**：规则覆盖率、规则数字化率、违规阻断率、规则平均生效时延、合规审计通过率。

**落地优先级**：**P0（阶段一与需求工作台并列首选，先建护栏再放车）**。

---

## 板块 5 · 安全与合规中心

**定位**：守住"不泄密、不违规、不带病上线"，把安全合规嵌进流水线。

**角色**：安全工程师、法务/合规、安全扫描 Agent、脱敏 Agent。

**功能模块**：代码安全扫描（SAST/依赖/密钥/许可证）；数据安全（敏感识别/脱敏/最小权限/对话防泄密）；合规策略下发（转规则入板块4）；AI 行为审计；模型/提示安全（防注入/越狱/输出过滤）。

**Agent 设计**：安全扫描 Agent（工具链+规则交叉验证）、脱敏 Agent（按数据分级）、合规审查 Agent（对照规则清单）。

**数据与规则**：读安全红线、数据分级、合规要求（板块4）；写漏洞模式、风险事件库。

**闸门**：🚪 G4 安全/合规审批（高危漏洞、合规风险人工审）。

**板块接口**：入 PR（板块2）、合规规则（板块4）；出安全签章→发布中心（板块6）。

**KPI**：高危漏洞逃逸率、密钥泄露事件、合规审计通过率、平均修复时长。

**落地优先级**：**P1（阶段二，基础策略阶段一就要有）**。

---

## 板块 6 · 发布中心

**定位**：上线从"人工熬夜变更"变成"AI 准备 + 人审批 + 自动灰度 + 可回滚"。

**角色**：发布经理、SRE、发布 Agent、闸门负责人。

**功能模块**：变更评审（AI 生成变更单：影响面/回滚方案/检查清单）；环境与配置管理；灰度/金丝雀发布（自动分流/指标守护/自动暂停）；一键回滚；发布报告（归档可审计）。

**核心流程**：
```
测试通过+安全签章 ─▶ [变更单生成] ─▶ 🚪G5上线审批(CAB)
                                          │ (人确认影响面/窗口)
                                          ▼
                          [灰度发布] ─▶ 指标守护 ─▶ 全量
                                          │
                                   异常 ─▶ [自动回滚/🚪人工]
```

**Agent 设计**：变更评估 Agent（影响面/回滚预案）、发布执行 Agent（灰度/监控）、守护 Agent（盯指标/异常暂停回滚）。

**数据与规则**：读变更窗口规则、回滚阈值、环境隔离规范；写变更历史、回滚预案库。

**闸门**：🚪 G5 上线审批（所有生产发布必过 CAB；对外/资金升级 L3 双确认）。

**板块接口**：入质量报告（板块3）、安全签章（板块5）；出上线版本→运维中心（板块7）。

**KPI**：变更失败率、平均回滚时长、发布频率、变更窗口合规率。

**落地优先级**：**P1（阶段二）**。

---

## 板块 7 · 运维中心

**定位**：从"被动救火"到"AI 巡检 + 智能研判 + 自愈 + 人在高风险节点把关"。目标运维自治 L2–L3，关键操作留人。

**角色**：SRE、运维 Agent 群、值班人员。

**功能模块**：AI 巡检（健康检查/容量预测/成本异常）；智能告警研判（去噪/根因/关联）；自愈预案库（重启/扩容/切流）；容量与成本；知识驱动（SOP 来自板块4）。

**核心流程**：
```
监控指标 ─▶ [告警研判Agent] ─▶ 有已知预案?
                              ├ 是(低风险) ─▶ [自动自愈] ─▶ 通知
                              └ 否/高风险 ─▶ 🚪G6 人工决策(删数据/资金/对外)
```

**Agent 设计**：巡检 Agent（周期健康评估）、根因分析 Agent（拓扑+日志+指标）、自愈 Agent（执行预案，受规则闸门约束）。

**数据与规则**：读自愈预案白名单、高风险操作清单、SOP；写故障案例库、根因模式库。

**闸门**：🚪 G6 高风险操作（删数据、资金、对外变更强制人工）。

**板块接口**：入上线版本（板块6）、SOP（板块4）；出运行数据/缺陷→反哺需求/研发（闭环）。

**KPI**：MTTR、告警准确率、自愈成功率、可用性 SLO 达成率、单位算力成本。

**落地优先级**：**P2（阶段三，阶段二先做基础监控与研判）**。

---

## 板块 8 · 算力与资源中心

**定位**：愿景"企业只投入算力"的落点。GPU、模型、Token、配额统一管理，像水电一样按需调度、可计费、可优化。

**角色**：平台运营/IT、算力调度引擎、各业务线（消费方）。

**功能模块**：算力池管理（自建 GPU + 弹性云）；模型网关与路由（多模型接入/按任务路由/故障切换）；配额与计费（按部门/项目核算）；成本看板（实时消耗/异常告警）；弹性与抢占（保高优先级）。

**调度模型**：
```
任务(难度/时延/成本预算) ─▶ [模型路由引擎] ─▶ 选模(旗舰/中档/小模型)
                              │
                              ▼
                    算力调度(自建优先/弹性云溢出)
```

**Agent 设计**：路由 Agent（按任务选模，平衡成本质量）、成本优化 Agent（识别浪费/建议缓存降档）。

**数据与规则**：读成本预算、模型白名单（信创要求）、数据驻留规则；写用量基线、成本模型。

**闸门**：超预算/接入新模型需审批。

**板块接口**：被所有板块消费（统一供算力与模型）。

**KPI**：算力利用率、单位需求 Token 成本、模型可用性、成本/需求下降率。

**落地优先级**：**P0（阶段一地基，与智能层同建）**。

---

## 板块 9 · AI 能力市场（嵌入层）

**定位**：愿景"AI 嵌入各应用前端及底层"的落点。平台能力封装成可复用技能/SDK/API，让任意业务系统都能挂上 AI，共用同一套治理与算力。

**角色**：业务系统开发者、全员（消费方）、能力运营。

**功能模块**：技能市场（Skill Store：需求解析/文档生成/数据问答/流程审批/报表）；前端 Copilot SDK（一行代码挂载对话助手/智能表单/智能搜索）；底层 API 网关（鉴权/限流/计费/审计）；能力治理（绑规则/灰度/下线）；领域 Agent 商店（财务/人力/客服）。

**嵌入三层**：
```
各业务应用前端(Web/App/小程序)
      │ 挂载
      ▼
[前端Copilot SDK] ◀────▶ [技能市场(Skill)]
      │                          │ 统一网关
      └──────────┬───────────────┘
                 ▼
   智能层 + 算力层 + 治理(横切)   ← 所有嵌入能力共用同一套规则与算力
```

**Agent 设计**：能力编排 Agent（组合复合能力）、嵌入适配 Agent（按宿主应用适配交互）。

**数据与规则**：读能力开放范围、数据访问权限、嵌入合规规则；写能力使用统计、复用度排行。

**闸门**：新技能上架、对外暴露需评审（防越权/防泄密）。

**板块接口**：上游调用智能层（板块8）、受规则约束（板块4）、安全（板块5）；下游被所有业务系统消费。

**KPI**：能力复用率、嵌入应用数、能力调用量、单能力 ROI。

**落地优先级**：**P2（阶段三，阶段二先建 SDK 与少量高频技能）**。

## 17. 技术选型建议（含国产化/信创）

| 层 | 推荐选型（可替换） |

| 应用搭建 | Dify / 扣子开源版（Apache 2.0，可私有化） |

| 向量库 | Milvus（国产，高性能） |
| 知识库 | 文档解析 + RAG + 结构化规则库（PG/图数据库） |
| 可观测 | Prometheus + Loki + Tempo + 自研成本看板 |

## 18. 风险、治理与成本

| 风险类别 | 具体风险 | 应对 |
|---|---|---|
| 质量/幻觉 | AI 生成错误代码/需求 | 闸门 + 自动测试 + 规则硬约束 + AI 交叉审查 |
| 安全合规 | 代码/数据泄露、违规 | 私有化部署、数据脱敏、全链路审计、密钥治理 |
| 成本 | 算力/Token 失控 | 模型路由、配额、成本看板、缓存复用 |
| 组织阻力 | 员工担心被替代 | 定位"AI 增强人"、培训、激励机制 |
| 供应商锁定 | 被单一模型/平台绑架 | 模型无关、开源优先、抽象接口 |
| 过度自治 | 盲目信任 AI 全自动 | 分级闸门、关键节点强制人工 |

## 19. 自建 vs 采购：策略建议

- ❌ 不建议从零自建全部——成本极高、周期 3 年+、追不上模型迭代。
- ❌ 不建议全采购单一商业平台——会被锁定，且无一个覆盖全愿景。
- ✅ 建议"混合控制平面"路线：

```
自研核心(护城河)        开源平台(省力)           商业/云(按需)
─────────────────       ─────────────           ───────────
· Orchestrator总控     · Dify/扣子(应用搭建)   · 模型API(多供应商)
· 规则引擎 RaC         · GitLab(代码/流水线)   · GPU云(弹性算力)
· 闸门/决策引擎        · Milvus(向量)          · 编码代理订阅
· 知识/规则资产层
```

**一句话**：把"规则治理中心 + 闸门引擎 + 知识资产 + Orchestrator"做厚（护城河），把"模型、算力、编码工具、应用搭建"做薄（用开源和云按需替换）。

## 20. 板块协同与优先级总览

### 19.1 一次完整需求的全链路（9 板块协同）
```
业务(板块1需求) ─▶ 规则(板块4约束) ─▶ 研发(板块2编码)
   │                                          │
   │                              测试(板块3) ◀─ 安全(板块5)
   │                                          │
   │                              发布(板块6) ◀─ 运维(板块7)
   │                                          │
   └──── 全程由算力(板块8)供能、能力市场(板块9)嵌入 ◀─ 反哺知识(板块4)
```

### 19.2 优先级矩阵

| 优先级 | 板块 | 阶段 |
|---|---|---|
| **P0** | 算力与资源中心、制度与规则治理中心、需求工作台、研发工作台 | 阶段一/二 |
| **P1** | 测试与质量中心、安全与合规中心、发布中心 | 阶段二 |
| **P2** | 运维中心、AI 能力市场 | 阶段三 |

### 19.3 关键依赖

- **规则治理中心（板块4）必须先于/同步于其他 AI 板块**——没有护栏就不能放车跑。
- **算力与资源中心（板块8）是所有 AI 的前提**。
- **需求工作台（板块1）是业务可见价值最快的切口**，建议作为首个试点入口。
